## ADR-001: Gorse 推荐系统运维策略

### 背景

Gorse 推荐系统存在内存持续增长问题（GitHub Issues 中有多个相关报告），需要制定运维策略确保服务稳定性。

### 问题

1. 定期重启 Gorse 是否导致训练结果丢失？
2. 重启后是否需要重新训练？训练期间能否继续推荐？
3. 如何在不影响用户体验的前提下释放内存？

### 分析

#### Gorse 数据持久化机制

```
┌─────────────────────────────────────────────────────────────────┐
│                     Gorse 持久化架构                              │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ PostgreSQL (Data Store)                                  │   │
│  │  ├── 原始数据 (items, users, feedbacks)                  │   │
│  │  ├── 模型参数 (trained model weights)                    │   │
│  │  ├── 推荐结果 (precomputed recommendations)              │   │
│  │  └── 系统配置                                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                           │                                      │
│                           │ 重启后加载                            │
│                           ▼                                      │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Redis (Cache Store)                                      │   │
│  │  ├── 推荐缓存 (Top-N 推荐结果)                           │   │
│  │  ├── 特征缓存 (用户/物品特征向量)                         │   │
│  │  └── 热门排行                                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Master/Worker/Server 内存                                │   │
│  │  ├── 模型运行时 (从 DB 加载)                             │   │
│  │  ├── 计算中间态 (重启丢失)                               │   │
│  │  └── 缓存预热 (重启后重建)                               │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

#### 各节点重启影响评估

| 节点 | 重启影响 | 恢复时间 | 对推荐服务的影响 |
|:---|:---|:---|:---|
| **Master** | 重新加载模型 + 配置 | 10-30秒 | 无影响 |
| **Worker** | 释放内存，重新加载模型 | 30秒-2分钟 | 离线推荐暂停，在线推荐正常 |
| **Server** | 重新连接 Redis | 5-10秒 | 短暂不可用（可多实例避免） |

### 决策

| 项目 | 决策 | 理由 |
|:---|:---|:---|
| **训练结果持久化** | 模型参数存储在 PostgreSQL | 重启不丢失 |
| **重启策略** | 滚动重启 (Rolling Restart) | 确保始终有实例在线 |
| **重启频率** | Worker 每天凌晨低峰期重启 | 释放内存，避免 OOM |
| **内存限制** | Docker `--memory` 限制 | 防止单节点内存无限增长 |
| **缓存策略** | Redis LRU + TTL | 自动淘汰冷数据 |

### 滚动重启流程

```
1. 重启 Worker-1 (保持 Worker-2 提供服务)
2. Worker-1 恢复后，重启 Worker-2
3. 最后重启 Master

期间保障:
- Server 节点始终在线 (从 Redis 获取推荐)
- 推荐服务不中断
- 离线推荐短暂延迟 (不影响用户体验)
```

### Gorse 内存优化配置

```yaml
# docker-compose.yml (Gorse Worker 配置)
services:
  gorse-worker:
    image: zhenghaoz/gorse-worker:latest
    deploy:
      resources:
        limits:
          memory: 32G      # 硬限制，防止 OOM
          cpus: '16'
    environment:
      GORSE_MAX_MEMORY: 28GB        # 软限制，预留缓冲
      GORSE_CACHE_SIZE: 1000000     # LRU 缓存条目上限
      GORSE_BATCH_SIZE: 1000        # 分批处理，减少内存峰值
    restart: unless-stopped
```

### 监控告警

| 指标 | 告警阈值 | 动作 |
|:---|:---|:---|
| Gorse Worker 内存 | > 80% 限制 | 触发重启 |
| Gorse Master 内存 | > 80% 限制 | 触发重启 |
| Redis 内存 | > 80% 限制 | 检查缓存策略 |
| 推荐 P99 延迟 | > 200ms | 检查模型加载状态 |

## ADR-002: 自建服务器 vs 云服务商成本分析

### 背景

Pigsty 的初衷是通过自建数据库服务器降低成本。上云方案虽然便捷，但长期成本高昂（"杀猪盘"）。需要在架构设计中考虑成本优化路径。

### 问题

1. 阿里云方案的年成本是多少？
2. 自建托管方案的年成本是多少？
3. 何时应该从云方案迁移到自建方案？

### 分析

#### 成本对比 (2000万 DAU 规模)

##### 阿里云方案 (月费)

| 组件 | 配置 | 月费 (万元) |
|:---|:---|:---|
| 数据库主库 | 32c256g 2TB NVMe | 1.5 |
| 数据库从库 ×2 | 16c128g 2TB NVMe | 1.6 × 2 |
| 应用服务器 ×4 | 16c64g 500GB | 1.2 × 4 |
| Gorse 集群 | 8台 | 3.0 |
| Redis ×3 | 8c32g | 0.6 × 3 |
| SLB + 带宽 | - | 0.5 |
| **总计** | - | **~15 万/月** |

##### 自建托管方案 (月费)

| 组件 | 配置 | 数量 | 月费 (万元) |
|:---|:---|:---|:---|
| 数据库服务器 | 32c256g 2TB NVMe | 1 | 托管 0.3 |
| 数据库从库 ×2 | 16c128g 2TB NVMe | 2 | 托管 0.2 × 2 |
| 应用服务器 ×4 | 16c64g 500GB | 4 | 托管 0.15 × 4 |
| Gorse 服务器 | 8台 | 8 | 托管 0.1 × 8 |
| Redis ×3 | 8c32g | 3 | 托管 0.1 × 3 |
| IDC 带宽 | 10Gbps | - | 2.0 |
| 运维人员 | DBA + DevOps | 2 | 3.0 |
| **总计** | - | - | **~7.5 万/月** |

##### 一次性硬件采购成本

| 组件 | 数量 | 采购成本 (万元) |
|:---|:---|:---|
| 数据库服务器 (32c256g) | 3 | 45 |
| 应用服务器 (16c64g) | 4 | 24 |

#### 投资回报分析

|:---|:---|:---|:---|

### 决策

| 阶段 | 方案 | 理由 |
|:---|:---|:---|

| **数据备份** | 每日全量 + 实时 WAL | 异地灾备 |
| **运维团队** | DBA + DevOps | 7×24 监控 |
| **DDoS 防护** | 高防 IP | 抵御攻击 |

## ADR-003: 用户行为数据写入架构

### 背景

用户行为数据（点赞、收藏、分享、评论、对话、日志等）会产生巨大的写入压力。初步估算峰值写入 QPS 超过 7000，远超单 PostgreSQL 的写入能力（2000-3000 TPS）。

### 问题

1. 如何承载 7000+ TPS 的写入压力？
2. 哪些数据需要强一致，哪些可以接受最终一致？
3. 如何设计分层写入架构？

### 分析

#### 完整用户行为模型

| 行为类型 | 频率 (每登录用户/天) | 日总量 (1400万登录) | 写入频率 |
|:---|:---|:---|:---|
| **浏览景点** | 50次 | 7亿次 | **仅日志，不写 DB** |
| **点赞** | 5次 | 7000万次 | **写 DB** |
| **收藏** | 2次 | 2800万次 | **写 DB** |
| **评论** | 1次 | 1400万次 | **写 DB** |
| **分享** | 0.5次 | 700万次 | **写 DB** |
| **对话/消息** | 2次 | 2800万次 | **写 DB** |
| **上传图片** | 0.2张 | 280万张 | **写 DB (元数据)** |
| **发布游记** | 0.1篇 | 140万篇 | **写 DB** |
| **行为日志** | 100次 | 140亿次 | **写日志系统，不写 PG** |

#### 写入压力估算

| 写入类型 | 日总量 | 平均/秒 | 峰值 (4x) |
|:---|:---|:---|:---|
| 点赞 | 7000万 | 810/秒 | 3240/秒 |
| 收藏 | 2800万 | 324/秒 | 1296/秒 |
| 评论 | 1400万 | 162/秒 | 648/秒 |
| 分享 | 700万 | 81/秒 | 324/秒 |
| 对话 | 2800万 | 324/秒 | 1296/秒 |
| 图片元数据 | 280万 | 32/秒 | 128/秒 |
| 游记 | 140万 | 16/秒 | 64/秒 |
| **写入总计** | **1.53亿** | **1768/秒** | **7072/秒** |

**单 PostgreSQL 写入上限**: 约 2000-3000 TPS → **必须分层处理！**

### 决策

#### 写入分流策略

| 行为 | 写入方式 | 数据路径 | 一致性 |
|:---|:---|:---|:---|
| **评论** | 同步写入 | 前端 → APISIX → PostgREST → PG | 强一致 |
| **游记** | 同步写入 | 前端 → APISIX → PostgREST → PG | 强一致 |
| **图片/视频** | 同步写入元数据 | 前端 → APISIX → PostgREST → PG | 强一致 |
| **点赞** | 异步写入 | 前端 → Redis Streams → Consumer → PG (批量) | 最终一致 |
| **收藏** | 异步写入 | 前端 → Redis Streams → Consumer → PG (批量) | 最终一致 |
| **分享** | 异步写入 | 前端 → Redis Streams → Consumer → PG (批量) | 最终一致 |
| **对话** | 异步写入 | 前端 → Redis → ClickHouse | 最终一致 |
| **行为日志** | 异步写入 | 前端 → Workers → VictoriaLogs | 最终一致 |

#### 分流后 PG 写入压力

| 写入类型 | 峰值 QPS | 写入方式 |
|:---|:---|:---|
| 评论 | 648/秒 | 同步 |
| 游记 | 64/秒 | 同步 |
| 图片元数据 | 128/秒 | 同步 |
| 点赞/收藏/分享 | 4860/秒 | **异步批量 → 50/秒** |
| 对话 | 1296/秒 | **异步 → 不写 PG** |
| **PG 实际写入** | **~900/秒** | - |

**从 7072/秒 降到 900/秒，减少 87%！**

### Redis Streams 消费者实现

```go
// db/syncer/internal/consumer/batch_writer.go

package consumer

import (
    "context"
    "time"
    "github.com/redis/go-redis/v9"
)

type BatchWriter struct {
    redisClient *redis.Client
    pgClient    *pgx.Conn
    batchSize   int
    interval    time.Duration
}

func (bw *BatchWriter) Start(ctx context.Context) {
    ticker := time.NewTicker(bw.interval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            bw.flush()
        }
    }
}

func (bw *BatchWriter) flush() {
    // 从 Redis Streams 读取批量数据
    messages, err := bw.redisClient.XRead(context.Background(), &redis.XReadArgs{
        Streams: []string{"feedbacks", "0"},
        Count:   int64(bw.batchSize),
        Block:   100 * time.Millisecond,
    }).Result()
    
    if err != nil || len(messages) == 0 {
        return
    }
    
    // 批量写入 PostgreSQL
    batch := &pgx.Batch{}
    for _, msg := range messages {
        batch.Queue("INSERT INTO user_feedback (user_id, item_id, type, timestamp) VALUES ($1, $2, $3, $4)",
            msg.Values["user_id"], msg.Values["item_id"], msg.Values["type"], msg.Values["timestamp"])
    }
    
    bw.pgClient.SendBatch(context.Background(), batch)
}
```

### 异步写入对账机制

```sql
-- 定时对账任务 (每小时执行)
CREATE OR REPLACE PROCEDURE reconcile_feedbacks()
LANGUAGE plpgsql
AS $$
DECLARE
    v_count INT;
BEGIN
    -- 统计 Redis 中待处理的反馈
    -- 与 PG 中已写入的对比
    -- 补录缺失的数据
    
    INSERT INTO user_feedback (user_id, item_id, type, timestamp)
    SELECT r.user_id, r.item_id, r.type, r.timestamp
    FROM redis_feedback_queue r
    LEFT JOIN user_feedback uf ON r.user_id = uf.user_id 
        AND r.item_id = uf.item_id 
        AND r.type = uf.type
    WHERE uf.id IS NULL;
    
    GET DIAGNOSTICS v_count = ROW_COUNT;
    RAISE NOTICE 'Reconciled % feedbacks', v_count;
END;
$$;
```
