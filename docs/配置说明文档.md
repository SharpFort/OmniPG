### 2.1 Nginx 统一 Web 入口 (Pigsty INFRA)

Pigsty 的 INFRA 模块自带 Nginx，作为所有 Web UI 的统一入口 (端口 80/443)。
通过不同路径代理到各组件，**这是 Pigsty 文档推荐的标准访问方式**。

| 端点 | 上游组件 | 原生端口 | 说明 |
|:---|:---|:---|:---|
| `/` | Nginx 首页 | 80 | Pigsty 主页、本地软件仓库 |
| `/ui/` | Grafana | 3000 | 监控大盘入口 |
| `/vmetrics/` | VictoriaMetrics | 8428 | 时序数据库 Web UI |
| `/vlogs/` | VictoriaLogs | 9428 | 日志数据库 Web UI |
| `/vtraces/` | VictoriaTraces | 10428 | 链路追踪 Web UI |
| `/vmalert/` | VMAlert | 8880 | 告警规则管理 |
| `/alertmgr/` | AlertManager | 9059 | 告警管理 Web UI |
| `/blackbox/` | Blackbox Exporter | 9115 | 黑盒探测 |
| `/haproxy/*` | HAProxy | 9101 | 负载均衡管理 |

> 来源: [Pigsty WebUI 文档](https://pigsty.cc/docs/setup/webui/)

### 2.2 APISIX 端口说明

APISIX 三个端口是**同一个容器**的不同监听端口 (非三个容器)：

| 端口 | 用途 | 说明 |
|:---|:---|:---|
| 9080 | HTTP 网关端口 | 客户端请求入口 |
| 9443 | HTTPS 网关端口 | 加密客户端请求入口 |
| 9180 | Admin API | 管理接口 (路由/插件/消费者配置) |

> 来源: [APISIX 架构文档](https://apisix.apache.org/docs/apisix/architecture-design/apisix/)

**最佳实践**: Admin API 端口 (9180) 不应暴露给公网。生产环境应通过防火墙限制 9180 仅允许内网访问。

## 四、关键配置说明

### 4.1 PostgreSQL 认证 (`pg_hba.conf`)

```conf
# 本地 Unix socket 认证 (peer)
local   all             postgres                                peer
local   all             all                                     peer

# 本地 TCP 认证 (scram-sha-256)
host    all             all             127.0.0.1/32            scram-sha-256
host    all             all             ::1/128                 scram-sha-256

# Docker 网桥认证 (scram-sha-256)
host    all             all             172.17.0.0/16           scram-sha-256
host    all             all             172.20.0.0/16           scram-sha-256
```

**要点**:
- 所有远程 TCP 连接强制使用 scram-sha-256 (禁用 md5)
- 支持 Docker 默认网桥 (172.17.0.0/16) 和自定义网桥 (172.20.0.0/16)
- 无重复/冲突条目

### 4.2 pgBouncer 认证 (`pgbouncer.ini`)

```ini
[pgbouncer]
auth_type = scram-sha-256      # 与 pg_hba.conf 保持一致
auth_file = /etc/pgbouncer/userlist.txt
```

**要点**:
- `auth_type = scram-sha-256` 必须与 `pg_hba.conf` 中的认证方法匹配
- `userlist.txt` 支持三种格式: 明文、md5 哈希、SCRAM-SHA-256 密钥
- 使用 scram-sha-256 时，明文密码即可，pgBouncer 会自动处理 SCRAM 握手

### 4.3 APISIX Standalone 模式

```yaml
# docker-compose.yml
environment:
  APISIX_STAND_ALONE: "true"   # 启用声明式配置
```

```yaml
# apisix.yaml - 路由中引用插件元数据
plugins:
  jwt-auth: {}   # 空对象 = 引用 Admin API 配置的 jwt-auth 元数据
```

**要点**:
- `APISIX_STAND_ALONE: "true"` 表示使用本地 YAML 文件作为配置源
- 路由中的 `jwt-auth: {}` 引用通过 Admin API (`setup_apisix.sh`) 配置的插件元数据
- 不再需要 etcd

### 4.4 JWT 认证配置

**开发环境** (HS256 对称密钥):
```env
JWKS_JSON={"keys":[{"kty":"oct","kid":"dev-hs256","alg":"HS256","k":"c2VjcmV0X2RldmVsb3BtZW50X2tleV9hdF9sZWFzdF9zZXZlbl9jaGFyYWN0ZXJzIQ=="}]}
```

**生产环境** (RS256 + Casdoor JWKS):
- 通过 `setup_apisix.sh` 自动从 Casdoor 获取 JWKS
- 插件元数据类型: `{"algorithm": "RS256", "key": "<jwks-json>"}`

### 4.5 Redis 配置

```conf
# deploy/redis.conf
bind 127.0.0.1
port 6379
appendonly yes
maxmemory 256mb
maxmemory-policy allkeys-lru
```

**Docker 中的 Redis**:
- `docker-compose.yml` 中新增 Redis 服务 (端口 6379)
- APISIX 限流使用 Docker Redis (`redis_host: app-redis`)
- 如需使用主机 Redis，需修改 `bind` 为 `0.0.0.0` (开发环境)

### 4.6 Docker Compose 网络

```yaml
networks:
  app-net:
    driver: bridge
    ipam:
      config:
        - subnet: 172.20.0.0/16
```

- 所有 Docker 服务通过 `app-net` 互相访问
- 主机通过 `host.docker.internal` 访问 WSL2 中的服务

## 六、一键启动脚本说明

### 6.1 start.sh 功能

`start.sh` 负责**启动已有服务**，执行以下操作：

1. 启动 PostgreSQL (Pigsty)
2. 启动 pgBouncer (Pigsty)
3. 启动 Redis (Pigsty)
4. 启动 etcd (Pigsty)
5. 启动 Grafana (Pigsty)
6. 检查 Docker 状态
7. 启动 Docker Compose 服务
8. 验证所有服务健康状态

### 6.2 start.sh 不负责初始部署

**`start.sh` 不会自动执行以下初始化操作**：

- ❌ Pigsty 首次部署 (`./deploy.yml`)
- ❌ PostgreSQL 用户/数据库创建
- ❌ PostgreSQL 扩展安装
- ❌ APISIX 路由和插件配置 (`setup_apisix.sh`)
- ❌ Casdoor 应用配置
- ❌ 数据库 SQL 脚本应用 (`apply-src.sh`)

### 6.3 首次部署完整流程

如果是全新环境，请按以下顺序执行：

```bash
# 1. 安装 Pigsty
curl -fsSL https://pigsty.cc/get | bash -s v4.4.0
cd ~/pigsty
./configure -i $(hostname -I | awk '{print $1}') -n -s

# 2. 编辑 pigsty.yml 配置模块
vim pigsty.yml

# 3. 部署所有模块
./deploy.yml

# 4. 部署 etcd
./etcd.yml

# 5. 创建 PostgreSQL 用户和数据库 (参考 scripts/README.md)
# 6. 安装扩展
# 7. 配置 pgBouncer
# 8. 启动 pgBouncer

# 9. 启用 Docker Desktop WSL2 集成
# 10. 复制环境变量
cp .env.example .env

# 11. 启动 Docker Compose
docker compose up -d

# 12. 初始化 APISIX 路由和插件
./scripts/setup_apisix.sh

# 13. 一键启动 (后续日常使用)
./scripts/start.sh
```

## 八、常见问题与注意事项

### 7.1 端口冲突

- **Grafana (3000)** 与 PostgREST 不能共用同一端口
- 如安装其他服务 (如 Redis on Docker)，注意不要与 WSL2 中的 Redis (6379) 冲突

### 7.2 密码安全

| 环境 | 建议 |
|:---|:---|
| 开发环境 | 可使用默认密码，但不要提交到 Git |
| 生产环境 | 必须使用强密码替换所有默认值 |
| APISIX Admin Key | 生成随机值: `openssl rand -hex 16` |

### 7.3 Docker Desktop WSL2 集成

确保在 Docker Desktop → Settings → Resources → WSL Integration 中启用 Ubuntu-26.04。

### 7.4 认证链

```
客户端 → APISIX (JWT验证) → PostgREST (DB查询) → pgBouncer → PostgreSQL
```

- APISIX 验证 JWT (HS256/RS256)
- PostgREST 使用 `authenticator` 角色连接 pgBouncer (:6432)
- pgBouncer 使用 scram-sha-256 连接 PostgreSQL (:5432)
- 所有环节认证方法必须一致

### 7.5 Casdoor 数据库

Casdoor 需要自己的数据库 (`casdoor`)，由 Pigsty 的 `pg_databases` 配置自动创建。

### 7.6 离线部署

如需离线部署:
1. 在有网络的环境中运行 `./infra.yml` 构建本地软件源
2. 打包 `/www/pigsty` 目录
3. 在离线环境中使用离线软件包

## 十一、故障排查

### PostgreSQL 连接失败

```bash
# 检查 HBA 规则
grep -v '^#' /etc/postgresql/18/main/pg_hba.conf | grep -v '^$'

# 检查监听地址
grep listen_addresses /etc/postgresql/18/main/postgresql.conf

# 重启
systemctl restart postgresql@18-main
```

### pgBouncer 认证失败

```bash
# 检查认证方法是否匹配
grep auth_type /etc/pgbouncer/pgbouncer.ini
grep "scram-sha-256\|md5" /etc/postgresql/18/main/pg_hba.conf

# 重启 pgbouncer
pkill pgbouncer
sudo -u postgres /usr/sbin/pgbouncer /etc/pgbouncer/pgbouncer.ini &
```

### Docker 容器无法连接主机服务

```bash
# 检查 extra_hosts 配置
docker exec app-apisix cat /etc/hosts | grep host.docker.internal

# 检查主机服务是否监听 0.0.0.0
ss -tlnp | grep 6432
```

### APISIX JWT 验证失败

```bash
# 检查插件元数据是否配置
curl -s http://localhost:9180/apisix/admin/plugin_metadata/jwt-auth \
  -H "X-API-KEY: edd1c9f034335f136f87ad84b625c8f1"

# 重新初始化
./scripts/setup_apisix.sh
```
