#### Pages (前端)

```toml

  { pattern = "omnipg.com/*", zone_name = "omnipg.com" }
]

[vars]
API_BASE_URL = "https://api.omnipg.com"
CASDOOR_ENDPOINT = "https://casdoor.omnipg.com"
```

#### Workers (边缘逻辑)

```javascript
// workers/api-gateway.js
export default {
  async fetch(request, env) {
    const url = new URL(request.url);
    
    // 1. JWT 验证 (边缘层)
    const token = request.headers.get('Authorization');
    if (token) {
      const isValid = await verifyJWT(token, env.JWT_SECRET);
      if (!isValid) {
        return new Response('Unauthorized', { status: 401 });
      }
    }
    
    // 2. 限流
    const clientIP = request.headers.get('CF-Connecting-IP');
    const isRateLimited = await checkRateLimit(clientIP, env.KV);
    if (isRateLimited) {
      return new Response('Rate Limited', { status: 429 });
    }
    
    // 3. 缓存检查
    const cache = caches.default;
    let response = await cache.match(request);
    if (response) return response;
    
    // 4. 回源
    response = await fetch(request);
    
    // 5. 缓存写入
    if (response.status === 200) {
      ctx.waitUntil(cache.put(request, response.clone()));
    }
    
    return response;
  }
};
```

#### R2 (图片存储)

```bash
# 创建 Bucket
r2 bucket create omnipg-images

# 图片上传流程
1. 用户上传图片 → 源站 API
2. 源站压缩/处理 → 写入 R3
3. 返回 R2 URL → 前端展示
4. CDN 缓存 R2 内容

# 图片 URL 结构
https://images.omnipg.com/attractions/{id}/thumbnail.webp   # 缩略图
https://images.omnipg.com/attractions/{id}/original.webp    # 原图 (会员)
https://images.omnipg.com/users/{id}/avatar.webp            # 头像
```

#### Stream (视频)

```bash
# 视频上传
curl -X PUT "https://api.cloudflare.com/client/v4/accounts/{account}/stream" \
  -H "Authorization: Bearer {token}" \
  -F "file=@video.mp4"

# 视频播放
# 自动转码为 HLS 自适应码率
# 支持 360p/720p/1080p 自动切换
```

#### KV (边缘缓存)

```javascript
// 缓存热门景点
await env.KV.put('attraction:hot:1', JSON.stringify(data), {
  expirationTtl: 3600  // 1小时
});

// 缓存评论列表
await env.KV.put('comments:attraction:123:page:1', JSON.stringify(data), {
  expirationTtl: 300   // 5分钟
});
```

### 6.3 Gorse 内存问题分析

根据 GitHub Issues 和社区反馈，Gorse 存在内存持续增长问题：

| 问题类型 | 描述 | 影响 | 解决方案 |
|:---|:---|:---|:---|
| **内存泄漏** | 模型训练后内存未释放 | 长时间运行后 OOM | 定期重启 Worker |
| **缓存膨胀** | Redis 缓存无限增长 | 内存不足 | 设置 TTL + LRU 淘汰 |
| **大模型加载** | 深度学习模型占用大量内存 | 推理延迟增加 | 模型量化 + 分批加载 |
| **并发问题** | 高并发时内存峰值 | 服务崩溃 | 限流 + 内存限制 |

**内存使用估算**：

| 数据规模 | 用户数 | 物品数 | 交互数 | 内存需求 |
|:---|:---|:---|:---|:---|
| 小型 | 10万 | 100万 | 1000万 | 4-8 GB |
| 中型 | 100万 | 500万 | 1亿 | 16-32 GB |
| 大型 | 1000万 | 1000万 | 5亿 | 64-128 GB |
| 超大型 | 5000万 | 5000万 | 20亿 | 256 GB+ |

## 八、性能瓶颈与解决方案

### 8.1 评论写入瓶颈

**问题**: 峰值 648条/秒，单 PG 写入上限约 2000-3000 TPS

**解决方案**:

| 方案 | 效果 | 复杂度 |
|:---|:---|:---|
| **批量写入** | 10x 提升 | 低 |
| **异步队列** | 100x 提升 | 中 |
| **分片** | 线性扩展 | 高 |

**推荐**: 异步队列 (Redis Streams / Kafka)

```
用户发布评论 → Redis Streams → Consumer 批量写入 PG
                (峰值缓冲)     (100条/批, 10ms间隔)
```

### 8.2 Argon2id 认证瓶颈

**问题**: Argon2id ~100ms/次，高并发时成为瓶颈

**解决方案**:

| 方案 | 效果 | 说明 |
|:---|:---|:---|
| **JWT 14天有效期** | 减少 90% 认证 | 仅首次登录需要 Argon2id |
| **OAuth 优先** | 减少 50% 认证 | Google/Apple 登录跳过密码 |
| **Casdoor 集成** | 统一认证 | 密码由 Casdoor 处理 |
| **边缘 JWT 验证** | 减少源站压力 | Workers 层验证 |

### 8.3 嵌套评论查询

**问题**: Reddit 型递归查询慢

**解决方案**:

| 方案 | 效果 | 说明 |
|:---|:---|:---|
| **LTREE 路径枚举** | 10x 提升 | 预计算路径 |
| **物化路径** | 5x 提升 | 存储完整路径 |
| **缓存热门评论** | 100x 提升 | KV 缓存前N页 |

### 8.4 Gorse 内存瓶颈

**问题**: 内存持续增长，可能导致 OOM

**解决方案**:

| 方案 | 效果 | 说明 |
|:---|:---|:---|
| **内存限制** | 防止 OOM | Docker `--memory` 限制 |
| **定期重启** | 释放内存 | Cron 每天凌晨重启 Worker |
| **LRU 缓存** | 限制缓存大小 | `GORSE_CACHE_SIZE` |
| **模型量化** | 减少 50% 内存 | 使用轻量级模型 |
| **监控告警** | 及时发现 | Prometheus + Grafana |
