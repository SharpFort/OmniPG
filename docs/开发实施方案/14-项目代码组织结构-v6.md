## 1. 核心架构策略：PostgreSQL Schema 级物理隔离

### 1.1 业务划分设计

在 Database-Driven 架构中，我们**直接使用 PostgreSQL 的 Schema（模式/命名空间）作为业务模块的物理边界**。

| Schema 名 | 职责 | 存储内容 | 访问权限 |
|:---|:---|:---|:---|
| `public` | 共享域 | RBAC 表（`sys_*`）、会话表、黑名单表 | 所有业务域可引用 |
| `sales` | 销售业务域 | 订单表、订单明细表、优惠券表等 | 仅销售域逻辑可修改 |
| `inventory` | 库存业务域 | 库存表、仓库表、入库单表等 | 仅库存域逻辑可修改 |
| `api_v1` | API 暴露层 | 面向前端的视图和 RPC 函数（**不放物理表**） | PostgREST 唯一暴露的 Schema |

### 1.2 为什么 `<schema>` 命名而不是 `<module>`？

`db/src/<schema>/` 中的 `<schema>` 指代 PostgreSQL 数据库的 Schema 名，不是业务上的"模块"概念。这种命名有以下优势：

| 维度 | `<schema>` | `<module>` |
|:---|:---|:---|
| 映射关系 | 与数据库 Schema 一一对应，开发时一目了然 | 需要额外映射表才能对应到数据库 |
| 权限控制 | `GRANT USAGE ON SCHEMA public TO app_user` | 无直接权限控制能力 |
| 搜索路径 | `SET search_path TO api_v1, public` | 无对应机制 |
| PostgREST | `db-schemas = "api_v1"` 直接对应 | 需要额外配置映射 |

### 2.2 Schema 隔离详细方案

#### 2.2.1 架构设计

```sql
-- 1. 为每个大租户创建独立 Schema
CREATE SCHEMA tenant_acme;
CREATE SCHEMA tenant_globex;

-- 2. 在租户 Schema 内创建相同的表结构
CREATE TABLE tenant_acme.orders (...);
CREATE TABLE tenant_globex.orders (...);

-- 3. 创建数据库角色
CREATE ROLE tenant_acme_user LOGIN PASSWORD '***';
CREATE ROLE tenant_globex_user LOGIN PASSWORD '***';

-- 4. 授予 Schema 级权限
GRANT USAGE ON SCHEMA tenant_acme TO tenant_acme_user;
GRANT ALL ON ALL TABLES IN SCHEMA tenant_acme TO tenant_acme_user;

GRANT USAGE ON SCHEMA tenant_globex TO tenant_globex_user;
GRANT ALL ON ALL TABLES IN SCHEMA tenant_globex TO tenant_globex_user;

-- 5. 撤销跨租户访问
REVOKE ALL ON SCHEMA tenant_globex FROM tenant_acme_user;
REVOKE ALL ON SCHEMA tenant_acme FROM tenant_globex_user;
```

#### 2.2.2 应用层连接策略

```go
// Policy Syncer 中的连接池管理
type TenantPoolManager struct {
    pools map[string]*sql.DB // tenant_id -> connection pool
}

func (m *TenantPoolManager) GetConnection(tenantID string) (*sql.DB, error) {
    // 根据 tenant_id 选择对应的 Schema 和连接
    schema := fmt.Sprintf("tenant_%s", tenantID)
    connStr := fmt.Sprintf("search_path=%s", schema)
    return sql.Open("postgres", connStr)
}
```

#### 2.2.3 Schema 隔离下的 PostgREST 配置

```ini
# 方案 1：每个租户独立 PostgREST 实例
# tenant_acme 的 PostgREST
db-schemas = "tenant_acme"

# tenant_globex 的 PostgREST  
db-schemas = "tenant_globex"

# 方案 2：共享 PostgREST + 动态 search_path
db-schemas = "api_v1"
db-extra-search-path = "public"

-- 通过 JWT claim 动态切换 Schema
CREATE OR REPLACE FUNCTION set_tenant_schema()
RETURNS void AS $$
DECLARE
    v_tenant_id text;
BEGIN
    v_tenant_id := current_setting('request.jwt.claims', true)::json->>'tenant_id';
    EXECUTE format('SET search_path TO tenant_%s, public', v_tenant_id);
END;
$$ LANGUAGE plpgsql;
```

## 3. 物理文件组织策略

### 3.1 DDL（建表/改表）文件组织原则

**规则：** "按业务 Feature/模块拆分，关联表放一起"

| 做法 | 说明 |
|:---|:---|
| ❌ 100 张表塞一个文件 | 无法协作，冲突频繁 |
| ❌ 1 张表拆 1 个文件 | 外键关联的表被拆散，难以理解 |
| ✅ 关联表放同一迁移文件 | 订单表 + 订单明细表 = 同一文件 |

### 3.2 migrations 路径下添加模块文件夹

```
db/migrations/
├── public/                          # RBAC 权限域迁移
│   ├── 20260707000001_init_tables.sql

│   ├── 20260707000003_create_casbin_view.sql

│   ├── 20260707000005_create_auth_functions.sql
│   ├── 20260707000006_create_permission_functions.sql
│   ├── 20260707000007_create_security_triggers.sql
│   ├── 20260707000008_enable_rls_policies.sql
```

### 3.3 DDL 与 src 的边界划分（修订版）

| 类型 | 存放位置 | 管理方式 | 修改方式 |
|:---|:---|:---|:---|
| **物理表 DDL** | `db/migrations/<schema>/` 时间戳文件 | Dbmate 增量迁移，**只增不改** | 新建迁移文件 |
| **索引** | `db/migrations/<schema>/` 时间戳文件 | Dbmate 增量迁移，**只增不改** | 新建迁移文件 |
| **视图** | `db/src/<schema>/views/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **函数** | `db/src/<schema>/functions/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **触发器** | `db/src/<schema>/triggers/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **种子数据** | `db/migrations/<schema>/` 时间戳文件 | Dbmate 增量迁移 | 新建迁移文件 |
| **RLS 策略** | `db/migrations/<schema>/` 时间戳文件 | Dbmate 增量迁移 | 新建迁移文件 |

> **关键规则：** 同一个对象（如 `casbin_rule` 视图）**只在一个地方定义**。如果已在 `db/migrations/` 中创建了首次定义，后续所有修改必须通过 `db/src/` 幂等源码覆盖。

## 4. 从 migrations 到 src 的迁移策略

### 4.1 为什么要迁移？

**当前问题：** 视图、函数、触发器混在 migrations 中，导致：
- 一个函数修改 20 次产生 20 个迁移文件，无法看到当前最新源码
- Git 协作时频繁冲突
- 无法独立测试单个函数

**目标：** DDL 用 migrations 管理（有版本顺序），PL/pgSQL 用 src 管理（幂等源码）。

### 4.2 迁移步骤

```
步骤 1：创建 db/src/ 目录结构
步骤 2：将现有 migrations 中的函数/视图/触发器提取到 db/src/
步骤 3：在 migrations 中只保留 CREATE 首次定义（用于新环境部署）
步骤 4：开发时修改 db/src/ 中的文件，通过 make apply-src 部署
```

### 4.3 提取示例

**原 migration 文件（混合）：**
```sql
-- 20260707000005_create_auth_functions.sql
-- migrate:up

-- DDL 部分（保留在 migration）
CREATE TABLE sys_user_session (...);

-- PL/pgSQL 部分（提取到 db/src/）
CREATE OR REPLACE FUNCTION user_login_sso(...)
... 100 行代码 ...
```

**迁移后：**

```sql
-- 20260707000005_create_auth_functions.sql（只保留 DDL）
-- migrate:up
CREATE TABLE sys_user_session (...);
```

```sql
-- db/src/public/functions/user_login_sso.sql（幂等源码）
CREATE OR REPLACE FUNCTION user_login_sso(...)
... 100 行代码 ...
```

### 4.4 Makefile 更新

```makefile
# 应用 db/src 中的幂等源码（函数/视图/触发器）
apply-src:
	@for f in $(shell find db/src -name "*.sql" | sort); do \
		echo "Applying $$f..."; \
		docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
	done
```

```
        └── change_password.sql      # 修改密码
```

### 5.2 单函数单文件规范

**规则：** 一个视图 = 一个文件，一个 RPC 函数 = 一个文件

**优势：**
- Git 协作不冲突
- 独立测试和版本管理
- 代码审查时一目了然

**文件命名规范：**
- 视图：`<业务域>_<功能>.sql`（如 `orders_my_orders.sql`）
- RPC：`<动词>_<名词>.sql`（如 `create_order.sql`）

### 5.3 文件模板

**视图文件模板（`db/src/api_v1/views/orders/my_orders.sql`）：**
```sql
-- 视图：个人订单列表
-- 描述：查询当前用户的订单列表，包含订单基本信息和状态
-- 创建时间：2026-07-21
-- 最后修改：2026-07-21

CREATE OR REPLACE VIEW api_v1.my_orders AS
SELECT 
    o.id,
    o.order_no,
    o.total_amount,
    o.status,
    o.created_at,
    o.updated_at
FROM sales.orders o
WHERE o.tenant_id = current_tenant_id()
  AND o.user_id = current_user_id()
  AND o.deleted_at IS NULL;

COMMENT ON VIEW api_v1.my_orders IS '个人订单列表（自动过滤租户和用户）';
```

**RPC 函数文件模板（`db/src/api_v1/rpc/orders/create_order.sql`）：**
```sql
-- RPC：创建订单
-- 描述：创建新订单并扣减库存
-- 创建时间：2026-07-21
-- 最后修改：2026-07-21

CREATE OR REPLACE FUNCTION api_v1.create_order(
    p_items jsonb,
    p_remark text DEFAULT NULL
)
RETURNS json AS $$
DECLARE
    v_order_id uuid;
    v_order_no varchar;
    v_total_amount numeric;
BEGIN
    -- 1. 生成订单号
    v_order_no := 'ORD' || to_char(now(), 'YYYYMMDD') || substr(md5(random()::text), 1, 8);
    
    -- 2. 计算总金额
    SELECT COALESCE(SUM((item->>'price')::numeric * (item->>'quantity')::int), 0)
    INTO v_total_amount
    FROM jsonb_array_elements(p_items) AS item;
    
    -- 3. 创建订单
    INSERT INTO sales.orders (order_no, user_id, tenant_id, total_amount, remark, status)
    VALUES (v_order_no, current_user_id(), current_tenant_id(), v_total_amount, p_remark, 'pending')
    RETURNING id INTO v_order_id;
    
    -- 4. 创建订单明细
    INSERT INTO sales.order_items (order_id, product_id, quantity, price)
    SELECT v_order_id, (item->>'product_id')::uuid, (item->>'quantity')::int, (item->>'price')::numeric
    FROM jsonb_array_elements(p_items) AS item;
    
    -- 5. 扣减库存
    PERFORM inventory.deduct_stock(p_items);
    
    RETURN json_build_object(
        'order_id', v_order_id,
        'order_no', v_order_no,
        'total_amount', v_total_amount
    );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

COMMENT ON FUNCTION api_v1.create_order(jsonb, text) IS '创建订单（含库存扣减）';
```
