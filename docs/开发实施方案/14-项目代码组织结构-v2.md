# 14-项目代码组织结构-v2

> **版本：** v2.0  
> **日期：** 2026-07-22  
> **项目：** OmniPG 零后端代码统一权限管理平台  
> **架构：** Pigsty + PostgreSQL + PostgREST + APISIX + Casdoor + Go Syncer  
> **变更说明：** 从 v1 的「表前缀隔离」升级为「多 Schema 隔离 + 目录映射」，获得更清晰的模块边界、权限隔离和 API 文档分组。

---

## 1. 项目整体架构组织哲学

这套项目结构的底层逻辑是：**"配置即代码，数据库即代码，一切皆可版本化"**。

1. **Pigsty 层（基础设施基座）：** 提供 PostgreSQL 集群、监控、高可用等基础能力
2. **APISIX 层（声明式配置）：** 存放网关路由和 Lua Serverless 脚本
3. **PostgREST 层（声明式配置）：** 存放 API 暴露策略与密钥映射
4. **Database 层（版本化迁移）：** 存放数据表、视图定义、PL/pgSQL 业务函数
5. **Syncer 层（Go 独立模块）：** 存放 Policy Syncer 的 Go 源码和单元测试
6. **Test 体系（双层测试）：** pgTAP 测试数据库逻辑 + Go test 测试 Syncer + E2E 全链路测试

## 2. 核心架构策略：多 Schema 模块隔离

在无后端（Database-Driven）架构中，我们**使用 PostgreSQL Schema 作为业务模块的逻辑边界**，实现比表前缀更彻底的隔离。

### 2.1 业务模块 Schema 划分

| Schema | 模块 | 说明 | 示例表 |
|:---|:---|:---|:---|
| `sys` | 权限域 | RBAC 核心表、会话、黑名单 | `sys.user`, `sys.role`, `sys.api` |
| `sales` | 销售域 | 订单、账单、客户 | `sales.orders`, `sales.order_items` |
| `inventory` | 库存域 | 库存、仓储、物流 | `inventory.stock`, `inventory.warehouse` |
| `hr` | 人力资源 | 员工、部门、考勤 | `hr.employees`, `hr.departments` |
| `finance` | 财务域 | 账户、流水、发票 | `finance.accounts`, `finance.transactions` |

### 2.2 跨模块数据关联

使用 Schema 隔离后，跨模块关联使用 `schema.table` 命名：

```sql
-- 跨模块查询（schema.table 关联）
SELECT o.id, o.amount, u.username
FROM sales.orders o
JOIN sys.user u ON o.user_id = u.id;

-- 跨模块外键（直接引用）
ALTER TABLE sales.orders 
ADD CONSTRAINT fk_user 
FOREIGN KEY (user_id) REFERENCES sys.user(id);
```

### 2.3 租户隔离方案

| 隔离级别 | 实现方式 | 适用场景 |
|:---|:---|:---|
| **行级隔离（推荐）** | `tenant_id` + RLS 策略 | 多租户共享数据库，性能高 |
| **数据库级隔离** | 每个租户独立数据库 | 金融/医疗等高隔离要求 |

```sql
-- 行级隔离示例
ALTER TABLE sales.orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON sales.orders
    USING (tenant_id = current_setting('app.current_tenant')::uuid);
```

> **决策说明：** Schema 隔离与租户隔离是不同维度的事情。Schema 用于模块边界，RLS 用于租户边界，两者完全兼容（Supabase 就是这种模式）。

### 2.4 Schema 隔离 vs 表前缀隔离对比

| 维度 | Schema 隔离（v2 采用） | 表前缀隔离（v1 废弃） |
|:---|:---|:---|
| **命名冲突** | ✅ 完全隔离，不同 Schema 允许同名对象 | 通过前缀规范避免 |
| **权限控制** | Schema 级 GRANT，批量授权方便 | 表级 GRANT，需逐表授权 |
| **跨域查询** | 需要 `schema.table` | 直接表名 |
| **迁移复杂度** | 中等（需指定 Schema） | 较低 |
| **PostgREST 暴露** | 多 Schema 暴露，URL 自动分组 | 单 Schema 暴露 |
| **Swagger 文档** | ✅ 自动按 Schema 分组 Tags | 需手动配置或混在一起 |
| **参考案例** | Supabase, Hasura, 现代 SaaS | Odoo, Nexterp, ERPNext |
| **适合规模** | 中大型（5-50 模块） | 中大型（10+ 模块） |
| **代码可读性** | `sales.orders` 语义清晰 | `sales_orders` 也可接受 |

## 3. 物理文件组织策略

### 3.1 DDL（建表/改表）文件组织

**规则：** "按业务模块分子目录，关联表放一起"

- 每个模块一个子目录：`migrations/sys/`, `migrations/sales/`
- 强关联的主子表应放在同一个迁移文件中
- Dbmate 可读取任意目录结构，方便按模块管理迁移文件

### 3.2 PL/SQL（函数/触发器/视图）文件组织

**正确做法：** "Idempotent Source Code（幂等源码化管理）"

在 `/db/src/` 目录下，**按模块分子目录**，模块内再按类型（schemas/functions/views/triggers）存放。

### 3.3 DDL 与 src 的边界划分

| 类型 | 存放位置 | 管理方式 | 修改方式 |
|:---|:---|:---|:---|
| **物理表 DDL** | `db/migrations/{module}/` 时间戳文件 | Dbmate 增量迁移，只增不改 | 新建迁移文件 |
| **索引** | `db/migrations/{module}/` 时间戳文件 | Dbmate 增量迁移，只增不改 | 新建迁移文件 |
| **自定义类型** | `db/src/{module}/types/` | `CREATE TYPE`，幂等源码 | 直接修改文件 |
| **视图** | `db/src/{module}/views/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **函数** | `db/src/{module}/functions/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **触发器** | `db/src/{module}/triggers/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **权限 GRANT** | `db/src/{module}/privileges/` | 独立 SQL 文件 | 直接修改文件 |
| **种子数据** | `db/migrations/{module}/` 时间戳文件 | Dbmate 增量迁移 | 新建迁移文件 |

> **决策说明：** `db/src/` 按模块分目录，模块内按类型分。这样每个模块自包含，开发人员修改某个模块时只需关注对应目录，减少认知负担和 Git 冲突。

### 3.4 Extensions 管理策略

采用 **"Pigsty 层预装 + 迁移文件显式启用"** 的双保险模式：

| 层级 | 职责 | 说明 |
|:---|:---|:---|
| **Pigsty 基础设施层** | 预装扩展共享库 | Pigsty 在 PostgreSQL 集群层面安装 `pgcrypto`、`pgtap` 等扩展的.so 文件和 SQL 文件 |
| **应用迁移文件** | 显式 `CREATE EXTENSION` | 在迁移文件中执行 `CREATE EXTENSION IF NOT EXISTS pgcrypto;`，确保数据库自包含和可迁移 |

**为什么需要迁移文件显式启用：**

1. **多数据库场景：** Pigsty 可能管理多个数据库，扩展是按数据库级别启用的。Pigsty 预装只是让扩展"可用"，但具体到某个数据库仍需 `CREATE EXTENSION`。
2. **CI/CD 可测试：** CI 环境可能使用精简版 PostgreSQL（不含 Pigsty），迁移文件中的 `CREATE EXTENSION` 确保测试能通过。
3. **版本控制：** 扩展版本变更（如 `pgcrypto` 1.3→1.4）可通过迁移文件追踪。

**示例迁移文件（启用扩展）：**

```sql
-- 20260707000000_enable_extensions.sql
-- 在 init_tables 之前执行

-- pgcrypto: 密码哈希、gen_random_uuid() 等
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- pg_trgm: 模糊搜索（可选，按需启用）
-- CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- pgvector: 向量搜索（可选，按需启用）
-- CREATE EXTENSION IF NOT EXISTS vector;
```

```sql
-- 20260707000000_enable_pgtap_test.sql
-- 仅限测试环境执行（通过 DBMATE_ENV=test 条件加载）

-- pgTAP: 数据库单元测试框架
CREATE EXTENSION IF NOT EXISTS pgtap;
```

**pgTAP 环境隔离策略：**

- 生产/预发布环境：不安装 pgtap，避免测试框架引入安全风险
- 开发/CI 环境：通过独立迁移文件或条件脚本启用 pgtap
- `db/extensions/pgtap.md` 记录版本信息，与 Pigsty 预装版本保持一致

### 3.5 业务视图 vs API 视图：职责区分

目录结构中存在两个"视图"目录，职责不同，避免混淆：

| 目录 | 职责 | 使用场景 | 可见性 |
|:---|:---|:---|:---|
| `db/src/{module}/views/` | **内部业务视图** | 简化复杂查询，供 PL/pgSQL 函数、触发器、后台服务（Syncer）内部使用 | 仅数据库内部可见，**不暴露为 API** |
| `db/api_v1/{module}/views/` | **API 暴露视图** | 面向最终 API 消费者，经过脱敏、格式化、权限过滤，专为前端展示优化 | PostgREST 自动暴露为 HTTP GET 接口 |

**判断标准：**

- 如果一个视图是给**其他 SQL 函数/触发器**调用的中间对象 → 放在 `db/src/{module}/views/`
- 如果一个视图是给**前端/外部消费者**直接查询的数据接口 → 放在 `db/api_v1/{module}/views/`

**示例：**

```sql
-- db/src/sys/views/v_user_list.sql
-- 内部视图：包含敏感字段（password_hash, email），供业务函数使用
CREATE OR REPLACE VIEW sys.v_user_list AS
SELECT id, username, email, password_hash, status, created_at
FROM sys.user
WHERE deleted_at IS NULL;
```

```sql
-- db/api_v1/sys/views/v_user_list.sql
-- API 视图：已脱敏（排除 password_hash），仅暴露前端需要的字段
CREATE OR REPLACE VIEW api_v1_sys.v_user_list AS
SELECT id, username, status, created_at
FROM sys.v_user_list;  -- 基于内部视图二次加工
```

> **准则：** `db/api_v1/` 下的所有对象是"对外 API 契约"，变更需考虑兼容性；`db/src/` 下的对象是"内部实现"，可自由重构。

## 5. PostgREST 多 Schema 暴露策略

### 5.1 Schema 命名规则

PostgREST 暴露的 API Schema 采用 `api_v1_{module}` 命名约定：

| API Schema | 对应业务 Schema | 说明 |
|:---|:---|:---|
| `api_v1_sys` | `sys` | 权限域 API |
| `api_v1_sales` | `sales` | 销售域 API |
| `api_v1_inventory` | `inventory` | 库存域 API |

### 5.2 PostgREST 配置

```ini
# postgrest.conf
db-uri = "postgres://postgres@localhost:5432/app_db"
db-schemas = "api_v1_sys, api_v1_sales, api_v1_inventory"
db-extra-search-path = "public"

# JWT 设置

```

### 5.3 URL 映射规则

整个请求链路存在两层路径映射，需要明确区分：

```
客户端请求                       APISIX 网关                    PostgREST
─────────────────────────────────────────────────────────────────────────
/api/v1/sys/v_user_list   ──→   /api_v1_sys/v_user_list   ──→   api_v1_sys.v_user_list
                               (路径重写)                       (Schema.对象)
```

**关键说明：**

1. **PostgREST 原生路径 = Schema 名称**：PostgREST 默认将 `db-schemas` 中的 Schema 名直接作为 URL 前缀。例如 `api_v1_sys` Schema 下的 `v_user_list` 视图，PostgREST 原生暴露路径为 `/api_v1_sys/v_user_list`。

2. **`/api/v1/` 前缀由 APISIX 路径重写添加**：为了让 URL 更符合 RESTful 业务语义（`/api/v1/{模块名}/{资源}`），APISIX 在转发前执行路径重写：`/api/v1/sys/*` → `/api_v1_sys/*`。这一规则定义在 `apisix/routes.yaml` 中。

3. **物理目录 ≠ URL 路径**：`db/api_v1/sys/` 是磁盘上的源码存放路径，最终部署到数据库 `api_v1_sys` Schema，URL 路径由 APISIX 决定。三者是「目录→Schema→URL」的映射关系。

| 浏览器请求URL | APISIX 重写后 | 数据库对象 |
|:---|:---|:---|
| `/api/v1/sys/v_user_list` | `/api_v1_sys/v_user_list` | `api_v1_sys.v_user_list` |
| `/api/v1/sys/rpc/user_login` | `/api_v1_sys/rpc_user_login` | `api_v1_sys.rpc_user_login()` |
| `/api/v1/sales/v_order_summary` | `/api_v1_sales/v_order_summary` | `api_v1_sales.v_order_summary` |
| `/api/v1/sales/rpc/create_order` | `/api_v1_sales/rpc_create_order` | `api_v1_sales.rpc_create_order()` |
| `/api/v1/inventory/rpc/deduct_stock` | `/api_v1_inventory/rpc_deduct_stock` | `api_v1_inventory.rpc_deduct_stock()` |

> **示例：APISIX 路径重写规则（apisix/routes.yaml 片段）**
> ```yaml
> uri: /api/v1/sys/*
> plugins:
>   proxy-rewrite:
>     regex_uri: ["^/api/v1/sys/(.*)", "/api_v1_sys/$1"]
> upstream:
>   type: roundrobin
>   nodes:
>     "postgrest:3000": 1
> ```

## 6. 核心模块与代码组织实战

### 6.1 统一工作流的 Makefile

```makefile
# Makefile
.PHONY: dev-env db-migrate db-rollback apply-src apply-api test-db test-syncer test-e2e test-all run-syncer

# 模块加载顺序（基础模块优先）
MODULES_L0 := sys
MODULES_L1 := sales inventory hr finance

# 1. 启动本地开发环境
dev-env:
	docker compose up -d

# 2. 执行数据库向前迁移
db-migrate:
	dbmate up

# 3. 回滚最近一次数据库迁移
db-rollback:
	dbmate rollback --single

# 4. 应用 db/src 中的幂等源码（按依赖顺序：L0 → L1）
apply-src:
	@echo "=== Applying L0: Base modules ==="
	@for mod in $(MODULES_L0); do \
		for f in $(shell find db/src/$$mod -name "*.sql" | sort); do \
			echo "Applying $$f..."; \
			docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
		done; \
	done
	@echo "=== Applying L1: Business modules ==="
	@for mod in $(MODULES_L1); do \
		for f in $(shell find db/src/$$mod -name "*.sql" | sort); do \
			echo "Applying $$f..."; \
			docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
		done; \
	done
	@echo "=== Done ==="

# 5. 应用 API 暴露层（在业务模块之后）
apply-api:
	@echo "=== Applying API exposure layer ==="
	@for f in $(shell find db/api_v1 -name "*.sql" | sort); do \
		echo "Applying API layer: $$f..."; \
		docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
	done

# 6. 运行数据库内部单元测试 (pgTAP)
test-db:
	@echo "Running PostgreSQL PL/pgSQL unit tests via pgTAP..."
	docker compose exec app-postgres pg_prove -U app_owner -d app_db --ext .sql db/tests/

# 7. 运行 Go 同步器的单元测试
test-syncer:
	@echo "Running Policy Syncer unit tests..."
	cd syncer && go test -v ./...

# 8. 运行全链路端到端集成测试
test-e2e:
	@echo "Running end-to-end API integration tests..."
	bash ./scripts/e2e-test.sh

# 9. 一键执行全栈测试 (用于 CI/CD 流水线)
test-all: test-db test-syncer test-e2e

# 10. 启动 Policy Syncer 同步器
run-syncer:
	cd syncer && go run cmd/syncer/main.go
```

> **决策说明：** `migrations/` 按模块分目录。Dbmate 支持读取任意目录结构，模块分目录便于按功能域管理和审查迁移历史。

### 6.3 全链路集成测试（E2E）

E2E（End-to-End）测试是模拟真实用户场景的全链路测试，覆盖从网关到数据库的完整请求流程。

#### 示例：`scripts/e2e-test.sh`

```bash
#!/usr/bin/env bash
set -eo pipefail

GATEWAY_URL="http://localhost:9080"

echo "=================================================="
echo "E2E Test: Starting Full-Stack Permission Workflow"
echo "=================================================="

# 1. 模拟用户登录
echo "Step 1: Simulating user login (Alice)..."
LOGIN_RESP=$(curl -s -X POST "${GATEWAY_URL}/api/v1/sys/rpc/user_login_sso" \
  -H "Content-Type: application/json" \
  -d '{"p_username": "alice", "p_password": "secure_password"}')

ACCESS_TOKEN=$(echo "${LOGIN_RESP}" | grep -o '"access_token":"[^"]*' | grep -o '[^"]*$')
echo "SUCCESS: Logged in. Access Token retrieved."

# 2. 访问受限接口 (正常放行)
echo "Step 2: Accessing restricted API with token..."
API_RESP=$(curl -s -o /dev/null -w "%{http_code}" -X GET "${GATEWAY_URL}/api/v1/sys/v_user_list?limit=1" \
  -H "Authorization: Bearer ${ACCESS_TOKEN}")

if [ "${API_RESP}" -eq 200 ]; then
  echo "SUCCESS: API accessed successfully (Status 200)."
else
  echo "FAILED: API access denied (Status ${API_RESP})."
  exit 1
fi

# 3. 模拟管理员撤销角色
echo "Step 3: Admin revokes Alice's role..."
# 删除操作执行后，触发器在 0.01 秒内自动拉黑当前的 ACCESS_TOKEN

# 4. 再次访问接口 (应该被阻断)
echo "Step 4: Accessing restricted API again (expecting block)..."
API_RESP_AFTER=$(curl -s -o /dev/null -w "%{http_code}" -X GET "${GATEWAY_URL}/api/v1/sys/v_user_list?limit=1" \
  -H "Authorization: Bearer ${ACCESS_TOKEN}")

if [ "${API_RESP_AFTER}" -eq 401 ]; then
  echo "SUCCESS: Token was successfully blacklisted instantly (Status 401)!"
else
  echo "FAILED: Token block failed (Status ${API_RESP_AFTER})."
  exit 1
fi

echo "=================================================="
echo "E2E Test: All tests passed successfully!"
echo "=================================================="
```

## 7. 开发与运维闭环优点

1. **模块边界清晰：** 每个业务模块拥有独立的 Schema、独立的迁移目录、独立的源码目录，修改一个模块不会影响其他模块。

2. **权限隔离安全：** Schema 级 GRANT 比表级 GRANT 更简洁，可以批量授权整个模块的访问权限。

3. **API 文档自动分组：** Swagger UI 自动按 Schema 分组显示接口，前端开发者可以快速定位对应模块的 API。

4. **开发体验顺畅：** 新开发者只需拉取 Git 仓库，安装 Docker，运行 `make dev-env && make db-migrate && make test-all`，即可在本地构建完整环境。

5. **多环境管理：** 通过 `.env.development`、`.env.staging`、`.env.production` 管理不同环境配置。

6. **生产级部署：** `deployments/` 目录支持 Docker 和 K8s 部署，与本地开发环境解耦。

## 8. 多人协作与部署流

### 8.1 物理表变更（Schema 变更）

- 开发 A 执行 `dbmate new sys/add_coupons`，在 `db/migrations/sys/` 下生成迁移文件
- 运行 `dbmate up` 在本地生效
- 提交 PR，CI 自动运行测试
- 合并后 CD 自动部署

### 8.2 业务存储过程开发

- 开发 A 在 `db/src/sales/functions/` 下新建 `apply_coupon.sql`
- 部署时运行 `make apply-src`，自动将整个 `db/src/` 下的 SQL 文件刷入数据库

### 8.3 API 交付

- 开发 A 在 `db/api_v1/sales/views/` 下创建视图，在 `db/api_v1/sales/rpc/` 下创建 RPC 函数
- PostgREST 自动将新视图/函数暴露为 HTTP 接口
- **不需要写任何一行 Go/Java 控制器代码，也不需要停机**

### 8.4 跨模块依赖与加载顺序

**问题：** 模块间存在依赖关系（如 `sales` 模块的视图可能引用 `sys` 模块的表和函数），`apply-src` 按字母顺序执行可能导致依赖未就绪而失败。

**解决方案：** 定义模块加载优先级，基础模块优先部署。

#### 8.4.1 模块依赖层级

| 层级 | 模块 | 说明 | 依赖 |
|:---|:---|:---|:---|
| **L0 基础层** | `sys` | 权限域，提供 RBAC、用户、会话等基础能力 | 无 |
| **L1 业务层** | `sales`, `inventory`, `hr`, `finance` | 业务模块，依赖 L0 | `sys` |
| **L2 暴露层** | `api_v1/*` | API 暴露层，依赖 L0 + L1 | `sys`, `sales`, `inventory` 等 |

#### 8.4.2 加载顺序规则

```
L0 (sys) → L1 (sales, inventory, hr, finance) → L2 (api_v1/*)
```

#### 8.4.3 Makefile 中的有序加载

```makefile
# 模块加载顺序（基础模块优先）
MODULES_L0 := sys
MODULES_L1 := sales inventory hr finance
MODULES_L2 := api_v1

# 4. 应用 db/src 中的幂等源码（按依赖顺序）
apply-src:
	@echo "=== Applying L0: Base modules ==="
	@for mod in $(MODULES_L0); do \
		for f in $(shell find db/src/$$mod -name "*.sql" | sort); do \
			echo "Applying $$f..."; \
			docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
		done; \
	done
	@echo "=== Applying L1: Business modules ==="
	@for mod in $(MODULES_L1); do \
		for f in $(shell find db/src/$$mod -name "*.sql" | sort); do \
			echo "Applying $$f..."; \
			docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
		done; \
	done
	@echo "=== Done ==="

# 5. 应用 API 暴露层（在业务模块之后）
apply-api:
	@echo "=== Applying API exposure layer ==="
	@for f in $(shell find db/api_v1 -name "*.sql" | sort); do \
		echo "Applying API layer: $$f..."; \
		docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
	done
```

#### 8.4.4 开发规范

1. **禁止循环依赖：** L1 业务模块之间不得互相引用（如 `sales` 不应直接调用 `inventory` 的内部函数）。跨模块协作应通过 L0 基础层的中介函数或事件机制。
2. **新增模块时：** 在 `MODULES_L1` 变量中注册，并声明其依赖。
3. **API 暴露层独立：** `api_v1/` 下的视图/函数只引用底层 Schema 的稳定接口，不直接依赖其他业务模块的内部实现。

## 9. 关键设计决策汇总

| 决策点 | 选择 | 原因 |
|:---|:---|:---|
| **模块隔离** | Schema 隔离 | 完全隔离，权限控制简洁，Swagger 自动分组 |
| **租户隔离** | RLS 行级 | 与 Schema 隔离兼容，性能最高，Supabase 采用 |
| **src 目录结构** | 按模块分目录，模块内按类型分 | 模块自包含，减少认知负担和 Git 冲突 |
| **migrations 结构** | 按模块分目录 | Dbmate 支持，便于按功能域审查迁移历史 |
| **单函数单文件** | 是 | 幂等部署粒度清晰，Git 冲突影响最小 |
| **api_v1 组织** | 多 Schema 暴露，按模块建子目录 | PostgREST 自动映射 URL，Swagger 自动分组 |
| **Extensions 管理** | Pigsty 预装 + 迁移文件显式启用 | 数据库自包含、可迁移、CI 可测试 |
| **视图分层** | 内部视图（src）vs API 视图（api_v1） | 内部实现与外部契约分离，支持脱敏和兼容性管理 |
| **加载顺序** | L0 基础层 → L1 业务层 → L2 暴露层 | 避免依赖未就绪，确保幂等部署成功 |
| **Pigsty 扩展** | 只记录版本 | 基础设施层由 Pigsty 管理，应用不重复管理 |
| **Monitoring** | 不纳入代码组织 | Pigsty 自带 Prometheus + Grafana |
