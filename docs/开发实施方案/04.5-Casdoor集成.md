# 04.5 — Casdoor 集成：OAuth/IAM 认证与 JWT 声明映射

> **定位：** 集成 Casdoor IAM 认证服务，配置 APISIX 的 OAuth/OIDC 流程，实现 JWT Claims 到 Casbin 角色的完整映射。本文档是在 00-05 实施计划之外额外增加的集成文档。
> **前置依赖：** 04-网关与同步器（已通过审查）、Casdoor 容器就绪（docker-compose.yml 已配置）
> **产出物：** Casdoor 应用配置完成 + APISIX authz-casdoor 端点 + JWT Claims 映射验证通过
> **预计耗时：** 2-3 小时

## 3. APISIX Casdoor 集成路由配置

### 3.1 获取 Casdoor 的 JWKS 公钥

Casdoor 的 JWKS 端点为 `/.well-known/jwks`（**无 .json 后缀**）：

```bash
# 获取 JWKS 公钥集
curl -s http://localhost:8000/.well-known/jwks | jq .

# 预期输出格式（标准 JWKS）：
# {
#   "keys": [
#     {
#       "kty": "RSA",
#       "kid": "cert-rsa",
#       "use": "sig",
#       "alg": "RS256",
#       "n": "BASE64URL_MODULUS...",
#       "e": "AQAB"
#     }
#   ]
}
```

保存 JWKS JSON 到环境变量：

```bash
# 提取公钥 JSON
export JWKS_JSON=$(curl -s http://localhost:8000/.well-known/jwks)

# 验证 JWKS 结构
echo $JWKS_JSON | jq '.keys | length'  # 应返回 > 0
```

### 3.2 配置 APISIX jwt-auth（使用 JWKS）

使用完整的 JWKS JSON 字符串作为 jwt-auth 的 secret（JWKS 格式包含 `keys` 数组）：

```bash
curl -X PUT http://127.0.0.1:9180/apisix/admin/plugin_metadata/jwt-auth \
  -H "X-API-KEY: edd1c9f034335f136f87ad84b625c8f1" \
  -H "Content-Type: application/json" \
  -d "{
    \"algorithm\": \"RS256\",
    \"key\": \"$JWKS_JSON\"
  }"
```

**验证：**

```bash
curl http://127.0.0.1:9180/apisix/admin/plugin_metadata/jwt-auth \
  -H "X-API-KEY: edd1c9f034335f136f87ad84b625c8f1" | jq '.value.key' | jq '.keys[0].alg'
# 预期输出: "RS256"
```

### 3.3 配置 APISIX authz-casdoor

`authz-casdoor` 插件处理 OAuth 重定向（未认证用户 → Casdoor 登录页 → 回调 APISIX）。

**方式 A：通过 Casdoor 全局配置（推荐）**

```bash
curl -X PUT http://127.0.0.1:9180/apisix/admin/plugin_metadata/casdoor \
  -H "X-API-KEY: edd1c9f034335f136f87ad84b625c8f1" \
  -H "Content-Type: application/json" \
  -d '{
    "casdoor_endpoint": "http://localhost:8000",
    "client_id": "zero-backend-app",
    "client_secret": "YOUR_CLIENT_SECRET",
    "callback_url": "http://localhost:9000/cb"
  }'
```

**方式 B：在路由级别配置（精细控制）**

将 `authz-casdoor` 插件添加到 §2.3 的业务路由 `api-v1` 中：

```bash
curl -X PUT http://127.0.0.1:9180/apisix/admin/routes/api-v1 \
  -H "X-API-KEY: edd1c9f034335f136f87ad84b625c8f1" \
  -H "Content-Type: application/json" \
  -d '{
    "uri": "/api/v1/*",
    "plugins": {
      "authz-casdoor": {
        "casdoor_endpoint": "http://localhost:8000",
        "client_id": "zero-backend-app",
        "client_secret": "YOUR_SECRET",
        "callback_url": "http://localhost:9000/cb"
      },
      "serverless-pre-function": {
        "phase": "rewrite",
        "functions": [
          "return function(conf, ctx)\n    -- ⚠️ 安全警告：此函数仅做 Base64 解码，不验证 JWT 签名\n-- 签名验证由 jwt-auth 插件在 access 阶段执行\n-- 禁止将 X-User-Role/X-User-Id 用于签名验证前的任何信任判定\n    local cjson = require(\"cjson\")\n    local base64 = require(\"ngx.base64\")\n    local auth_header = ngx.req.get_headers()[\"Authorization\"]\n    if auth_header and string.sub(auth_header, 1, 7) == \"Bearer \" then\n        local token = string.sub(auth_header, 8)\n        local parts = {}\n        for part in string.gmatch(token, \"[^.]+\") do\n            table.insert(parts, part)\n        end\n        if #parts >= 2 then\n            local payload_b64 = parts[2]\n            payload_b64 = string.gsub(payload_b64, \"-\", \"+\")\n            payload_b64 = string.gsub(payload_b64, \"_\", \"/\")\n            local rem = #payload_b64 % 4\n            if rem > 0 then\n                payload_b64 = payload_b64 .. string.rep(\"=\", 4 - rem)\n            end\n            local decoded = base64.decode_base64(payload_b64)\n            if decoded then\n                local status, jwt_json = pcall(cjson.decode, decoded)\n                if status and jwt_json then\n                    if jwt_json.roles then\n                        local roles_str = table.concat(jwt_json.roles, \",\")\n                        ngx.req.set_header(\"X-User-Role\", roles_str)\n                    end\n                    if jwt_json.user_id or jwt_json.sub then\n                        ngx.req.set_header(\"X-User-Id\", jwt_json.user_id or jwt_json.sub)\n                    end\n                    if jwt_json.tenant_id then\n                        ngx.req.set_header(\"X-Tenant-Id\", jwt_json.tenant_id)\n                    end\n                end\n            end\n        end\n    end\nend"
        ]
      },
      "jwt-auth": {},
      "authz-casbin": {
        "username": "X-User-Role"
      },
      "cors": {
        "allow_origins": "${CORS_ORIGINS:http://localhost:5173}",
        "allow_methods": "GET,POST,PUT,DELETE,PATCH,OPTIONS",
        "allow_headers": "Authorization,Content-Type,Accept-Profile,Prefer",
        "expose_headers": "Content-Range,Location",
        "allow_credential": true,
        "max_age": 1728000
      },
      "limit-req": {
        "rate": 100,
        "burst": 50,
        "key_type": "var",
        "key": "remote_addr",
        "rejected_code": 429,
        "rejected_msg": "{\"error\":\"rate limit\",\"message\":\"Too many requests\"}"
      }
    },
    "upstream": {
      "type": "roundrobin",
      "nodes": {
        "postgrest_server:3000": 1
      }
    }
  }'
```

### 3.5 创建 JWKS 端点路由（面向 PostgREST）

PostgREST 需要获取 JWKS 公钥验证 JWT。APISIX 提供 Casdoor JWKS 代理路由：

```bash
# 方式 1：通过 Casdoor 直接提供（推荐，Casdoor 自带 JWKS 端点）
# PostgREST 直接连接: http://localhost:8000/.well-known/jwks

# 方式 2：通过 APISIX mocking（如 PostgREST 容器无法直接访问 Casdoor）
curl -X PUT http://127.0.0.1:9180/apisix/admin/routes/jwks \
  -H "X-API-KEY: edd1c9f034335f136f87ad84b625c8f1" \
  -H "Content-Type: application/json" \
  -d '{
    "uri": "/jwks",
    "plugins": {
      "mocking": {
        "content_type": "application/json; charset=utf-8",
        "response_status": 200,
        "response_example": |
          {
            "keys": [
              {
                "kty": "RSA",
                "kid": "cert-rsa",
                "use": "sig",
                "alg": "RS256",
                "n": "AUTO_REPLACE_FROM_CASDOOR",
                "e": "AQAB"
              }
            ]
          }
      }
    }
  }'
```

**重要：** mocking 中的 `n` 值必须与 Casdoor JWKS 端点返回的实际 `n` 一致！自动化做法：使用 Casdoor JWKS 直接作为秘密，不通过 mocking。

## 4. 本地登录回退方案

### 4.1 架构定位

当 Casdoor 不可用或无外部 IdP 配置时，系统回退到本地密码登录模式：

```
Casdoor 可用 → OAuth 流程 → Casdoor 签发 JWT
   ↓ 不可用
本地回退 → PostgREST /rpc/user_login_sso → PL/pgSQL 直接签发 JWT
```

### 4.2 本地登录配置（02 已定义）

02 数据库已创建 `user_login_sso` 函数，它仍然**直接签发 JWT**（使用 SG `sys_secret` 表中的 `jwt_private_key_pem`）。

**关键点：** 本地登录和 Casdoor 共享同一个 JWKS 公钥验证方式，因此 PostgREST 和 APISIX 的 jwt-secret 配置无需变化。

```bash
# 检查 02 的迁移是否已包含 jwt_private_key
psql -d app_db -c "SELECT EXISTS (SELECT FROM information_schema.columns WHERE table_name = 'sys_secret')"
```

### 4.3 回退触发逻辑（前端）

前端 Vue3 应用需要实现认证回退检测：

```typescript
// 伪代码
class AuthService {
  async login(username: string, password: string) {
    // 优先：尝试 Casdoor OAuth 流程
    try {
      const casdoorAvailable = await this.checkCasdoorHealth();
      if (casdoorAvailable) {
        // 重定向到 Casdoor 登录页
        window.location.href = this.buildCasdoorLoginUrl();
        return;
      }
    } catch (e) {
      console.warn('Casdoor unavailable, falling back to local login');
    }

    // 回退：本地 POST 到 PostgREST user_login_sso
    const response = await fetch('http://localhost:9000/rpc/user_login_sso', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ p_username, p_password })
    });

    if (response.ok) {
      const { access_token, username } = await response.json();
      localStorage.setItem('access_token', access_token);
      // refresh_token 在 httpOnly Cookie 中自动处理
    }
  }
}
```

### 4.4 PostgREST 密钥一致性配置

**关键约束：** 如果 Casdoor 管理密钥对，`sys_secret` 表中的 `jwt_private_key_pem` 也需要保持一致，或者废弃本地 JWT 签名，全部走 Casdoor。

**两种模式对比：**

| 模式 | 适用场景 | 密钥管理 |
|:---|:---|:---|
| **A: Casdoor 统一管理**（推荐） | 所有认证必须经过 Casdoor | Casdoor 管理 RSA 密钥对，`sys_secret` 不存储 `jwt_private_key_pem`，本地登录函数也通过 Casdoor API |
| **B: 双路径并存** | 本地登录保留，OAuth 走 Casdoor | 两个系统必须共享同一 RSA 密钥对（公钥一致即可） |

**推荐模式 A 的修改方式：**

修改 02 的 `user_login_sso` 函数，当 Casdoor 可用时调用其 API 换取本地 JWT：

```sql
-- 修改 user_login_sso 在 Casdoor 可用时的行为
CREATE OR REPLACE FUNCTION user_login_sso(p_username text, p_password text)
RETURNS json AS $$
DECLARE
    v_casdoor_available boolean := false;
    v_casdoor_response jsonb;
BEGIN
    -- 检查 Casdoor 是否可用
    SELECT net.http_get INTO v_casdoor_response
    FROM net.http_get('http://casdoor:8000/api/get-user?username=' || p_username)
    LIMIT 1;
    
    IF v_casdoor_response IS NOT NULL THEN
        -- 调用 Casdoor 的内置密码登录 API
        -- 返回由 Casdoor 签发的 JWT
        ... Casdoor API 调用 ...
    END IF;

    -- 否则回退到本地密码验证
    -- ...
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

## 5. JWT Claims 与角色映射

### 5.1 Casdoor JWT Payload 标准结构

Casdoor 签发的 JWT 包含以下关键声明（Claims）：

```json
{
  "iss": "http://localhost:8000",
  "sub": "00000000-0000-0000-0000-100000000001",
  "aud": "zero-backend-app",
  "exp": 1720464000,
  "iat": 1720462200,
  "jti": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "name": "admin",
  "preferredUsername": "admin",
  "email": "admin@example.com",
  "emailVerified": true,
  "phone": "",
  "phoneVerified": false,
  "region": "CN",
  "owner": "built-in",
  "application": "zero-backend-rbac",
  "organization": "built-in",
  "roles": ["role_admin", "role_user"]
}
```

**角色映射核心挑战：** Casdoor 的 `roles` 字段是 Casdoor 组织内角色名，可能不同于 PG 中的 `sys_role.role_code`。需要建立映射。

### 5.2 角色名称映射策略

Casdoor 与 PG 的角色名称**应当保持一致**（最佳实践），但如果不一致时：

**方式 A：名称一致（推荐）**

在 Casdoor 中创建的角色名称与 PG `sys_role.role_code` 完全一致：

```
Casdoor Role Name → PG sys_role.role_code
──────────────────────────────────────────
role_admin     → role_admin
role_editor    → role_editor
role_guest     → role_guest
```

此时 Base64 解码的 JWT 中的 `roles` 数组可直接用于 X-User-Role 头。

**方式 B：映射模式不一致**

使用 APISIX 的 serverless-pre-function 做映射：

```lua
-- 在 serverless-pre-function 中添加映射
local role_mapping = {
  ["admin"] = "role_admin",
  ["editor"] = "role_editor",
  ["guest"] = "role_guest"
}
if jwt_roles then
    local mapped_roles = {}
    for _, r in ipairs(jwt_roles) do
        table.insert(mapped_roles, role_mapping[r] or r)
    end
    ngx.req.set_header("X-User-Role", table.concat(mapped_roles, ","))
end
```

### 5.3 Claims 到 HTTP Header 映射

04 文档中 serverless-pre-function 的映射逻辑：

| JWT Claim | HTTP Header | Casbin 用途 |
|:---|:---|:---|
| `roles`（数组 → 逗号字符串） | `X-User-Role` | Casbin `r.sub` |
| `sub` 或 `user_id` | `X-User-Id` | 审计/RLS |
| `tenant_id` | `X-Tenant-Id` | 多租户 RLS |

**注意：** Casdoor 的标准 claim 中不包含 `tenant_id` 或 `user_id`，需要通过 Casdoor 的自定义 tokenFields 配置扩展：

在 Casdoor 应用配置中的 `tokenFields` 中添加：
```
"tokenFields": ["sub", "name", "preferredUsername", "roles", "owner", "organization", "application"]
```

如果 Casdoor 不直接支持 `tenant_id`（需要自定义 claim 扩展），则需要在 JWT 解析后做二次映射查询。

**务实做法：** 使用 `sub` 作为用户唯一标识，通过 PostgREST 的 Role Switching（`jwt-role-claim-key`）关联到 PG 角色后，从 PG 获取 `tenant_id` 设置 RLS。

### 5.4 跨组件 Claim 映射总览

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Casdoor 签发   │     │   APISIX 层     │     │  PostgREST 层    │
│                 │     │                 │     │                  │
│ JWT Payload:    │     │ serverless-    │     │ jwt-role-claim- │
│ sub, name,      │────▶│ pre-function:  │────▶│ key: .roles[0]  │
│ roles[], ...    │     │ X-User-Role    │     │                  │
│                 │     │ X-User-Id      │     │ RLS:             │
│                 │     │ X-Tenant-Id    │     │ request.jwt.     │
│                 │     │                │     │ claims→tenant    │
└─────────────────┘     └─────────────────┘     └─────────────────┘
        │                       │                       │
        │                       │                       │
        ▼                       ▼                       ▼
  r.sub = role_admin    authz-casbin 匹配      RLS 策略过滤数据
                         (角色→API→方法)
```

### 5.5 前端 JWT 获取流程

```typescript
// Vue3 + Casdoor SDK
import Sdk from "casdoor-javascript-sdk";

const sdk = new Sdk({
  serverUrl: "http://localhost:8000",
  clientId: "zero-backend-app",
  organizationName: "built-in",
  appName: "zero-backend-rbac",
  redirectPath: "/cb",
  scope: "read"
});

// 检测未登录 → 自动重定向到 Casdoor
async function checkLogin() {
  const token = sdk.getTokenFromCAS();
  if (!token && !sdk.isSilentSigninRequestedOrNull()) {
    sdk.silentSignin(); // 或 sdk.signin()
    return false;
  }
  return token;
}

// API 请求拦截器
axios.interceptors.request.use(async (config) => {
  let token = await checkLogin();
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});
```

## 附录 A：Casdoor 常见问题排查

| 故障现象 | 排查命令 | 解决方案 |
|:---|:---|:---|
| Casdoor 容器反复重启 | `docker compose logs casdoor` | 检查 pgbouncer 连接、casdoor database 存在 |
| 404 on JWKS endpoint | 确认访问路径是 `/.well-known/jwks` 不是 `/.well-known/jwks.json` | Casdoor v3 已移除 .json 后缀 |
| APISIX 验签失败 | 比对 `jwt-secret` 与 Casdoor JWKS 中的 `n` 值 | 确保同密钥对 |
| X-User-Role 头为空 | 检查 JWT 是否包含 `roles` claim | Casdoor 应用配置 tokenFields 需含 "roles" |
| OAuth 重定向后无限循环 | 检查 callback_url 是否在 Casdoor 应用 redirectUris 中 | 确认两者完全一致 |
