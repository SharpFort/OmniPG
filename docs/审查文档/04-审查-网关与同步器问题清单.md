## 🔴 阻塞级问题

### B1. APISIX Admin API 端口全部写错 — 所有管理操作失败

> **01 端口规划表（第 40 行）：** 9080 = APISIX API 网关 HTTP（数据面），**9180 = APISIX Admin API（控制面）**。

但 04 文档中**所有** `curl` 命令和 Go Syncer 代码都使用 **9080** 访问 Admin API：

| 位置 | 错误 URL | 应改为 |
|:---|:---|:---|
| §1 第 15 行 | `http://127.0.0.1:9080/apisix/admin/plugin_metadata/...` | **9180** |
| §2.1 第 61 行 | `http://127.0.0.1:9080/apisix/admin/routes/jwks` | **9180** |
| §2.2 第 70 行 | `http://127.0.0.1:9080/apisix/admin/plugin_metadata/jwt-auth` | **9180** |
| §2.3 第 84 行 | `http://127.0.0.1:9080/apisix/admin/routes/api-v1` | **9180** |
| §2.4 第 127 行 | `http://127.0.0.1:9080/apisix/admin/plugin_configs/1` | **9180** |
| Go Syncer 第 171 行 | `ApisixAdminURL = "...9080/apisix/admin/..."` | **9180** |

- [B1-1] 这是一个**系统性错误**——端口 9080 是数据面（处理客户端请求），不响应 Admin API。使用 9080 访问 `/apisix/admin/*` 会返回 404。所有 curl 命令和 Syncer 代码都需要全局替换 9080→9180。

### B2. Policy Syncer CSV 格式与 DB MD5 计算不匹配 — 对账永远失败

> 两个严重的不一致导致 10 分钟对账（Reconcile）**永远不会匹配**：

**不一致一：分隔符不同**
- DB 侧（第 296 行）：`concat_ws(',', ptype, v0, v1, ...)` → 分隔符是 `,`（逗号）
- Go 侧（第 359 行）：`strings.Join(elements, ", ")` → 分隔符是 `, `（逗号+空格）

**不一致二：NULL 列处理不同**
- DB 侧：`concat_ws` 自动跳过 NULL 值，且**不产生尾部空列**
- Go 侧：`formatToCSV` 截断尾部 NULL，但结果行可能有不同的列数

- [B2-1] DB 和 Go 的 CSV 生成逻辑必须完全一致。建议：
  - 方案 A：统一使用无空格、固定列数的 CSV 格式
  - 方案 B：DB 侧改用与原 CSV 完全相同的输出逻辑（如用 PL/pgSQL 函数生成）
  - 方案 C：放弃全文 MD5 对账，改用 `max(updated_at)` 时间戳或版本号对账
- [B2-2] 无论选哪种方案，需要对账逻辑编写单元测试（用已知策略集验证 hash 一致）。

### B3. Go Syncer 缺少 Advisory Lock 选主 — 多实例会冲突

> 00 文档明确提到 "多实例 + PostgreSQL Advisory Lock 选主"。但 04 的 Go 代码中**完全没有实现**。

- [B3-1] 当前代码如果启动 2+ 个 Syncer 实例，会发生：
  - 两个实例同时 LISTEN 同一个 `casbin_channel`
  - 策略变更时两个实例**同时** PUT 到 APISIX Admin API
  - 可能产生竞态条件（后到达的旧策略覆盖先到达的新策略）
- [B3-2] Advisory Lock 选主逻辑应该在什么时候实现？建议：
  ```go
  // 伪代码
  tx, _ := db.Begin()
  tx.Exec("SELECT pg_try_advisory_lock(12345)")
  // 如果获取锁 → 成为 Leader，进入事件循环
  // 如果未获取 → 成为 Standby，定期重试
  ```
- [B3-3] 如果本期不实现，至少需要在文档中明确标注 "当前版本仅支持单实例部署"。

## 🟡 重要级问题

### M1. serverless-pre-function 中手动解析 JWT 的风险

> 第 92-93 行的 Lua 代码手动 Base64 解码 JWT payload（不验证签名）。签名验证由后面的 `jwt-auth` 插件完成。

- [M1-1] 如果攻击者发送一个格式正确的伪 JWT（绕过 base64 解码成功，但签名无效），`X-User-Role` 头已被设置。`jwt-auth` 插件会在后面拒绝请求吗？还是 `serverless-pre-function` 在 `rewrite` 阶段注入的 header 已经污染了后续处理？
- [M1-2] APISIX 插件执行顺序：`rewrite` → `access`。`serverless-pre-function` 在 rewrite 阶段，`jwt-auth` 在 access 阶段。如果 jwt-auth 拒绝请求，rewrite 阶段注入的 header 不会生效。但请确认 authz-casbin 也在 access 阶段且在 jwt-auth 之后执行。
- [M1-3] 手动 JWT 解析代码没有处理 `roles` 字段缺失的情况——如果 JWT 没有 `roles` claim，`roles_str` 为 nil，`X-User-Role` 头为空字符串，Casbin 的 sub 为空，匹配行为是什么？

### M2. `keyMatch2` vs `regexMatch` 的路由匹配

> model.conf 第 19 行使用 `keyMatch2(r.obj, p.obj)` 做 URL 路径匹配。`keyMatch2` 是 Casbin 内置函数，支持 `:id` 通配符和 `*` 递归匹配。

- [M2-1] PostgREST 的 URL 格式是 `/sys_user?id=eq.UUID`（查询参数），而 `keyMatch2` 只匹配路径部分。`sys_api.path` 中存储的路径模式（如 `/sys_user`）是否包含查询参数？如果不包含，GET `/sys_user?id=eq.xxx` 和 DELETE `/sys_user?id=eq.xxx` 在 Casbin 中如何区分？
- [M2-2] `r.act == p.act` 是精确匹配 HTTP 方法——确认 APISIX 的 `ctx.var.method` 是否直接映射到 Casbin 的 `r.act`。

### M3. Go Syncer 的错误处理不足

> 以下位置忽略了错误或处理不完整：

- [M3-1] 第 379 行：`body, _ := io.ReadAll(resp.Body)` — 忽略读取错误。如果响应体读取失败，`body` 为 nil，后续 `json.Unmarshal` 会 panic 或返回错误结果。
- [M3-2] 第 392 行：`payload, _ := json.Marshal(metadata)` — 忽略序列化错误。
- [M3-3] 第 405 行：`body, _ := io.ReadAll(resp.Body)` — 同上。
- [M3-4] 第 375-377 行：如果 APISIX 返回 404（策略从未被设置），返回空字符串。但空字符串的 MD5 与 DB 策略的 MD5 必然不同，会触发不必要的全量同步。首次启动时这是正常的，但日志应区分 "首次同步" 和 "hash 不匹配"。

### M4. Syncer 未实现优雅关闭

> `main()` 函数中第 237 行进入 `StartEventLoop` 后**永久阻塞**。如果进程收到 SIGTERM/SIGINT：

- [M4-1] `defer listener.Close()` 不会被执行（因为 main 永远到不了 defer 语句之后的代码——实际上 listener.Close() 被 defer 了但 main 阻塞在 StartEventLoop 中）。需要 signal handling：
  ```go
  sigCh := make(chan os.Signal, 1)
  signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)
  // 在 select 中同时监听 ctx.Done() 和 sigCh
  ```
- [M4-2] 关闭时是否应该等待进行中的同步完成（draining）？

### M5. `db-uri` 端口和 sslmode 与 01/03 不一致

> 第 170 行：`PostgresDSN = "...@127.0.0.1:5433/...?sslmode=disable"`

- [M5-1] 与 03 审查 M1 相同：开发环境没有 HAProxy（端口 5433 不通）。Syncer 在 Docker Compose 环境中无法连接 PG。
- [M5-2] `sslmode=disable` 与 03 的 `sslmode=verify-full` 不一致。需要多环境配置。

### M6. `crypto/md5` 的使用

> 第 154 行导入 `crypto/md5`，第 309 行使用 `md5.Sum()`。

- [M6-1] MD5 在密码学上已不安全，对账场景虽不涉及安全但有更好的选择（SHA256）。特别是 Go 1.22+ 的标准库 `crypto/sha256` 同样易用。是否替换为 SHA256？
- [M6-2] 如果改用 SHA256，DB 侧的 `md5()` 函数也需改为 `encode(digest(..., 'sha256'), 'hex')`。

## 🟢 增强级问题

### E1. Go 编译指令与 01 不一致

- [E1-1] 01 第 480 行：`go mod init syncer`（模块名 `syncer`）
- [E1-2] 04 第 418 行：`go mod init policy-syncer`（模块名 `policy-syncer`）

模块名不一致。且 01 中已经执行过一次 `go mod init` 和 `go get`，04 又重复执行。如果 agent 先执行 01 再执行 04，会创建两个不同的 go.mod。请统一。

### E2. JWKS 公钥硬编码在多处

公钥占位符 `YOUR_RSA_PUBLIC_KEY_MODULUS_BASE64URL` 出现在：
- 04 §1 model.conf：不需要（model 不含 key）
- 04 §2.1 JWKS mocking 路由（第 52 行）
- 04 §2.2 jwt-auth plugin_metadata（第 75 行）
- 03 postgrest.conf 的 `jwt-secret`（第 32 行）

- [E2-1] 密钥轮换时需要同时更新至少 3 处配置。是否有自动化或文档化的轮换流程？建议在文档中增加"密钥轮换操作清单"。

### E3. CORS origin 硬编码

> 第 101 行：`"allow_origins": "https://admin.yourdomain.com"`

- [E3-1] 开发环境前端可能在 `http://localhost:5173`（Vite 默认端口）。这个 CORS 配置会导致开发环境前端请求被拒绝。建议从环境变量读取或提供开发/生产两套配置。

### E4. Syncer 缺少结构化日志

> 所有日志使用标准 `log.Println`。在生产环境中，结构化日志（JSON 格式、带时间戳和级别）更便于采集和分析。

- [E4-1] 是否考虑使用 `log/slog`（Go 1.21+ 内置的结构化日志）？

### E5. policy syncer 未容器化

- [E5-1] Syncer 是 Go 编译的二进制文件，可以打包为 Docker 镜像加入 Docker Compose。01 的 Compose 中没有 syncer 服务。是计划在阶段 4 手动部署，还是应该在 Compose 中增加一个 syncer 容器？
