# 零后端架构安全性分析

> **项目：** 零后端代码统一权限管理系统（PostgreSQL + PostgREST + APISIX + Casdoor + Go Policy Syncer）
> **编写日期：** 2026-07-13
> **目的：** 评估 Database-Driven 架构在 SQL 注入、数据暴露、权限控制等安全维度的表现，并与传统 ORM 方案对比。

---

## 一、你的架构天然安全的地方（ORM 做不到或做不到这么好）

| 安全特性 | 你的架构 | 传统 ORM (C#/Go) |
|:---|:---|:---|
| **参数化查询** | PostgREST 和 PL/pgSQL 函数**天然参数化**，HTTP 参数只是绑定值，**不可能**拼接到 SQL 中 | ORM 默认参数化，但开发者可手动 `.Raw()` / `.Where("id=" + id)` 绕过 |
| **SQL 注入攻击面** | **极小** — 唯一风险是 PL/pgSQL 内部手动写 `EXECUTE '...' \|\| arg`（可规范避免） | **大** — 每个手写 SQL 语句都是潜在注入点 |
| **权限控制粒度** | **行级 (RLS)** — 数据库自动过滤用户看不到的数据 | 通常在代码层手动过滤，易遗漏 |
| **业务逻辑收敛** | 逻辑在数据库内，攻击者**无法绕过** API 直接调用函数逻辑 | 后端代码可被绕过（直接调内部接口、消息队列等） |
| **数据暴露最小化** | 只暴露 `api_v1` schema，底层表**对应用完全不可见** | ORM 实体类常直接映射全表，易暴露敏感列 |

> **核心结论：** PostgREST 本身就是"最安全的 ORM"，因为它**不允许发送任意 SQL**，只允许调用预定义的函数和视图。

## 二、你的架构的实际风险点（5 个）

### 🔴 风险 1：PL/pgSQL 动态 SQL 拼接（唯一真正的 SQL 注入点）

```sql
-- ❌ 危险！动态拼接用户输入
CREATE FUNCTION search(p_name TEXT)
RETURNS SETOF sys_user AS $$
BEGIN
    RETURN QUERY EXECUTE 'SELECT * FROM sys_user WHERE name = ''' || p_name || '''';
    -- 攻击者输入: ' OR '1'='1' --
    -- 结果: SELECT * FROM sys_user WHERE name = '' OR '1'='1' --'
END;
$$ LANGUAGE plpgsql;
```

```sql
-- ✅ 安全写法 1：USING 子句（参数化）
CREATE FUNCTION search(p_name TEXT)
RETURNS SETOF sys_user AS $$
BEGIN
    RETURN QUERY EXECUTE 'SELECT * FROM sys_user WHERE name = $1' USING p_name;
END;
$$ LANGUAGE plpgsql;

-- ✅ 安全写法 2：format() + %I（标识符引用）
CREATE FUNCTION search(p_table TEXT, p_name TEXT)
RETURNS SETOF sys_user AS $$
BEGIN
    RETURN QUERY EXECUTE format('SELECT * FROM %I WHERE name = $1', p_table) USING p_name;
END;
$$ LANGUAGE plpgsql;
```

**对策：** 在代码审查规则中**禁止** PL/pgSQL 内的 `||` 字符串拼接，强制使用 `USING` 或 `format(%I/%L)`。

---

### 🔴 风险 2：PostgREST 配置不当暴露底层表

```properties
# ❌ 危险！暴露了 public schema（包含 sys_user.password_hash）
db-schemas = "public, api_v1"

# ✅ 正确！只暴露 api_v1，底层表不可见
db-schemas = "api_v1"
db-extra-search-path = "public"
```

如果错误配置，攻击者可能直接 `GET /password_hash?select=*`，或 `DELETE /sys_user?id=eq.1`。

**对策：**
- `db-schemas` 只放 `api_v1`
- 所有物理表放在 `public`（或不放任何业务 schema），通过 `api_v1` 视图暴露
- `password_hash` 等敏感列**永远不出现在视图的 SELECT 列表中**

---

### 🟡 风险 3：JWT 密钥管理

| 场景 | 风险 |
|:---|:---|
| 私钥泄露 | 攻击者可签发任意 JWT |
| JWKS 端点不可用 | APISIX 无法验签，所有请求被拒绝 |
| 密钥未轮转 | 长期使用的密钥一旦泄露影响持续 |

**对策：**
- **私钥只存 Casdoor 内部**，PostgREST/APISIX 只存 **JWKS 公钥**
- APISIX 配置 JWKS 缓存 + 失败时最后一次有效公钥兜底
- 支持密钥轮转（JWKS 多 key + kid 匹配）

---

### 🟡 风险 4：pg_notify 伪造 → 策略同步劫持

```sql
-- 任何人都可以手动发送 pg_notify 触发全量同步
SELECT pg_notify('casbin_change', 'full_sync');
```

**对策：**
- pg_notify 触发器由**数据库 DML 事件**（INSERT/UPDATE/DELETE on sys_role_permission）触发，不对外暴露
- Syncer 端做**防抖 + 对账**，即使收到伪造通知，全量同步只是性能问题，不会造成权限错乱
- 限制 pg_notify 发送权限：`REVOKE ALL ON FUNCTION pg_notify FROM public;`

---

### 🟡 风险 5：审计日志被篡改

```sql
-- 如果有数据库写权限，可以删除审计日志
DELETE FROM sys_audit_log WHERE action = 'admin_login';
```

**对策：**
- 审计触发器使用 `SECURITY DEFINER`（以 postgres 身份运行），普通用户无法绕过
- 审计表只读：`REVOKE UPDATE, DELETE ON sys_audit_log FROM app_owner;`
- 审计日志同时写入**只读存储**（如 S3/对象存储）做异地备份

## 三、与 ORM 方案的安全性对比总结

| 维度 | 你的架构 (PostgREST + PL/pgSQL) | 传统 ORM |
|:---|:---|:---|
| **SQL 注入** | 天然免疫（除非 PL/pgSQL 内部手动拼接） | 默认安全，但手写 SQL 易引入 |
| **权限粒度** | 行级 (RLS) + 列级（视图排除） | 通常在代码层手动实现 |
| **数据暴露** | 只暴露 API schema，底层表不可见 | 实体类常直接映射全表 |
| **业务逻辑** | 在数据库内，无法绕过 | 可被内部接口/消息队列绕过 |
| **输入验证** | 函数参数类型检查 + 视图约束 | 框架注解 + 服务层校验 |
| **业务逻辑复杂度** | PL/pgSQL 调试困难 | 代码更直观，调试方便 |

你的方案里已经有以下安全机制，比大多数 ORM 项目更完善：

| 已有安全设施 | 位置 | 作用 |

## 五、结论

> **你的架构在安全性上 ≥ 传统 ORM 方案，且在"数据暴露最小化"和"权限粒度"上更强。**

唯一需要警惕的是 **PL/pgSQL 内部的手动字符串拼接**，这可以通过代码审查规范 + 模板化工具规避（如强制所有动态 SQL 使用 `format(%I, %L)` + `USING` 子句）。

你的方案**不需要"防范 SQL 注入"**，因为 PostgREST + PL/pgSQL 的默认行为就已经是参数化的。安全工作的重心应该放在：
1. 配置正确性（db-schemas 暴露范围）
2. 密钥管理（JWKS 私钥隔离）
3. 审计防篡改（SECURITY DEFINER + 只读权限）
