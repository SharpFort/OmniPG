## 🔴 P0 阻塞级

### P0-1. Syncer `depends_on` 中 `condition_healthy` 拼写错误 — 服务永远不会启动

> **位置：** `docker-compose.yml` 第314行

```yaml
syncer:
  depends_on:
    postgres:
      condition: service_healthy
    apisix:
      condition: condition_healthy   # ← ⚠️  拼写错误！
```

**问题：** `condition: condition_healthy` 是无效语法。Docker Compose 期望 `condition: service_healthy`、`condition: service_started` 或 `condition: service_completed_successfully`。

**后果：**
- `docker compose up -d syncer` 报错：`invalid depends_on condition: "condition_healthy"`
- 或 Docker Compose v3.8 中表现为忽略该条件导致 Syncer 在 APISIX 未就绪时立即启动
- `policy-syncer` 容器反复重启（启动时连不上 APISIX Admin API → 报错退出 → restart 循环）

**修复方案：**
```yaml
depends_on:
  postgres:
    condition: service_healthy
  apisix:
    condition: service_healthy   # ← 修复
```

- [P0-1] **必须修复。** 阻塞 Syncer 启动。

## 🟡 P1 重要级

### P1-1. 缺少 Redis 容器 — APISIX `limit-req` 插件不可用

> **位置：** `apisix/config.yaml` 第551行 + `docker-compose.yml` 全文无 redis 服务

**问题：** `apisix.yaml` 中的 `limit-req` 插件配置了 `policy: redis` 和 `redis_host: redis`，但 docker-compose.yml 中完全没有定义 `redis` 服务。

**后果：**
- APISIX 初始化时尝试连接 `redis:6379`，连接失败
- 路由中配置的 `limit-req` 插件无法工作，**限流功能完全失效**
- 开发环境无法提前发现 Redis 相关的 APISIX 配置问题

**修复方案：**

添加 Redis 服务：
```yaml
redis:
  image: redis:7-alpine
  container_name: app-redis
  restart: unless-stopped
  ports:
    - "${REDIS_PORT:-6379}:6379"
  networks:
    - app-net
  volumes:
    - redis_data:/data
  healthcheck:
    test: ["CMD", "redis-cli", "ping"]
    interval: 10s
    timeout: 3s
    retries: 5
```

同时在 `volumes:` 段增加：
```yaml
volumes:
  redis_data:
```

### P1-2. Casdoor 数据库初始化脚本未挂载到容器 — Casdoor 启动即报错

> **位置：** `docker-compose.yml` 第103-107行 + `scripts/init-db.ps1` 第986行

**问题：** Casdoor 服务连接 `casdoor` 数据库（`dbname=casdoor`），但 docker-compose.yml 仅挂载了两个 init 脚本：

```yaml
volumes:
  - ./db/init/01-extensions.sql:/docker-entrypoint-initdb.d/01-extensions.sql:ro
  - ./db/init/02-schemas.sql:/docker-entrypoint-initdb.d/02-schemas.sql:ro
  # ❌ 缺少 ./db/init/03-casdoor-db.sql
```

`02-schemas.sql` 创建了 `casdoor` 角色（SQL 02-schemas.sql 第689行），但 **没有创建 `casdoor` 数据库**。`03-casdoor-db.sql` 负责创建:

```sql
SELECT 'CREATE DATABASE casdoor OWNER casdoor'
WHERE NOT EXISTS (SELECT FROM pg_database WHERE datname = 'casdoor')\gexec
```

**后果：**
1. `docker compose up -d` 后立即启动 Casdoor → 连接 `casdoor` 数据库失败
2. 解决依赖手动运行 `.\scripts\init-db.ps1`（其中包含了手动 psql 执行 03-casdoor-db.sql）
3. 但 `depends_on: pgbouncer: service_started` 不等待 init-db 完成

**修复方案：**

方案 A（推荐）：将 `03-casdoor-db.sql` 加入 docker-compose 挂载
```yaml
volumes:
  - ./db/init/01-extensions.sql:/docker-entrypoint-initdb.d/01-extensions.sql:ro
  - ./db/init/02-schemas.sql:/docker-entrypoint-initdb.d/02-schemas.sql:ro
  - ./db/init/03-casdoor-db.sql:/docker-entrypoint-initdb.d/03-casdoor-db.sql:ro
```

方案 B：在 `02-schemas.sql` 末尾追加 casdoor 数据库创建（但 `\connect` 不能在同一脚本中用 `docker-entrypoint-initdb.d` 的方式执行）

### P1-3. `api_v1.check_token_blacklist` 未定义 — PostgREST db-pre-request 404

> **位置：** `docker-compose.yml` 第219行 `PGRST_DB_PRE_REQUEST: "api_v1.check_token_blacklist"`

**问题：** 08 中的 `02-schemas.sql` 创建了 `api_v1` schema，但实际上 **没有将 `check_token_blacklist()` 函数迁移到 `api_v1` schema**。07 文档的 Migration 005 在 `public` schema 创建了该函数，但 PostgREST 配置 `db-pre-request = "api_v1.check_token_blacklist"` 要求在 `api_v1` 中可见。

**后果：**
- PostgREST 启动时触发 `SET search_path TO api_v1; SELECT api_v1.check_token_blacklist()`
- 函数不存在 → PostgREST 无法启动（或静默跳过但不执行黑名单检查）

**修复方案：**

在 `02-schemas.sql` 末尾追加 `api_v1 schema 包装函数`：
```sql
CREATE OR REPLACE FUNCTION api_v1.check_token_blacklist()
RETURNS void
LANGUAGE sql
SECURITY DEFINER
SET search_path = public, pg_temp
AS $$ SELECT public.check_token_blacklist() $$;
```

### P1-4. PostgREST `jwt-secret` 占位符 `${JWKS_JSON:-***}` — 开发环境无法启动

> **位置：** `docker-compose.yml` 第218行 + `.env.example` 第405行

**问题：** `.env.example` 中 `JWKS_JSON=***`，用户复制后若不修改，PostgREST 收到 `***` 作为 JWT secret，格式无效。

**修复方案：**

方案 A（开发环境 — 对称密钥）：
```
JWKS_JSON={"keys":[{"kty":"oct","kid":"dev-key","alg":"HS256","k":"c2VjcmV0X2RldmVsb3BtZW50X2tleV9hdF9sZWFzdF9zZXZlbl9jaGFyYWN0ZXJzIQ=="}]}
```

方案 B（兼容 Casdoor RS256）：
通过 04.5-Casdoor集成文档配置，或提供脚本自动生成

### P1-5. Syncer healthcheck 检测到的是"进程存活"而非"功能健康"

> **位置：** `docker-compose.yml` 第322-326行

```yaml
healthcheck:
  test: ["CMD", "sh", "-c", "ps aux | grep -v grep | grep policy-syncer || exit 1"]
```

**问题：** `ps aux | grep policy-syncer` 检测的是进程是否存活。但 Syncer 可能：进程存活但 PG 连接断开、进程存活但 APISIX PUT 超时。`health: healthy` 状态不代表"策略实时同步正常"。

**建议：** 结合 Syncer 的 `/healthz` 端点（07/P1-1 建议实现），在 Dockerfile 中添加：
```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:8080/healthz || exit 1
```

## 🟢 P2 增强级

### P2-1. 路径分隔符不一致

> **位置：** `scripts/start.ps1` 第799行 vs `scripts/start.sh` 第903行

**问题：**
- PowerShell 版本使用反斜杠：`$ProjectRoot\db\init`（混合使用，部分用 `\` 部分 `/`）
- bash 版本使用正斜杠：`$PROJECT_ROOT/db/init`

**建议：** PowerShell 中统一使用 `/`（PowerShell 对 `/` 兼容，反斜杠在某些命令中需要转义）。

### P2-2. 缺少 `redis_data` volume 声明

**问题：** 如果添加 Redis 服务，需要在 top-level `volumes:` 声明中加入 `redis_data`。

---

### P2-3. Pgbouncer 日志配置

> **位置：** `pgbouncer.ini` 第460行

```ini
logfile = /var/log/pgbouncer/pgbouncer.log
pidfile = /var/run/pgbouncer/pgbouncer.pid
```

**问题：** 容器内 `/var/log/pgbouncer/` 和 `/var/run/pgbouncer/` 可能不存在。需要在 Dockerfile 或 entrypoint 中创建目录。

**建议：** 改用 stdout 输出，或确认 pgbouncer 官方镜像会自动创建这些目录。

---

### P2-4. 缺少停机/清理脚本

**问题：** 文档§4 提供了 `start.ps1` 但没有 `stop.ps1` / `reset.ps1`（含数据清理）。

**建议：** 增加：
```powershell
# stop.ps1
docker compose down

# reset.ps1
docker compose down -v
docker compose up -d --build
```
