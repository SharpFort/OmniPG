## 🔴 阻塞级问题（代码无法运行）

### B1. `generate_rs256_jwt()` — 核心函数从未定义

> **位置：** Migration 005 `user_login_sso()` 第 373 行 和 `refresh_token_rtr()` 第 464 行都调用了 `generate_rs256_jwt(v_payload, v_private_key, 'key-v1')`。

**9 个 migration 文件中没有任何一个定义了这个函数。**

这是整个认证系统的中枢——登录、刷新 Token 都依赖它生成 RS256 签名的 JWT。没有这个函数，登录接口直接报错 `function generate_rs256_jwt(jsonb, text, text) does not exist`。

- [B1-1] `generate_rs256_jwt()` 的实现代码在哪里？应该在哪一个 migration 中创建（建议在 005 之前或作为 005 的一部分）？
- [B1-2] 该函数的实现语言是什么？如果使用 plpython3u（调用 PyJWT），则受 00 审查 A3-1 的阻塞影响（plpython3u 在 Pigsty 中可能不存在）。替代实现路径：
  - plv8 + JavaScript JWT 库（Pigsty 支持，但仅 RPM）
  - plpgsql + pgcrypto 手动构造 JWT（可实现但复杂）
  - pgjwt 扩展（PGXN 有 `pgjwt`，但只支持 HMAC，不支持 RS256）
  - **请提供该函数的完整实现代码，至少包括函数签名和实现语言。**

### B2. `sha256()` — 辅助函数从未定义

> **位置：** `user_login_sso()` 第 351 行 `sha256(v_new_rt::bytea)` 和 `refresh_token_rtr()` 第 408 行。

- [B2-1] pgcrypto 提供的哈希函数是 `digest(data, type)`（如 `digest('hello', 'sha256')`），不是 `sha256()`。是否遗漏了一个包装函数：
  ```sql
  CREATE OR REPLACE FUNCTION sha256(data bytea) RETURNS text AS $$
    SELECT encode(digest(data, 'sha256'), 'hex');
  $$ LANGUAGE sql IMMUTABLE;
  ```
  如果没有这个包装，`sha256()` 调用会直接失败。请确认或补充。

### B3. RLS 策略引用了不存在的 JWT claim

> **位置：** Migration 008 第 730 行：
> ```sql
> current_setting('request.jwt.claims', true)::json->>'role' = 'super_admin'
> ```

JWT payload 中存的是 `roles` 数组（`["super_admin", "role_editor"]`），不是 `role` 单值。`json->>'role'` 永远为 NULL，导致这个 RLS 策略中的超级管理员豁免条件**永远不会生效**。

- [B3-1] 是否应该改为检查 `roles` 数组中是否包含 `super_admin`？
  ```sql
  current_setting('request.jwt.claims', true)::json->'roles' ? 'super_admin'
  ```

## 🟡 重要级问题（功能不完整或有缺陷）

### M1. 多租户模型不完整

> 00 审查 E2 曾问过"sys_role、sys_api、sys_menu 是否有 tenant_id"。02 文档的 DDL 给出了答案：**没有**。

- [M1-1] 角色、API、菜单都是**全局共享**的（跨租户）。这意味着：
  - 租户 A 的管理员可以看到租户 B 的角色（通过 `/sys_role` API）
  - 租户 A 分配的角色（如 `role_admin`）在 Casbin 策略中对所有租户用户的 API 访问都生效
  - 这是有意设计还是疏忽？如果是 SaaS 多租户，建议至少 `sys_role` 加 `tenant_id`。
- [M1-2] 如果角色和 API 是全局的，那么 SaaS 场景下"租户自定义角色"怎么实现？每个租户都需要独立管理自己的角色和 API 权限。

### M2. `sys_api` 缺少 (path, method) 唯一约束

> 00 审查 E1-3 问过这个问题。02 的 DDL 中 `sys_api` 只有 PK 唯一约束。

- [M2-1] 同一路径+方法的 API 可以重复插入。Seed 数据使用 `ON CONFLICT DO NOTHING`（第 854 行），但由于没有 (path, method) 唯一约束，第二次执行 seed 会创建**重复行**，导致 Casbin 策略中出现重复规则。
- [M2-2] 建议添加：`CREATE UNIQUE INDEX idx_api_path_method_unique ON sys_api(path, method);`

### M3. `sys_menu` ON CONFLICT 行为不一致

- [M3-1] Seed 数据中 sys_menu INSERT 语句（第 809、818 行）使用 `ON CONFLICT DO NOTHING` 但未指定冲突列。默认是 PK，所以用相同 UUID 的重复插入会被跳过。但这意味着：如果改了 seed 数据的内容（如修改菜单名称），不会更新已有记录——需要手动 DELETE 再 INSERT。
- [M3-2] 是否应该用 `ON CONFLICT (id) DO UPDATE SET ...` 使 seed 可重入更新？

### M4. `sys_department` 递归外键的级联删除风险

> 第 60 行：`parent_id UUID REFERENCES sys_department(id) ON DELETE CASCADE`

- [M4-1] 删除一个父部门时，所有子部门、孙部门级联删除。依赖该部门的 `sys_user` 的 `dept_id` 被 SET NULL（第 73 行）——用户不会丢失。但整个部门子树被物理删除，无法恢复。
- [M4-2] 是否需要软删除（`deleted_at`）替代物理删除？00 审查 E1-1 问过软件删除的问题，02 中似乎选择了物理删除方案。
- [M4-3] 部门树的深度是否有限制？递归 CTE 在超过一定深度后性能下降。建议文档说明最大深度。

### M5. 缺少 `updated_at` 自动更新触发器

> `sys_user` 和 `sys_menu` 表定义了 `updated_at` 列，但没有任何触发器在 UPDATE 时自动更新它。

- [M5-1] `updated_at` 需要手动在每次 UPDATE 语句中 SET？还是遗漏了触发器？标准做法：
  ```sql
  CREATE OR REPLACE FUNCTION update_updated_at()
  RETURNS TRIGGER AS $$
  BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
  END;
  $$ LANGUAGE plpgsql;
  ```

### M6. `get_user_menu()` 返回扁平列表而非嵌套树

> 第 556-576 行使用递归 CTE 查出了所有菜单项，但最终 `json_agg` 输出是一个**扁平数组**（每个元素带 parent_id）。前端需要自行重建树结构。

- [M6-1] 是否有意设计为扁平输出让前端处理？还是应该在后端递归构造嵌套 JSON？
- [M6-2] 如果是扁平输出，前端的树重建算法复杂度是 O(n²)（每个节点需要找父节点），菜单超过 1000 项时性能影响？

### M7. `set_config('response.headers', ...)` 格式需验证

> 第 380、470 行使用：
> ```sql
> PERFORM set_config('response.headers', '[{"Set-Cookie": "..."}]', true);
> ```

- [M7-1] PostgREST v14 的 `response.headers` 要求值是 JSON 数组格式。当前写法 `'[{"Set-Cookie": "..."}]'` 看起来正确，但需要确认：
  - PostgREST 是否支持 `Set-Cookie` 的多个属性（HttpOnly, Secure, SameSite, Path, Max-Age）通过这个方式正确传递？
  - 线上搜索 `response.headers` 在 PostgREST 配置文档中未直接找到——是通过 `set_config` 在事务内临时设置的吗？请提供 PostgREST 官方文档链接确认此机制。

### M8. Refresh Token 存储在数据库中为 SHA256 哈希

> 第 185 行：`refresh_token_hash VARCHAR(64)` — 存储的是 hex 编码的 SHA256（64 字符）。

- [M8-1] SHA256 对 Refresh Token（256 位随机值）足够安全。但注意 `sha256()` 如果返回 hex 编码（64 char），而 `digest()` 默认返回 bytea——需要 `encode(digest(...), 'hex')` 才能得到 64 字符 hex 字符串。这与 B2 问题关联。
- [M8-2] `refresh_token_hash` 有 UNIQUE 约束。两个不同用户碰巧生成相同的 RT（概率极低但非零），会导致第二个用户登录失败。错误处理是什么？

### M9. `pg_notify` payload 过于简化

> 第 267 行：`PERFORM pg_notify('casbin_channel', 'reload');`

- [M9-1] payload 只有字符串 `'reload'`，不包含任何变更上下文。Policy Syncer 收到通知后必须全量重新查询 `casbin_rule` 视图。在小规模（<10万条规则）场景可以接受，但规模增大后全量同步的延迟会增加。
- [M9-2] 是否需要传递变更类型（INSERT/UPDATE/DELETE）和受影响的行 ID，让 Syncer 做增量更新？

### M10. `sys_user_role_request` 表位置不当

> 审批流表在第 7 节（Migration 006）中创建，而不是在 Migration 001/002 中。

- [M10-1] 这个表依赖 `sys_user` 和 `sys_role`（FK），逻辑上应放在 Migration 002。放在 006 不会导致技术问题（migration 顺序正常），但组织和维护上不直观。是否有特殊原因？
- [M10-2] `applicant_id` 和 `approver_id` 都是 UUID——它们引用的是 `sys_user.id` 吗？缺少 FOREIGN KEY 约束声明。

### M11. 缺少 Token 黑名单定期清理机制

> `sys_token_blacklist` 表会持续增长。已过期的 jti 没有清理机制。

- [M11-1] 清理过期黑名单记录的定时任务在哪里？pg_cron？pg_timetable？还是依赖应用层？
- [M11-2] 类似地，`sys_user_session` 中已过期的会话记录是否需要清理？

### M12. `sys_menu` 中 `permission_code` 未加唯一约束

> 第 121 行：`permission_code VARCHAR(100)` — 没有 UNIQUE 约束。

- [M12-1] 如果两个按钮菜单使用相同的 `permission_code`（如 `user:add`），前端如何处理？建议添加唯一约束防止重复。

### E2. 种子数据使用硬编码 UUID

> 所有种子数据使用了固定 UUID（如 `00000000-...-000000000001`）。

- [E2-1] 这些 UUID 是否来自 UUID v5（基于 namespace）还是随意编写？如果是随意编写的（如 `00000000-...-200000000001`），在多个环境间迁移时，外部系统引用这些 ID 会不一致。建议使用 `uuid_generate_v5()` 基于 namespace 生成确定性 UUID。

### E3. `sys_department` 缺少多租户隔离

> 部门表没有 `tenant_id` 字段。

- [E3-1] 部门是租户内的还是全局的？结合 `sys_user` 有 `tenant_id` 来看，部门应该在租户内。但当前 DDL 中部门无租户隔离——租户 A 可以看到租户 B 的部门。

### E4. `sys_user_session` 缺少 `client_ip` 和 `user_agent` 的写入逻辑

> 第 189-190 行定义了 `client_ip` 和 `user_agent` 列，但 `user_login_sso()` 的 INSERT 语句（第 353-354 行）没有插入这两个字段。

- [E4-1] 这两个字段的值从哪里获取？PostgREST 的 `request.headers`？当前代码中未实现任何写入。

### E5. 密码哈希未加盐独立管理

> 第 71 行：`password_hash VARCHAR(255)` — 使用 `crypt()` + `gen_salt('bf', 10)`。

- [E5-1] bcrypt `gen_salt('bf', 10)` 是标准选择。但 cost factor 10 在高并发登录场景下（每次验证需要 ~100ms 的 CPU），PostgREST 的连接池可能被登录请求耗满。是否需要说明 cost factor 10 的并发容量？

### E6. migration down 脚本不完整

> 每个 migration 都有 `-- migrate:down` 部分，但采用 DROP CASCADE 策略。

- [E6-1] Migration 回滚（down）在生产环境中的策略是什么？文档第 33 行说"生产环境禁止回滚，采用 roll-forward 策略"——down 脚本仅用于开发环境吗？是否需要明确说明？
