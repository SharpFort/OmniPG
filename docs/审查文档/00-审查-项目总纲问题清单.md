## B. 安全模型

### B1. JWT 私钥管理（⚠️ 新增：plpython3u 缺失的影响）
> plpython3u 在 Pigsty 中可能不存在（见 A3-1），这会直接导致"数据库内 JWT 签名"的策略无法在 Pigsty 上实施。

- [B1-0] 🔴 **前置问题**：如果 plpython3u 不可用，JWT 签名从 PG 内部移到外部 sidecar（Go/Python），私钥就**不再存储在 `sys_secret` 表中**。整个 00 文档第 6.2 节的 JWT 链路描述需要重写。请先确定 A3-1 的答案后再回答以下 B1 问题。
- [B1-1] 私钥以什么形式存储？明文？加密？如果 PG 被攻破（SQL 注入），攻击者能否通过调用同一个 SECURITY DEFINER 函数读出私钥？是否有第二层防护？
- [B1-2] 私钥的轮换流程？如果泄露，如何不中断服务更换 JWT 签名密钥？是否支持多密钥并存（smooth rotation）？
- [B1-3] JWKS 端点 (`/well-known/jwks.json`) 由 APISIX mocking 路由提供——APISIX 如何获取公钥？硬编码？动态从 PG 拉取？手动同步？

### B2. 缺失的安全措施
- [B2-1] CSRF 防护：前端 Vue3 SPA + httpOnly Cookie (Refresh Token)。APISIX 是否需要 CSRF 插件？未涉及。
- [B2-2] CORS 策略：多组件（APISIX, PostgREST, Swagger UI, MinIO）暴露在不同端口/域名，CORS 统一在哪里配置？允许哪些 Origin？
- [B2-3] SQL 注入：PL/pgSQL 中动态 SQL（EXECUTE）风险是否评估？哪些函数使用了动态 SQL？
- [B2-4] 威胁模型：攻击者画像（外部/内部/已认证）？最高优先级防范的攻击场景？

### B3. Token 黑名单链路
- [B3-1] Redis 是"高并发 Token 黑名单缓存"，但 PostgREST `db-pre-request` 直接查 `sys_token_blacklist` 表。Redis 在哪里被查询？APISIX 插件？还是两层（APISIX 查 Redis + PostgREST 查 PG）？
- [B3-2] Redis 与 PG 黑名单表一致性如何保证？Redis 缓存未命中时的 fallback 逻辑？
