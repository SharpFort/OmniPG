## 🔴 P0 阻塞级

### P0-1. APISIX jwt-auth 使用 HS256 与 PostgREST RS256 矛盾 — 同一 JWT 无法双重验证

> **位置：** §4.2 第127-128行

```json
{
  "algorithm": "HS256",
  "secret": "dev_jwt_secret_min_32_chars___"
}
```

**问题：**
APISIX 配置为 HS256（对称加密，使用固定 secret），但：

| 组件 | JWT 验证方式 | 密钥格式 |
|:---|:---|:---|
| 07 Migration / user_login_sso | **RS256 + plpython3u** | RSA 私钥签名 |
| 09 PostgREST | **RS256 + JWKS** | RSA 公钥验证 |
| 10 APISIX | **HS256 + secret** | 对称密钥 |

**后果：**
- PG 内用 RS256 签发的 JWT（如果 plpython3u 还能工作）
- APISIX 用 HS256 验证 → **签名算法不匹配 → 401 Unauthorized**
- 或反向：如果 APISIX 用 HS256 验证通过，PostgREST 用 RS256 验证必然失败

**修复方案：**
统一使用 RS256 + Casdoor JWKS：
```json
{
  "algorithm": "RS256",
  "jwks": "{\"keys\":[{\"kty\":\"RSA\",\"kid\":\"cert-built-in\",\"use\":\"sig\",\"alg\":\"RS256\",\"n\":\"从Casdoor获取\",\"e\":\"AQAB\"}]}"
}
```

### P0-2. APISIX upstream nodes 服务名错误 — 业务路由和 Casdoor 回调 502

> **位置：** §7.1 第382行 + §7.3 第424行

```json
// 错误：使用容器名而非服务名
"nodes": {
  "app-postgrest:3000": 1,  // ← 应改为 "postgrest:3000"
  "app-casdoor:8000": 1     // ← 应改为 "casdoor:8000"
}
```

**问题：**
08 docker-compose.yml 定义的服务名称是：

| docker-compose.yml 服务名 | container_name | APISIX upstream 应使用 |
|:---|:---|:---|
| `postgrest` | `app-postgrest` | **`postgrest:3000`** |
| `casdoor` | `app-casdoor` | **`casdoor:8000`** |
| `postgres` | `app-postgres` | `postgres:5432` |
| `apisix` | `app-apisix` | `apisix:9080` |

APISIX 通过 Docker Compose 内部 DNS 解析服务名，应使用**服务名**（`postgrest`, `casdoor`），而非 `container_name`（`app-postgrest`, `app-casdoor`）。

**后果：**
- APISIX 尝试解析 `app-postgrest` → DNS 解析失败（Docker bridge 网络不通过 container_name 暴露）
- **Bad Gateway 502** — 所有业务请求和 Casdoor 回调都会失败

**修复：**
```json
// §7.1 api-v1-route.json
"nodes": {
  "postgrest:3000": 1
}

// §7.3 casdoor-callback.json
"nodes": {
  "casdoor:8000": 1
}
```

## 🟡 P1 重要级

### P1-1. CORS `allow_origins: "*"` + `allow_credential: true` 必被浏览器拒绝

> **位置：** §7.3 第398行

```json
"cors": {
  "allow_origins": "*",
  "allow_credential": true
}
```

**问题：** 
W3C CORS 规范明确禁止：`Access-Control-Allow-Origin: *` + `Access-Control-Allow-Credentials: true` 同时存在。

浏览器（Chrome/Firefox/Safari）会拒绝此类响应，报错：
```
The value of the 'Access-Control-Allow-Origin' header in the response must not be the wildcard '*' when the request's credentials mode is 'include'.
```

**修复方案：**

方案 A（推荐，明确域名）：
```json
"allow_origins": "http://localhost:5173,http://localhost:5174"
```

方案 B（正则匹配）：
```json
"allow_origins": "~.*"  // 允许所有（但不发送 credentials 头）
```

### P1-2. §7.1 JSON 文件缺少 `serverless-pre-function` — Casbin 鉴权 sub 为空

> **位置：** §7.1 api-v1-route.json 第374-378行

```json
"plugins": {
  "jwt-auth": {},
  "authz-casbin": {
    "username": "X-User-Role"
  }
}
```

**问题：**
§5.1 业务路由示例（第172-173行）包含 `serverless-pre-function` Lua 代码，负责从 JWT payload 提取 `roles` 数组并设置 `X-User-Role` header。

但导出的 JSON 文件 `api-v1-route.json` **没有** serverless-pre-function。部署到 etcd 后：
- `X-User-Role` header 永远为空
- authz-casbin `username: "X-User-Role"` 读取空字符串
- Casbin sub = "" → regexMatch 匹配空 sub → 所有Policy不匹配 → **所有请求 403 拒绝**

**修复：**
在 JSON 中加入 serverless-pre-function：
```json
{
  "serverless-pre-function": {
    "phase": "rewrite",
    "functions": ["return function(conf, ctx) ... end"]
  },
  "jwt-auth": {},
  "authz-casbin": {
    "username": "X-User-Role"
  }
}
```

### P1-3. apisix-setup.ps1 脚本变量损坏 — 无法输出验证命令

> **位置：** §6 第355行

```powershell
Write-Host "  curl -H 'X-API-KEY: $env:A...KEY' http://localhost:9180/apisix/admin/routes"
```

**问题：**
`$env:A...KEY` 是损坏的变量引用，PowerShell 解析时会尝试找名为 `A...KEY` 的环境变量，返回 `$null`。

**修复：**
```powershell
Write-Host "  curl -H 'X-API-KEY: $env:APISIX_ADMIN_KEY' http://localhost:9180/apisix/admin/routes"
```

### P1-5. §4.3 JWKS 端点实现与 §7.4 jwks-route.json 不一致

> **位置：** §4.3 第138-151行（mocking 插件）vs §7.4 第433-442行（public-api 插件）

| §4.3 方式一 | §7.4 JSON 文件 |
|:---|:---|
| `uri: /well-known/jwks` | `uri: /well-known/jwks` |
| plugins: **mocking** | plugins: **public-api** |
| 返回静态 JWKS JSON | 无上游配置，public-api 单独使用无意义 |

**问题：**
`public-api` 插件用于"将 PostgREST 的某个端点公开（无需 JWT 验证）"，但需要指向一个 upstream 服务（通常是 PostgREST）。单独使用 `public-api` 不配置 upstream，APISIX 无法处理请求。

**修复：**
方案 A（对齐 §4.3，mocking 插件）：
```json
{
  "uri": "/well-known/jwks",
  "plugins": {
    "mocking": {
      "content_type": "application/json; charset=utf-8",
      "response_status": 200,
      "response_example": "{...真实 JWKS...}"
    }
  }
}
```

方案 B（重定向到 Casdoor）：
```json
{
  "uri": "/well-known/jwks",
  "plugins": {
    "redirect": {
      "uri": "http://casdoor:8000/.well-known/jwks"
    }
  }
}
```

方案 C（代理转发到 Casdoor）：
```json
{
  "uri": "/well-known/jwks",
  "upstream": {
    "nodes": { "casdoor:8000": 1 }
  }
}
```

## 🟢 P2 增强级

### P2-1. HS256 Secret 仅 31 字符，低于 32 字节最小要求

> **位置：** §4.2 第128行 `"dev_jwt_secret_min_32_chars___"`

**问题：**
- `dev_jwt_secret_min_32_chars___` 长度为 31 个字符
- HS256 要求密钥 ≥ 256 位（32 字节）
- 若 secret 为 ASCII，31 字符 < 32 字节 → APISIX 可能拒绝或补齐为弱密钥

**修复：** 改为 `dev_jwt_secret_at_least_32_chars!!`

---

### P2-2. RPC 路径覆盖未明确

> **位置：** §5.1 业务路由 `uri: "/api/v1/*"`

**问题：**
- PostgREST RPC 端点路径为 `/rpc/function_name`
- APISIX 路由 `/api/v1/*` 重写 `^/api/v1/(.*)` → `/$1`
- 所以 `/api/v1/rpc/user_login_sso` 会变成 `/rpc/user_login_sso`，匹配 PostgREST ✅
- 但如果前端直接调用 `/rpc/user_login_sso`（不经 APISIX 网关），无法享受 JWT 验证

**建议：** 明确说明前端应通过网关 `/api/v1/*` 访问，不要直连 PostgREST。

---

### P2-3. serverless-pre-function Lua 缺少错误处理和日志

> **位置：** §5.1 第172行 Lua 代码

**问题：**
- 如果 JWT payload 缺少 `roles` 字段，X-User-Role header 不设置
- 如果 roles 不是数组而是字符串，`table.concat` 会报错
- 没有任何 `ngx.log` 记录异常情况

**建议：** 添加防御性代码：
```lua
if type(jwt_json.roles) == "string" then
    roles_str = jwt_json.roles
elseif type(jwt_json.roles) == "table" then
    roles_str = table.concat(jwt_json.roles, ",")
else
    roles_str = ""
    ngx.log(ngx.WARN, "JWT missing or invalid roles claim")
end
```

### P2-4. §4.3 Mocking JWKS n 值为短占位符 — RS256 验证必然失败

> **位置：** §4.3 第146行 `"n": "sInpb5E1_ym0f1RfjxN4W04OMJkdTxXEF9sPse2_6SkiCzJIe0Qn0G7bhQ=="`

**问题：**
- 这是 44 字符的 base64url（约 256 位 = 32 字节）
- 真实 2048 位 RSA modulus 需要 342+ 字符的 base64url
- APISIX jwt-auth 会尝试解析 JWKS → modulus 长度不匹配 → RS256 公钥加载失败 → 所有 JWT 验证失败

**修复：** 使用真实 RSA 密钥的 JWKS，或从 Casdoor 获取。
