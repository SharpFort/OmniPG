## 🔴 P0 阻塞级

### P0-1. JWT 签发路径与 04.5-Casdoor 架构直接矛盾 — PostgREST 无法验证签名

> **位置：** §3.1 方案 A 第148-170行「从 sys_secret 读取私钥」

**问题：**
09 §3.1 方案 A 推荐：
1. 从 `sys_secret` 读取 RSA 私钥
2. 提取公钥构造 JWKS JSON
3. PostgREST 用此 JWKS 验证 JWT

但此方案与已确认的架构矛盾：

| 来源 | JWT 签发方 | 公钥来源 | 私钥位置 |
|:---|:---|:---|:---|
| 00 总纲 §6.2 | **Casdoor** | Casdoor JWKS 端点 | **不暴露给 PG** |
| 04.5-Casdoor集成 | **Casdoor** | APISIX 轮询 JWKS | **Casdoor 内部** |
| 07 / 09 当前 | `plpython3u + PyJWT` | PG 内提取 | `sys_secret.jwt_private_key_pem` |

**后果：**
- PG 内 `plpython3u` 不存在（07 P0-1 已确认）→ 无法签发 JWT
- 即使 PG 能签发 JWT，需两个密钥对：PG 内一对 + Casdoor 一对，**密钥管理混乱**
- APISIX jwt-auth 也需验签（通过 Casdoor JWKS 端点），两处验证使用不同密钥 → **结果不一致**

**修复方案（对齐 04.5 + v2 审查成果）：**

完全移除 09 §3.1 方案 A/B 中关于"从 sys_secret 读取私钥"的路径，统一为：

```ini
# 开发环境：Casdoor 运行时的 JWKS
jwt-secret = "${JWKS_JSON}"

# JWKS_JSON 获取方式：
# 开发：docker exec app-casdoor cat /app/conf/init_data.json | jq '.certs[].certificate'
# 或从 Casdoor 管理面板直接复制 JWKS
```

Casdoor 启动后自动生成 RS256 证书并暴露 `/.well-known/jwks` 端点。PostgREST 和 APISIX 共用同一套 JWKS：

```powershell
# 从 Casdoor 容器获取 JWKS
$JWKS = docker exec app-casdoor cat /app/conf/init_data.json | ConvertFrom-Json | Select-Object -ExpandProperty certs | Select-Object -First 1 | ForEach-Object { $_.certificate }
# 设置到 .env
```

**User_login_sso / refresh_token_rtr 必须改为调用 Casdoor 端点获取 JWT**（对齐 07 P0-1 方案 A）。

## 🟡 P1 重要级

### P1-1. `api_v1.check_token_blacklist` 函数不存在 — PostgREST 启动失败

> **位置：** §2.1 第91行 `PGRST_DB_PRE_REQUEST: "api_v1.check_token_blacklist"`

**问题：**
07 Migration 005 在 **public** schema 创建了 `check_token_blacklist()` 函数。PostgREST `db-schemas = "api_v1"` 意味着搜索路径以 `api_v1` 为主，但 `db-pre-request` 限定了 `api_v1.check_token_blacklist`。

由于 PostgREST 用 `SET search_path TO api_v1, public` 执行请求，理论上 `public.check_token_blacklist()` 可见。但 `db-pre-request` 的限定格式 `api_v1.check_token_blacklist` 会严格在 `api_v1` 中查找。

**后果：** PostgREST 400 错误：`function api_v1.check_token_blacklist() does not exist`

**修复：**
在 08 的 `02-schemas.sql` 追加：
```sql
CREATE OR REPLACE FUNCTION api_v1.check_token_blacklist()
RETURNS void
LANGUAGE sql
SECURITY DEFINER
SET search_path = public, pg_temp
AS $$ SELECT public.check_token_blacklist() $$;
```

### P1-2. `jwt-role-claim-key` 格式与 Casdoor JWT 不匹配

> **位置：** §1 第69行 `jwt-role-claim-key = "roles[0]"`

**问题：**
PostgREST 的 `jwt-role-claim-key` 通过 JSON-path 语法提取角色。Casdoor JWT 的 roles 数组通常在 `roles` 或 `tag` claim 下：

```json
{
  "sub": "user-uuid",
  "roles": ["super_admin"],
  "tag": "super_admin"
}
```

`jwt-role-claim-key = "roles[0]"` 提取出 `"super_admin"`（字符串），PostgREST 用此字符串切换到名为 `super_admin` 的 PG 角色。

**潜在问题：**
1. JWT 中 roles 数组为空 → PostgREST 切换到空角色 → 认证失败（PostgREST 行为：切换失败返回 401）
2. PG 中必须存在名为 `super_admin` 的角色（07 Migration 只创建 `web_anon` / `authenticated`，**没有 `super_admin` PG 角色**）

**后果：** 登录 → JWT 含 roles=["super_admin"] → PostgREST 尝试 `SET ROLE super_admin` → 角色不存在 → **401 Unauthorized**

**修复：**
- 在 `02-schemas.sql` 中创建 PG 角色：
  ```sql
  CREATE ROLE super_admin NOLOGIN NOINHERIT;
  CREATE ROLE role_admin NOLOGIN NOINHERIT;
  CREATE ROLE role_editor NOLOGIN NOINHERIT;
  CREATE ROLE role_guest NOLOGIN NOINHERIT;
  ```
- 或通过中间角色映射：用 `jwt-role-claim-key = ".role"` 指向单个字符串 claim，再由 JWT 签发方控制

### P1-3. `check_token_blacklist()` 不过滤已过期 JTI — 黑名单无限膨胀

> **位置：** §4.2 第214行 `WHERE jti = v_jti`

**问题：** 黑名单无时间过滤：
```sql
SELECT 1 FROM sys_token_blacklist WHERE jti = v_jti
```

Token 过期后（expired_at < now()），只要 jti 仍在黑名单中，就永远拦截。这导致：
- 黑名单随时间线性增长，即使 Token 自然过期也无法移除
- 检查效率随黑名单大小下降

**修复：**
```sql
SELECT 1 FROM sys_token_blacklist 
WHERE jti = v_jti AND expired_at > now()
```

同时依赖 pg_cron 清理过期 jti（07 缺少，需补充）。

### P1-4. JWKS 格式变量名不一致 — 开发/生产环境均有歧义

> **位置：** §1 第35行 `jwt-secret = "$(JWKS_JSON)"`

**问题：** 08 的 `.env.example` 第405行变量名为 `JWKS_JSON`，第376行有一个旧的 `JWK_JSON`（少 S）。09 文档统一用 `JWKS_JSON`，但 PostgREST 环境变量名是 `PGRST_JWT_SECRET`。

**后果：** 用户复制 08 的 `.env.example` 后，可能会混淆：

| 变量名 | 出现位置 | 用途 |
|:---|:---|:---|
| `JWKS_JSON` | 08 `.env.example`, 09 postgrest.conf | PostgREST JWT 密钥 |
| `JWK_JSON` | 08 `.env.example` | 废弃/重复 |
| `JWKS_JSON` | 08 docker-compose.yml | 注入 PostgREST |

**修复：** `.env.example` 中删除 `JWK_JSON`，统一为 `JWKS_JSON` 并添加注释说明格式。

### P1-5. CORS 配置未与 APISIX 网关 CORS 协调

> **位置：** §1 第66行 `cors-origins = "http://localhost:5173"`

**问题：** PostgREST 在开发环境和 APISIX 网关后面都启用了 CORS 09:
```ini
cors-origins = "http://localhost:5173"
```

APISIX §2.3 第101行：
```json
"allow_origins": "${CORS_ORIGINS:http://localhost:5173}"
```

双重 CORS 可能导致：
- 浏览器先向 APISIX 发 OPTIONS → APISIX 返回 CORS 头 → 浏览器再发实际请求到 APISIX → 转发到 PostgREST → PostgREST 也返回 CORS 头 → **重复 CORS 头**

**建议：** 开发环境中 PostgREST 端口 3000 仅被 APISIX 网关访问（不在浏览器直接调用），可关闭 PostgREST CORS：
```ini
# 通过 APISIX 网关访问时不需要 PostgREST 自己的 CORS
# cors-origins = "" 
```

## 🟢 P2 增强级

### P2-1. 缺少错误处理配置

**问题：** 未配置 `db-plan-enabled` 或 `log-level`。PostgREST v14 支持结构化日志。

**建议：**
```ini
log-level = warn
db-plan-enabled = true  # 开发环境可选，用于 EXPLAIN 分析
```

---

### P2-2. 缺少连接池相关配置

**问题：** 通过 Pgbouncer 连接但未配置 `db-pool` 和 `db-pool-acq-useconds`。

PostgREST 默认连接池大小 = CPU 核数，开发环境可用：
```ini
db-pool = 10
db-pool-acq-useconds = 5000
```

---

### P2-3. 缺少优雅关闭配置

**问题：** PostgREST 支持 `server-unix-socket` 和优雅关闭超时，09 未提及。

**建议：** 生产环境需指定优雅关闭时间以等待进行中的请求完成。
