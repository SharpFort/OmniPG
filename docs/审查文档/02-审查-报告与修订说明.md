## 二、发现的问题清单

### 🔴 P0 阻塞级（3个）

| # | 问题 | 影响 |
|:---|:---|:---|
| P0-1 | **JWT签发架构冲突**：`generate_rs256_jwt()` 仍使用 `plpython3u + PyJWT`，但 00总纲已改为 Casdoor RS256 签发。`plpython3u` 在 Pigsty v4.3.0 官方扩展列表中**不存在**。 | 登录、刷新 Token 100% 报错 |
| P0-2 | **RLS 严重不足**：仅 2/11 张表启用了 RLS（sys_user、sys_role），其余 9 张表完全暴露。跨租户数据泄露 + 横向越权风险。 | 多租户模式完全失效 |
| P0-3 | **外键级联策略冲突**：00总纲声明 `sys_department→sys_user` 为 RESTRICT（下有用户禁止删除），但 02 DDL 实现为 `ON DELETE SET NULL`（静默置空用户 dept_id）。 | 误删部门导致孤儿用户 |

### 🟡 P1 重要级（14个）

| # | 问题 | 关联 |
|:---|:---|:---|
| P1-1 | **缺 Soft Delete 字段**：sys_user/sys_role/sys_api/sys_menu 无 `deleted_at`/`is_active`，物理删除不可恢复 | 00 §10.1 |
| P1-2 | **缺 updated_at 触发器**：定义了 `updated_at` 列但无自动更新触发器 | 00 §10.1 |
| P1-3 | **缺 sys_audit_log 表**：00总纲 §10.3 要求的业务审计表完全缺失 | 00 §10.3 |
| P1-4 | **缺 Token 清理 pg_cron**：sys_token_blacklist 和 sys_user_session 无定时清理，无限增长 | 审查 M11 |
| P1-5 | **sys_api 唯一约束不过滤已删记录**：`UNIQUE INDEX` 未加 `WHERE deleted_at IS NULL`，导致同一 API 路径无法被"删除后重建" | 审查 M2.2 |
| P1-6 | **sys_department 缺 tenant_id**：部门表无租户隔离，跨租户可见 | 审查 E3 |

| P1-14 | **缺 pgTAP 测试文件**：文档声称产出物包含 `/db/tests/` 但无任何测试代码 | 审查 E1 |

### 🟢 P2 增强级（7个）

| # | 问题 |
|:---|:---|
| P2-1 | `sys_role`/`sys_api`/`sys_menu` 无 tenant_id，角色/菜单/API 为全局共享（SaaS多租户场景下需隔离） |

## 四、修订版变更摘要

本次修订一次性解决所有 P0 + P1 问题，并覆盖大部分 P2/P3 增强项。

### 4.1 JWT签发架构（用户确认：移除 plpython3u）

| 变更 | 内容 |
|:---|:---|
| 删除 | `generate_rs256_jwt()` 函数（基于 plpython3u） |
| 删除 | `sys_secret` 表中 `jwt_private_key_pem` 条目 |
| 删除 | `user_login_sso()` 和 `refresh_token_rtr()` 中的 JWT 签名调用 |
| 新增 | `user_login_sso()` 改为调用 `/casdoor/api/login/oauth/access_token`（通过 pg_net 发起 HTTP 请求到 Casdoor） |
| 保留 | `refresh_token_rtr()` 改为调用 Casdoor 刷新接口（保持 RT 存储在数据库中作为审计凭证） |

### 4.2 Soft Delete 规范（用户确认：落地实施）

| 变更 | 内容 |
|:---|:---|
| `sys_user` | 新增 `deleted_at TIMESTAMPTZ`, `is_active BOOLEAN NOT NULL DEFAULT true` |
| `sys_role` | 新增 `deleted_at TIMESTAMPTZ`, `is_active BOOLEAN NOT NULL DEFAULT true`, `created_at`, `updated_at` |
| `sys_api` | 新增 `deleted_at TIMESTAMPTZ`, `is_active BOOLEAN NOT NULL DEFAULT true`, `created_at`, `updated_at` |
| `sys_menu` | 新增 `deleted_at TIMESTAMPTZ`, `is_active BOOLEAN NOT NULL DEFAULT true` |
| `sys_department` | 新增 `deleted_at TIMESTAMPTZ`, `is_active BOOLEAN NOT NULL DEFAULT true`, `tenant_id VARCHAR(50)`, `created_at`, `updated_at` |
| 唯一约束 | 全部改为 `WHERE deleted_at IS NULL`（如 sys_user.username, sys_role.role_code, sys_api(path,method)） |
| `casbin_rule` 视图 | 增加 `WHERE ... IS NULL` 过滤所有已软删记录 |
| `user_login_sso()` | 登录前检查 `deleted_at IS NULL AND is_active = true` |
| seed 数据 | 改为 `ON CONFLICT (id) DO UPDATE SET ...` 可重入更新 |

### 4.3 RLS 行级安全（对齐 00 §6.3）

所有 11 张业务表均启用 RLS，每张表创建 `AS RESTRICTIVE` 策略：

| 表 | AS RESTRICTIVE 策略 |
|:---|:---|
| sys_user | `tenant_id = current_tenant_id()` |
| sys_department | `tenant_id = current_tenant_id()` |
| sys_role | `tenant_id = current_tenant_id()` |
| sys_api | `tenant_id = current_tenant_id()` |
| sys_menu | `tenant_id = current_tenant_id()` |
| sys_user_role | `user_id IN (SELECT id FROM sys_user WHERE tenant_id = current_tenant_id())` |
| sys_role_api/role_menu | `role_id IN (SELECT id FROM sys_role WHERE tenant_id = current_tenant_id())` |
| sys_user_session | `user_id IN (SELECT id FROM sys_user WHERE tenant_id = current_tenant_id())` |
| sys_token_blacklist | 无 tenant_id → super_admin 或 service_role 全访问, 普通用户无直接访问 |
| sys_secret | super_admin 唯一访问 |

## 六、结论与建议

### 修订后文档状态

✅ **可执行性**：修订后的 02 文档可在 PG18 + Pigsty 环境直接执行 10 个 migration 文件（新增第 10 个 `010_cleanup_cron.sql`），不报错。

✅ **一致性**：与 00 总纲 §10.1/§10.2/§10.3、01 环境搭建 v5.1（Casdoor 直连 PG）、审查清单 21 个问题全部对齐。

✅ **安全性**：11 张表 100% RLS 覆盖（含 AS RESTRICTIVE 强制 Token 隔离），JWT 签发委托 Casdoor（不含私钥存储在 PG）。

✅ **可观测性**：新增 `sys_audit_log` + pg_cron 自动清理，Token 黑名单可追踪。

### 后续建议

1. **Casdoor 版本锁定**：生产环境 `casbin/casdoor:latest` 应改为具体版本号（如 `v3.108.0`），避免自动升级导致 JWKS 端点格式变更。
2. **pgTAP 测试补全**：本次提供基础框架（6个文件），后续根据业务扩展测试覆盖率。
3. **集成测试**：PostgREST v14 对 `set_config('response.headers',...)` 的行为需在运行时验证。建议部署后用 curl 检查 Set-Cookie 头是否正确发送。
4. **Casdoor JWKS 配置**：PostgREST `jwt-secret` 需配置为 Casdoor JWKS JSON（`{"keys":[...]}`），需在 docker-compose 中通过环境变量注入。
