## 🔴 P0 阻塞级

### P0-1. JWT 签发架构与 Casdoor 方案矛盾 — Migration 005 无法执行

> **位置：** `Migration 005` 第416-428行 `generate_rs256_jwt()`

**问题：**
07 重新引入了基于 `plpython3u + PyJWT` 的 `generate_rs256_jwt()` 函数，并在 `user_login_sso()` (第453行) / `refresh_token_rtr()` (第545行) 中调用它进行 JWT 签名。但这与已确认的架构直接矛盾：

| 文档 | 架构决策 |
|:---|:---|
| `00-项目总纲` §6.2 | JWT 签发由 Casdoor 完成，PG 不存储私钥 |
| `04.5-Casdoor集成` | APISIX jwt-auth 通过 Casdoor JWKS 公钥验签 |
| `02-v2审查报告` §4.1 | `plpython3u` 已移除，改为调用 Casdoor `/casdoor/api/login/oauth/access_token` |
| `Pigsty v4.3.0` 扩展 list | **不包含 `plpython3u`** |

**后果：**
- 开发环境 Docker 镜像 `pgcharles/pgtap:18` 无 `plpython3u`
- 生产环境 Pigsty 无此扩展，migration 005 执行失败
- 即使强制安装 `plpython3u`，仍需 `pip install pyjwt` 进 PostgreSQL 的 Python 运行时

**修复方案：**

**方案 A（推荐，对齐 v2 / 04.5）：** 移除 `generate_rs256_jwt()`，重写 `user_login_sso()` / `refresh_token_rtr()` 调用 Casdoor 端点通过 `pg_net` 获取 JWT：

```sql
-- Migration 005 重写
CREATE OR REPLACE FUNCTION user_login_sso(p_username text, p_password text)
RETURNS json AS $$
DECLARE
    v_response http_response;
    v_user_rec RECORD;
    -- ... 其他变量
BEGIN
    -- 1. 验证用户名密码（bcrypt）
    -- 2. 生成 RT，存储会话
    -- 3. 调用 Casdoor 签发 JWT
    v_response := http_post(
        'http://casdoor:8000/api/login/oauth/access_token',
        'grant_type=password&username=' || p_username || '&password=' || p_password,
        'application/x-www-form-urlencoded'
    );
    v_new_at := v_response.content::json->>'access_token';
    -- 4. 注入 Cookie + 返回
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**方案 B（临时回退 — plpython3u 不可行）：** 若坚持纯 PG 内签名，需：
- 将 RSA 私钥移至 Casdoor，`user_login_sso` 仍调用 Casdoor 签 JWT
- 或在开发环境通过 Docker 额外安装 plpython3u（偏离 Pigsty 生产路径）

### P0-2. 种子数据 RSA 私钥为无效占位符 — 登录即报 PEM 解析错误

> **位置：** `Migration 009` 第998-1004行

**问题：**
即使假设 `plpython3u` 已解决，sys_secret 中插入的私钥是：

```
-----BEGIN RSA PRIVATE KEY-----
MIIEowIBAAKCAQEA0Z3VS5JJcds3xfn/ygWyF8PbnGy0AHB7MhgHcTz6sE2yPB
kC7v5jX2wK1y0aBcDeFgHiJkLmNopQrStUvWxYz01234567890abcdefghijklmnop
...[注意：请替换为实际的 2048 位 RSA 私钥，以上仅为占位符]
-----END RSA PRIVATE KEY-----
```

**缺陷：**
1. PEM 总长度约 200 字符，远不及 2048 位 RSA 应有的 ~1600+ 字符
2. 包含中文注释 `...[注意...]`，非法 base64 字符
3. `generate_rs256_jwt()` 调用 `jwt.encode()` 时必然触发 `ValueError: Could not parse PEM data`

**修复：**
- 注入前用 `openssl genrsa -out private.pem 2048` 生成真实 2048 位密钥
- 生产环境：私钥应通过 KMS/Vault 注入，不落地 migration 文件
- 或留空，依赖 Casdoor 签 JWT（对齐 P0-1 方案A）

### P0-3. `sys_secret` 表无 RLS 保护 — 任何认证用户可读取 JWT 私钥

> **位置：** `Migration 008 RLS 策略` 第866-973行

**问题：**
Migration 008 仅为 4 张表创建了 RLS：`sys_user` / `sys_department` / `sys_role` / `sys_menu`。**sys_secret、sys_token_blacklist、sys_api、sys_user_session、sys_user_role_request 均无 RLS**。

特别地，`sys_secret` 存储着 JWT 私钥。PostgREST 公开 `GET /sys_secret` 后，任何通过认证的用户（即便是最基础的 `role_guest`）均可执行：

```bash
curl http://localhost:3000/sys_secret \
  -H "Authorization: Bearer <任意合法JWT>"
```

**返回：** `{"key_name":"jwt_private_key_pem", "key_value":"-----BEGIN RSA PRIVATE KEY-----..."}`

**后果：** 攻击者获得 JWT 私钥后，可签发任意 roles 包括 `super_admin` 的 JWT，**完全绕过 Casdoor**，系统彻底沦陷。

**修复：** 至少为 sys_secret 创建 RLS：
```sql
ALTER TABLE sys_secret ENABLE ROW LEVEL SECURITY;

-- 仅超级管理员可读（通过 service_role 绕过 RLS）
CREATE POLICY secret_no_anonymous ON sys_secret
AS RESTRICTIVE USING (false);  -- 默认拒绝所有

-- 需要通过 SECURITY DEFINER 函数暴露
-- 或仅允许特定角色访问
```

## 🟡 P1 重要级

### P1-1. `casbin_rule` 视图未过滤无效数据 — 已禁用角色/API 仍可通过鉴权

> **位置：** `Migration 003` 第330-343行

**问题：** `create_casbin_view.sql` 中视图未加入任何 WHERE 过滤：
```sql
CREATE OR REPLACE VIEW casbin_rule AS
SELECT ...
FROM sys_role_api ra
JOIN sys_role r ON ra.role_id = r.id
JOIN sys_api a ON ra.api_id = a.id;
```

07 的 DDL 为 sys_role/sys_api/sys_menu 增加了 `is_active` 字段，但视图未利用它。结果：

- 角色 `is_active = false` → 其绑定 API 仍在 casbin_rule 中 → **网关仍放行该角色对这些 API 的访问**
- API `is_active = false` → 策略仍在 casbin_rule → **已下线接口仍可通过鉴权**

**安全影响：** 管理员"禁用角色"、"关闭 API 资源"的操作对网关鉴权**完全无效**。

**修复：**
```sql
CREATE OR REPLACE VIEW casbin_rule AS
SELECT ...
FROM sys_role_api ra
JOIN sys_role r ON ra.role_id = r.id AND r.is_active = true
JOIN sys_api a ON ra.api_id = a.id AND a.is_active = true;
```

### P1-2. RLS 缺失 7 张表 — 跨租户数据大面积泄露

> **位置：** `Migration 008 RLS 策略`

**问题：** 以下表完全无 RLS，任何认证用户可跨租户读写：

| 表 | 风险类型 | 危害示例 |
|:---|:---|:---|
| `sys_api` | 越权读取 | A 租户管理员查看 B 租户 API 资源列表 |
| `sys_menu` | 越权读取 | A 租户获取 B 租户菜单树 |
| `sys_user_role` | 跨租户关联 | 给用户分配其他租户角色 |
| `sys_role_api` | 策略越权 | 修改其他租户角色-API 绑定 |
| `sys_user_session` | 隐私泄露 | 查看其他用户的活跃会话 JTI |
| `sys_user_role_request` | 审批流泄露 | 查看/审批其他租户申请 |

**注：** sys_token_blacklist 同样无 RLS，但黑名单通常不需要租户隔离。而 sys_secret（见 P0-3）必须受保护。

**修复：** 迁移 008 扩展为每张表创建 `AS RESTRICTIVE` + 具体策略，对齐 02 v2 §4.3。

### P1-3. 缺少 `updated_at` 自动更新触发器

> **位置：** `Migration 001-002` 所有表定义了 `updated_at` 但无触发器

**问题：**
DDL 中所有表包含：
```sql
updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
```

但没有任何 migration 创建触发器自动更新该列。结果：

- 记录 UPDATE 后 `updated_at` 永远不会改变（仍然是 `created_at` 的值）
- 00 总纲 tracking 文档中要求的"按 updated_at 排序"功能失效
- 02 v2 M5 已识别此问题，使用通用触发器 `update_updated_at()` 修复

**修复：** 在 Migration 002 末尾添加通用触发器函数，Migration 001-002 各表绑定：
```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 为每个含 updated_at 的表绑定
CREATE TRIGGER trg_user_updated_at BEFORE UPDATE ON sys_user
FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

### P1-4. `get_user_menu()` 返回扁平数组 — 前端需重建菜单树

> **位置：** `Migration 006` 第690-749行

**问题：**
函数定义为 `RETURNS json`，返回扁平数组（按 sort_order 排序的顶级菜单 + 按钮），不含 `children` 嵌套结构。前端组件期望接收形如：

```json
[{
  "id": "...", "parent_id": null, "children": [
    { "id": "...", "parent_id": "...", "children": [...] }
  ]
}]
```

的树形结构。后端扁平输出需要前端递归组装，增加前端复杂度且不符合常规对接约定。

**修复：** 使用递归 CTE + `jsonb_build_object` 直接在 PG 内构造嵌套 JSON（02 v2 §4.10已实现）。

---

### P1-5. `pg_notify` payload 仅为 `"reload"` — 策略 Syncer 只能全量同步

> **位置：** `Migration 004` 第373行

```sql
PERFORM pg_notify('casbin_channel', 'reload');
```

**问题：** 与 v2 M9 不一致。Syncer 收到 `"reload"` 后不知道变更内容，只能触发全量 SELECT + PUT。如果后续 Syncer 升级为"根据 payload 做增量更新"，则需要更丰富的 JSON payload。

**建议：** 增强为 JSON：
```sql
PERFORM pg_notify('casbin_channel', json_build_object(
    'op', TG_OP,
    'table', TG_TABLE_NAME,
    'ts', extract(epoch from now())::bigint
)::text);
```

### P1-6. 种子数据 `ON CONFLICT DO NOTHING` — 无法重入更新

> **位置：** `Migration 009` 所有 INSERT 语句

```sql
ON CONFLICT DO NOTHING;
```

**问题：** 首次执行可成功开发部署。但管理员手动修改种子数据（如新增菜单、API）后重新运行，不会更新已有记录。

**影响：** 
- 开发时修改种子需先 `DELETE` 再重新运行 migration
- 与 v2 M3 不一致（v2 改为 `ON CONFLICT (id) DO UPDATE SET ...`）

**建议：** 改为 `ON CONFLICT (id) DO UPDATE` 可重入。

### P1-7. Cookie `Secure` 标志在开发环境 HTTP 下失效

> **位置：** `Migration 005` user_login_sso 第528行 / refresh_token_rtr 第624行

```sql
'...;Path=/; HttpOnly; Secure; SameSite=Strict; Max-Age=604800'
```

**问题：**
1. `Secure` 标志要求必须 HTTPS 传输才存储 Cookie。开发环境 Vite 默认 `http://localhost:5173`，浏览器不存储 `Secure` Cookie → 刷新 Token 时找不到 RT
2. `Path=/` 导致 refresh_token 随所有 PostgREST 请求发送（非必要暴露），v2 用 `/rpc/refresh_token` 更精确

**修复（开发环境）：** `Secure` → 仅生产启用；`Path=/` → `/rpc/refresh_token`

---

### P1-8. 缺少 Token 清理机制 — 数据无限增长

> **位置：** 全文档无 cleanup 函数/pg_cron 任务

**问题：** `sys_token_blacklist` 和 `sys_user_session` 只会插入不会清理。即使 Token 过期 (`expired_at < now()`)，记录也永久保留。

**影响：**
- 磁盘空间占用（百万级 Token 黑名单）
- 全表扫描性能下降
- `check_token_blacklist()` 检查效率降低

**修复：**
1. 在 `check_token_blacklist()` 中加入 `AND expired_at > now()` 过滤已过期 jti
2. 创建 pg_cron 定时任务清理过期记录（对齐 v2 M11）
3. 或在用户会话表上建 partial index WHERE expired_at > now()

---

### P1-9. 审批流表创建位置偏后

> **位置：** `Migration 006` 第755-768行

**问题：** `sys_user_role_request` 在第六个 migration 创建。但 `approve_role_request`（也在 006）依赖此表，若审批功能在早期被调用则会报错。

原 v2 将其前置到 Migration 002。当前顺序虽然能工作（同 migration 内顺序执行），但一旦 migration 文件拆分出错就需要重新排序。

### P1-10. 缺失审计表 `sys_audit_log`

> **位置：** 00 总纲 §10.3 要求，07 未实现

**问题：** 00 总纲明确要求所有业务变更记录到审计日志。07 既无 `sys_audit_log` 表也无审计触发器。

**影响：** 无法满足企业合规审计要求。

### P2-3. 硬编码 UUID 跨环境一致性

**问题：** seed 数据中大量使用 `00000000-0000-0000-0000-XXXXXXXXXXXX` 格式的 UUID。开发/测试/生产环境执行相同 migration 会生成相同 ID，但迁移数据跨环境恢洉时可能冲突。

**建议：** 若为初始种子则保持；若为测试数据建议用 `gen_random_uuid()`。
