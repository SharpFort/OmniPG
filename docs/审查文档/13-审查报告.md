## 🟡 P1 重要级

### P1-1. 生产 APISIX 镜像版本 3.11.0 与开发 3.17.0 不一致

> **位置：** §3.4 第284行

```yaml
image: apache/apisix:3.11.0-debian
```

**问题：**
- 开发环境（08）使用 `apache/apisix:3.17.0-debian`
- 生产环境（13）使用 `apache/apisix:3.11.0-debian`
- 版本跨度 6 个次版本（约 18 个月）
- v3.11.0 可能存在已知安全漏洞和 Bug，这些在 v3.17.0 中已修复

**修复：**
```yaml
image: apache/apisix:3.17.0-debian  # 与开发环境保持一致
```

### P1-2. PostgreSQL HA 配置逻辑矛盾 — `replicas: 1` 与 Patroni 不兼容

> **位置：** §3.2 第196行 + §3.3

```yaml
# StatefulSet
replicas: 1  # 主库单节点，通过 Patroni 做 HA
```

**问题：**
- Patroni 是 **基于 etcd 的 PostgreSQL 高可用方案**，需要 **至少 3 个 PG 节点**才能实现自动故障转移（Raft 多数派）
- `replicas: 1` 意味着只有一个 PG 节点，Patroni 无法实现 HA
- 文档§1.2 资源规划写 "1+2"（1 主 + 2 副本），但 StatefulSet 中 `replicas: 1`

**修复方案：**

方案 A（推荐 — Patroni 3 节点）：
```yaml
# 使用 Patroni 的 3 节点 PostgreSQL
replicas: 3
# + 配置 Patroni 使用 etcd 作为 DCS
```

方案 B（简化 — 单节点 + 定时备份）：
如果不要 Patroni HA，应明确说明生产为单可用区单节点，RTO/RPO 指标需要手动管理。

### P1-3. Syncer 引用未定义的 `apisix-admin` Service

> **位置：** §3.5 第357行

```yaml
APISIX_ADMIN_URL: "http://apisix-admin.system.svc.cluster.local:9180/apisix/admin/plugin_metadata/authz-casbin"
```

**问题：**
- §3.4 APISIX Deployment 仅定义了 Pod，**没有定义 Service**
- Syncer 引用了 `apisix-admin` 这个 Service 名称，但文档中没有创建该 Service

**修复：**
添加 APISIX Service 定义：
```yaml
apiVersion: v1
kind: Service
metadata:
  name: apisix-admin
  namespace: system
spec:
  selector:
    app: apisix
  ports:
    - port: 9180
      targetPort: 9180
      name: admin
    - port: 9080
      targetPort: 9080
      name: http
```

### P1-5. PostgREST `cache-enabled` 参数在 v14.x 中不存在

> **位置：** §9.3 第951行

```ini
# postgrest.conf
cache-enabled = true
```

**问题：**
- PostgREST v14.x（包括 v14.14）**没有 `cache-enabled` 这个配置参数**
- PostgREST 的计划缓存通过 `db-plan-cache` 控制（v12+）
- 如果用户复制此配置，PostgREST 会报错：`unknown configuration parameter 'cache-enabled'`

**修复：**
```ini
# 正确的缓存配置（v14）
db-plan-cache = true  # 缓存查询计划（v12+）
# 或者移除该配置，默认行为即可
```

### P2-2. 缺少 `pg_hba.conf` 生产配置

> **位置：** 全文无 `pg_hba.conf` 配置

**问题：**
- 生产环境 PostgreSQL 需要配置 `pg_hba.conf` 限制连接来源
- 当前配置依赖 K8s Service 但未限制哪些 Pod 可以连接 PG

**修复：**
在 PostgreSQL StatefulSet 的 `command` 中添加：
```yaml
command:
  - postgres
  - -c
  - hba_file=/etc/postgresql/pg_hba.conf
```

并在 ConfigMap 中提供 `pg_hba.conf`：
```
# TYPE  DATABASE        USER            ADDRESS                 METHOD
local   all             postgres                                peer
host    app_db          app_owner       10.244.0.0/16           scram-sha-256
host    app_db          authenticator   10.244.0.0/16           scram-sha-256
host    all             all             0.0.0.0/0               reject
```

### P2-3. 备份 CronJob 缺少 `AWS_REGION` 环境变量

> **位置：** §6.1 第614-615行

```yaml
- name: AWS_ACCESS_KEY_ID
  valueFrom: ...
- name: AWS_SECRET_ACCESS_KEY
  valueFrom: ...
# ❌ 缺少 AWS_REGION
```

**问题：**
- `aws s3 cp` 命令需要 `AWS_REGION` 环境变量
- 未配置时 AWS CLI 会尝试从 metadata 获取，在跨 region 场景下会报错

**修复：**
```yaml
- name: AWS_REGION
  value: "us-east-1"  # 或通过 configmap 注入
```

### P2-4. 成本估算节点单价偏低

> **位置：** §10.1 第965行

```markdown
| Kubernetes 节点 | C4/8G | 4 | $120 | 高性价比 PostgreSQL 专用节点 |
```

**问题：**
- $120/月 对应 4C8G 节点，这在中国云厂商（阿里云/腾讯云）勉强够用
- 但在 AWS/Azure/GCP 上，类似规格（如 `t3.xlarge` 4C16G）的价格在 **$150-200/月**
- 且 PostgreSQL 专用节点建议至少 8C16G，成本会更高

**建议：** 更新为 "$150-200/月" 或注明 "国内云厂商价格"。
