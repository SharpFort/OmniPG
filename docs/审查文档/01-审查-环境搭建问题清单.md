## 二、🔴 阻塞级问题（不解决无法开发）

### B1. Docker Compose 方案与 Pigsty 存在根本性矛盾

> Pigsty 是一个 Ansible-based 部署系统（官网："Linux-native, no K8s or Docker required"），通过 `curl https://pigsty.io/get | bash` + `./deploy.yml` 在裸机/VM上部署。Pigsty 不是 Docker 镜像，也没法用 Docker Compose "运行 Pigsty"。

文档 01 的 Docker Compose 用 `pgcharles/pgtap:16` 镜像作为 PG——这**完全不是 Pigsty**。

- [B1-1] 整个方案的目标部署环境是什么？是**生产环境用 Pigsty（Ansible+裸机），开发环境用 Docker Compose 模拟**？还是**从头到尾都不用 Pigsty，只用 Docker Compose**？
- [B1-2] 如果是"生产 Pigsty + 开发 Docker"，这两个环境的 PG 版本、扩展集、配置参数完全不同，如何保证开发环境的代码能在生产跑？文档中没有任何兼容性保证。
- [B1-3] 如果是"全程 Docker Compose 不用 Pigsty"，那 Pigsty 在方案中只是一个"扩展商店"的角色（用它的 RPM 仓库装扩展），而不是运行 PG 的平台——这与 00 文档中"Pigsty PostgreSQL 集群 (Patroni ×3)"的描述完全不一致。

### B2. plpython3u — JWT 签名的核心依赖可能不存在

> 00 文档和 01 文档都依赖 `plpython3u` + PyJWT 实现 RS256 JWT 签名。但 Pigsty v4.3.0 的 531 个扩展列表中**未见 plpython3u**。

- [B2-1] Pigsty v4.3.0 是否实际包含 plpython3u？如果包含，它在哪个 RPM 包中？`postgresql16-plpython3` 是否在 Pigsty 的 yum/apt 仓库中？
- [B2-2] 如果 plpython3u 不可用，JWT 签名的替代方案是什么？可选方向：
  - plv8（Pigsty 支持，RPM only） + JavaScript JWT 库？
  - pgjwt + pgcrypto（纯 SQL JWT，但只支持 HMAC 不支持 RS256）？
  - pgrx (Rust) 编写自定义扩展做 JWT 签名？
  - 放弃"数据库内 JWT 签名"，改为 Go/Python sidecar 签名？
- [B2-3] Docker Compose 中的 `pgcharles/pgtap:16` 镜像是否包含 plpython3u？文档验收项 E15 测试 `DO $$ import json; plpy.info(...) $$ LANGUAGE plpython3u`——这个测试是否实际在目标镜像上跑通过？

### B3. pg_smtp_client — 不存在的扩展

> GitHub 全站搜索 `pg_smtp_client` 仅返回 1 个仓库。这不是一个已知的 PG 扩展，在 PGXN 上也搜不到。

- [B3-1] `pg_smtp_client` 的准确来源是什么？是某个特定发行版（如 Pigsty）自带的？还是指其他邮件扩展的别名？
- [B3-2] 如果这个扩展不存在，邮件发送功能的实现方案是什么？pg_net 可以发 HTTP 请求（调邮件 API），但这和 SMTP 直发是两回事。

## 三、重大技术问题

### T1. etcd 双集群问题

> Pigsty v4.3.0 使用 etcd（3.6.10）做 Patroni leader election。APISIX 也使用 etcd 做路由/策略存储。Docker Compose 中放了一个独立的 bitnami/etcd:3.5。

- [T1-1] 生产环境中 Pigsty 的 etcd 和 APISIX 的 etcd 是同一套还是两套？如果是同一套，Pigsty v4.3.0 是否允许在它的 etcd 中存储 APISIX 的 key（/apisix/ 前缀）？Patroni 使用 /service/ 前缀，是否有冲突？
- [T1-2] 如果是两套 et c d，那么在 Docker Compose 开发环境中是否也应该模拟两套？当前 Compose 只有一个 etcd 节点。
- [T1-3] Docker Compose 中 etcd 是单节点（无集群），而生产环境的 Pigsty 默认部署 3 节点 etcd 集群。开发/生产环境差异如何管理？

### T2. PostgREST 认证角色链路不完整

> 文档创建了 `web_anon` 和 `authenticated` 两个 NOLOGIN 角色，但缺少关键的 `authenticator` 角色。

- [T2-1] PostgREST 文档明确要求一个带 LOGIN 权限的 `authenticator` 角色（用于连接 PG），然后通过 JWT 中的 `role` claim 切换到 `web_anon` 或 `authenticated`。文档 01 的 `02-schemas.sql` 中没有创建 `authenticator` 角色——这个角色在哪里创建？
- [T2-2] Docker Compose 中 `PGRST_DB_URI: postgres://app_owner:***@postgres:5432/app_db` — 用 `app_owner` 直连？这绕过了 PostgREST 的标准认证链（authenticator → JWT → switch role）。这是开发环境的简化，还是方案设计上的偏差？

### T3. 缺少 Pgbouncer 连接池

> Pigsty v4.3.0 内置 Pgbouncer。生产环境中 PostgREST → Pgbouncer → PostgreSQL 是标准部署模式。PostgREST 本身是无状态的且可能部署多个实例，直接连 PG 会造成连接数膨胀。

- [T3-1] Docker Compose 中没有 Pgbouncer 容器。开发环境不配 Pgbouncer 会导致：
  - 连接池行为与生产环境不同
  - 事务模式和会话模式在 Pgbouncer 下的限制（如 LISTEN/NOTIFY 需要会话模式）无法在开发环境验证
- [T3-2] Docker Compose 是否应该加入 Pgbouncer 服务？

### T4. SYNcer 初始化过早

> 文档第 6 节在环境搭建阶段就 `go mod init syncer; go get github.com/lib/pq`——但 Syncer 的完整逻辑要到阶段 4 才开发。

- [T4-1] 初始化模块后没有代码也没有编译验证（`go build`），这个步骤的实际价值是什么？
- [T4-2] `lib/pq` 已进入维护模式，Go 社区标准是 `pgx v5`。pgx v5 原生支持 LISTEN/NOTIFY（通过 `pgxpool` + `Listen`），性能和功能都优于 lib/pq。是否应该使用 pgx？

### T5. Grafana/Prometheus 监控栈完全缺失

> 00 文档提到 Prometheus、Grafana、pgaudit 审计日志链路。Pigsty v4.3.0 内置完整监控栈（Grafana 13.0.1 + Prometheus/VictoriaMetrics + 40+ Dashboard）。

- [T5-1] Docker Compose 中只有 Loki，没有 Prometheus、Grafana、node_exporter、pg_exporter。是计划在后续阶段加入，还是开发环境不需要监控？
- [T5-2] 如果开发环境不需要监控，那么阶段 4/5 的端到端验收如何验证监控指标？`pgaudit → Vector → Loki` 的链路在 Docker Compose 中是否完整（Vector 也未在 Compose 中出现）？

## 四、技术细节问题

### D1. Docker 镜像选择

- [D1-1] `pgcharles/pgtap:16` — 这是社区维护的 PG16 + pgTAP 镜像。Docker Hub 上最后更新时间？是否仍在维护？是否有安全漏洞未修补？
- [D1-2] 是否考虑使用官方 `postgres:16` 镜像并在 init 脚本中手动安装 pgTAP 和其他扩展？这样可以更好地控制扩展版本。
- [D1-3] APISIX 镜像：`apache/apisix:3.10.0-debian`（2024-08）。是否有已知 CVE？3.10.0 的 authz-casbin 插件是否有 bug 在后续版本修复？
- [D1-4] `swaggerapi/swagger-ui:latest` — 使用 `latest` tag 意味着每次 `docker compose pull` 可能拉取不同版本，导致不可复现的环境。建议锁定版本。

### D2. shared_preload_libraries 配置

> Docker Compose 中 PG 的命令行参数：
> `-c shared_preload_libraries='pgaudit,pg_cron,pgsodium'`

- [D2-1] `pg_net` 是否需要 `shared_preload_libraries`？如果 `pg_net` 使用了 background worker，不预加载会导致功能不完整。
- [D2-2] `pg_timetable` 是一个独立进程（不是 PG 扩展），不需要在 shared_preload_libraries 中。但 Docker Compose 中没有 pg_timetable 服务——计划中的定时任务（如 Token 清理）由什么执行？pg_cron 替代？
- [D2-3] Pigsty 的生产配置中 shared_preload_libraries 列表通常更长（含 pg_stat_statements, auto_explain 等）。两者的差异在文档中是否需要说明？

### D3. APISIX 配置文件

- [D3-1] `config.yaml` 中 `allow_admin: - 0.0.0.0/0` — 这允许任意 IP 访问 Admin API。开发环境可以接受，但安全风险是否应在文档中标注警告？
- [D3-2] admin_key 明文硬编码在 config.yaml 和 .env 中。是否有密钥轮换指引？
- [D3-3] `apisix.yaml` 占位文件内容为空路由和空 upstream。但 APISIX standalone 模式下这个文件应该是完整的路由定义。当前配置使用 etcd 模式（`config_provider: etcd`），所以 apisix.yaml 不会被读取——那么创建这个文件的意义是什么？是否应该注明"此文件仅在 standalone 模式下使用"？

### D4. 端口规划

- [D4-1] 端口规划表中列出了 Prometheus(9090) 和 Grafana(3001)，但 Docker Compose 中没有这两个服务。是否应标注为"预留"或"后续加入"？
- [D4-2] PG 端口映射到宿主机 5432，如果开发者本地已经有 PG 运行，会端口冲突。文档第 9 节有提到——但冲突检查应该放在启动前作为验收项。

### D5. Dbmate 安装

- [D5-1] Windows 安装路径 `$env:LOCALAPPDATA\Microsoft\WindowsApps\` — 这是 Microsoft Store 应用的特殊目录，普通用户可能没有写入权限。建议改为 `$env:USERPROFILE\.local\bin\` 或 `$env:LOCALAPPDATA\Programs\`。
- [D5-2] `.env` 中添加的 `DATABASE_URL` 使用 `***` 替代密码——Dbmate 的 URL 中密码是否需要明文？

### D6. 验收清单

- [D6-1] E7 验收项期望安装的扩展列表包含 `pg_smtp_client`，但 init 脚本中并没有创建这个扩展。是遗漏还是故意省略？
- [D6-2] E5 "全部 healthy/running" — Loki 容器没有定义 healthcheck，`docker compose ps` 可能只显示 "running" 而非 "healthy"。验收标准需要区分。
- [D6-3] E12 验收命令 `| head -c 50` — 这是 Unix 命令，PowerShell 中不可用。需要适配 Windows 环境（如 `Select-Object -First 1` 或省略管道部分）。
