## 1. 项目整体架构组织哲学

这套项目结构的底层逻辑是：**“配置即代码，数据库即代码，一切皆可版本化”**。

1.  **APISIX 层（声明式配置）：** 存放网关路由和 Lua Serverless 脚本。
2.  **PostgREST 层（声明式配置）：** 存放 API 暴露策略与密钥映射。
3.  **Database 层（版本化迁移）：** 存放数据表、视图定义、PL/pgSQL 和 PL/Python 业务函数。
4.  **Syncer 层（Go 独立模块）：** 存放 Policy Syncer 的 Go 源码和单元测试。
5.  **Test 体系（双层测试）：**

    *   使用 **Go standard test** 测试 Syncer 的事件循环、防抖逻辑。
    *   使用 **E2E 脚本** 模拟网关到数据库的全链路接口集成测试。

## 3. 核心模块与代码组织实战

为了让这套项目结构顺畅跑通，我们通过一个精心设计的 **`Makefile`** 将各个零散的技术组件（Docker、Dbmate、pgTAP、Go、APISIX）粘合在一起。

### 3.1 统一工作流的 `Makefile`

在根目录下创建 `Makefile`，开发人员只需执行几条极简命令，即可完成本地开发生命周期：

```makefile
# Makefile
.PHONY: dev-env db-migrate db-rollback apply-src test-db test-syncer test-e2e test-all run-syncer

# 1. 启动本地开发环境 (PG, APISIX, PostgREST)
dev-env:
	docker compose up -d

# 2. 执行数据库向前迁移 (Dbmate)
db-migrate:
	dbmate up

# 3. 回滚最近一次数据库迁移
db-rollback:
	dbmate rollback --single

# 4. 应用 db/src 中的幂等源码（函数/视图/触发器）
apply-src:
	@for f in $(shell find db/src -name "*.sql" | sort); do \
		echo "Applying $$f..."; \
		docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
	done

# 5. 运行数据库内部单元测试 (pgTAP)
test-db:
	@echo "Running PostgreSQL PL/pgSQL unit tests via pgTAP..."
	docker compose exec app-postgres pg_prove -U app_owner -d app_db --ext .sql db/tests/

# 6. 运行 Go 同步器的单元测试
test-syncer:
	@echo "Running Policy Syncer unit tests..."
	cd syncer && go test -v ./...

# 7. 运行全链路端到端集成测试
test-e2e:
	@echo "Running end-to-end API integration tests..."
	bash ./scripts/e2e-test.sh

# 8. 一键执行全栈测试 (用于 CI/CD 流水线)
test-all: test-db test-syncer test-e2e

# 9. 启动 Policy Syncer 同步器
run-syncer:
	cd syncer && go run main.go
```

### 3.2 数据库迁移层（`db/migrations/`）的组织艺术

在传统项目开发中，数据库脚本往往散落在多个文件夹中。在我们的架构下，**数据库就是整个项目的核心引擎**。

使用 Dbmate 能够将数据库结构变动、存储过程部署、触发器编写紧密集成：

#### 示例：`db/migrations/20260706000000_blacklist_triggers.sql`
```sql
-- migrate:up
-- 部署“角色变动拉黑旧 AT”的触发器逻辑

CREATE OR REPLACE FUNCTION blacklist_at_on_role_change()
RETURNS TRIGGER AS $$
DECLARE
    v_user_id uuid;
    v_session RECORD;
BEGIN
    IF (TG_OP = 'INSERT' OR TG_OP = 'UPDATE') THEN
        v_user_id := NEW.user_id;
    ELSE
        v_user_id := OLD.user_id;
    END IF;

    FOR v_session IN 
        SELECT active_jti, expired_at 
        FROM sys_user_session 
        WHERE user_id = v_user_id AND is_used = FALSE AND active_jti IS NOT NULL
    LOOP
        INSERT INTO sys_token_blacklist (jti, expired_at)
        VALUES (v_session.active_jti, v_session.expired_at)
        ON CONFLICT (jti) DO NOTHING;
    END LOOP;

    RETURN NULL;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER trg_blacklist_on_role_change
AFTER INSERT OR UPDATE OR DELETE ON sys_user_role
FOR EACH ROW EXECUTE FUNCTION blacklist_at_on_role_change();

-- migrate:down
-- 回滚时安全清理触发器和函数

DROP TRIGGER IF EXISTS trg_blacklist_on_role_change ON sys_user_role;
DROP FUNCTION IF EXISTS blacklist_at_on_role_change();
```

## 4. 这套项目结构的开发与运维闭环优点

1.  **极度高聚：**

## 一、 核心架构策略：PostgreSQL Schema 级物理隔离

在传统的 Go/Java 开发中，我们用“包（Package）”或“微服务”来隔离业务。在无后端（Database-Driven）架构中，我们**直接使用 PostgreSQL 的 Schema（模式/命名空间）作为业务模块的物理边界** [1.1.9]。

### 1. 业务划分设计
不要把所有表都建在 `public` schema 里。应该按业务域创建独立的 Schema：
*   **`public`（默认共享域）：** 存放我们之前建立的 11 张 `sys_*` RBAC 表、会话表、黑名单表（与 07-Database-Migrations.md 对齐）。所有业务域的表都可以通过 `public` schema 访问。
*   **`sales`（销售业务域）：** 存放 `orders`（订单表）、`order_items`（订单明细表）等。
*   **`inventory`（库存业务域）：** 存放 `stock`（库存表）、`warehouse`（仓库表）等。
*   **`api_v1` / `api_v2`（API 暴露域）：** **这是 PostgREST 唯一暴露给网关的 Schema** [1.1.9]。它里面不放任何物理表，**只放面向前端的"只读视图"和"业务 RPC 存储过程"** [1.1.9]。它作为一个"胶水层"，安全地调用底层 `public`、`sales`、`inventory` 中的数据。

这样设计后，底层的物理表被完美隐藏，前端只能通过 `api_v1` 的视图和函数进行受控操作，安全性达到了极高标准 [1.1.9]。

> **与 07-Database-Migrations.md 对齐说明：** 07 文档将所有 `sys_*` 表创建在 `public` schema，PostgREST 配置为 `db-schemas = "api_v1"` + `db-extra-search-path = "public"`。本文档的 Schema 设计与此完全一致，确保 `api_v1` 视图能直接 JOIN `public.sys_*` 表。

## 二、 物理文件组织策略：多表与存储过程如何拆分？

当表数量变多时，文件的组织应该遵循 **“DDL 增量化、PL/SQL 幂等化（源码化）”** 的核心原则。

### 1. DDL（建表/改表）文件组织原则
对于物理表和外键（DDL），由于其状态是不可逆的，**必须使用 Dbmate 这种带有时间戳的增量迁移文件来管理** [2.1.2]。
*   **规则：** **"按业务 Feature/模块拆分，关联表放一起"**。
    *   *不要*把 100 张表塞在一个 SQL 文件里。
    *   *不要*把 1 张表拆成 1 个文件（如果它们是强关联的，如主子表）。
    *   **正确做法：** 订单表（`orders`）和订单明细表（`order_items`）由于存在外键和强业务绑定，应当放在**同一个**迁移文件 `20260710000000_create_sales_orders.sql` 中。而商品表、RBAC 表则属于完全不同的业务域，必须放在独立的迁移文件中。

### ⚠️ 时间戳规范与 07-Database-Migrations.md 对齐

本文档示例中使用的迁移时间戳（如 `20260701000000`）**仅作目录结构示意**。实际项目应统一使用 07 文档定义的 `202607070000xx` 系列：

```
├── migrations/
│   ├── 20260707000001_init_tables.sql                        # 11 张 sys_* 表建表 DDL
│   ├── 20260707000002_create_relation_sessions_blacklist.sql # 会话与黑名单关联
│   ├── 20260707000003_create_casbin_view.sql                 # casbin_rule 视图
│   ├── 20260707000004_create_notify_triggers.sql             # pg_notify 触发器
│   ├── 20260707000005_create_auth_functions.sql              # 认证函数（SSO/RTR）
│   ├── 20260707000006_create_permission_functions.sql        # 权限函数
│   ├── 20260707000007_create_security_triggers.sql           # 安全触发器
│   ├── 20260707000008_enable_rls_policies.sql                # RLS 策略
│   └── 20260707000009_seed_data.sql                          # 种子数据
```

> **核心原则：** 迁移文件一旦创建，**原名永不修改**。如需回滚，使用 `dbmate rollback --single` 或新增修正迁移文件。

### 2. PL/SQL（函数/触发器/视图）文件组织原则
如果把所有的 `CREATE FUNCTION`、`CREATE VIEW` 像物理表一样写在增量迁移文件里，在 Git 协作时会发生灾难：
*   如果一个复杂的订单计算函数修改了 20 次，你会产生 20 个迁移文件，在 Git 中你**根本无法直观地看到这个函数的当前最新源码是什么样子的**。
*   **正确做法：“Idempotent Source Code（幂等源码化管理）”**
    我们在 Git 仓库中开辟一个 `/db/src/` 目录，按照 `Schema ──> 函数/视图/触发器` 的层级，**为每一个函数、视图、触发器建立一个独立的 `.sql` 文件**。这些文件全部使用 `CREATE OR REPLACE` 声明。
    *   **开发时：** 开发者在 `/db/src/` 对应的独立文件中直接修改业务函数逻辑。
    *   **部署时：** 迁移脚本或 CI/CD 只需利用一个简单的 Shell 脚本，扫描 `/db/src/` 并将最新的函数代码一键“重载（Apply）”到数据库中。

### ⚠️ DDL 与 src 的边界划分（解决 casbin_rule 双源管理冲突）

为避免 `casbin_rule` 等视图出现"DDL 迁移 + 幂等源码"双重管理导致的同步问题，明确划分规则：

| 类型 | 存放位置 | 管理方式 | 修改方式 |
|:---|:---|:---|:---|
| **物理表 DDL** | `db/migrations/` 时间戳文件 | Dbmate 增量迁移，**只增不改** | 新建迁移文件 |
| **索引** | `db/migrations/` 时间戳文件 | Dbmate 增量迁移，**只增不改** | 新建迁移文件 |
| **视图** | `db/src/<schema>/views/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **函数** | `db/src/<schema>/functions/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **触发器** | `db/src/<schema>/triggers/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **种子数据** | `db/migrations/` 时间戳文件 | Dbmate 增量迁移 | 新建迁移文件 |

> **关键规则：** 同一个对象（如 `casbin_rule` 视图）**只在一个地方定义**。如果已在 `db/migrations/` 中创建了首次定义，后续所有修改必须通过 `db/src/` 幂等源码覆盖，不要在迁移文件中重复定义。

## 三、 扩展后的多模块项目目录结构

以下是一个整合了 `auth`、`sales`（订单项目）和 `inventory`（库存项目）三大域的**完整大中型项目骨架**：

```text
my-enterprise-platform/
├── Makefile                        # 统一构建、迁移、测试指令
├── docker-compose.yml              # 统一编排
│
├── apisix/                         # APISIX 路由配置
│
├── postgrest/                      # PostgREST 配置
│
├── db/                             # 【核心：数据库即代码】
│   ├── schema.sql                  # 全局物理表结构快照 (Dbmate 自动生成)
│   │
│   ├── migrations/                 # 1. 物理表 DDL 增量迁移历史 (只增不改)
│   │   ├── 20260707000001_init_tables.sql                        # 11 张 sys_* 表建表 DDL
│   │   ├── 20260707000002_create_relation_sessions_blacklist.sql # 会话与黑名单关联
│   │   ├── 20260707000003_create_casbin_view.sql                 # casbin_rule 视图（首次定义）
│   │   ├── 20260707000004_create_notify_triggers.sql             # pg_notify 触发器
│   │   ├── 20260707000005_create_auth_functions.sql              # 认证函数（SSO/RTR）
```

## 四、 这种组织方式下的多人协作与部署流

### 1. 物理表变更（Schema 变更）
*   **开发 A** 需要在订单模块增加一个“优惠券”功能。
*   他只需执行 `dbmate new add_coupons_to_sales` [2.1.4]，并在生成的迁移文件中写入 DDL：
    `CREATE TABLE sales.coupons (...); ALTER TABLE sales.orders ADD COLUMN coupon_id ...;` [2.1.2]
*   运行 `dbmate up` 在本地生效 [2.1.2]。

### 2. 业务存储过程开发
*   **开发 A** 随后需要编写核销优惠券的计算存储过程。
*   他直接在 `/db/src/sales/functions/` 下新建一个文件 `apply_coupon.sql`，在里面用标准的 PL/pgSQL 或 PL/Python 编写业务逻辑。
*   **部署时：** 项目提供一个极简的部署脚本（可配在 `Makefile` 中，如 `make apply-src`），通过一行 `psql -f` 指令自动递归扫描并将整个 `/db/src/` 下所有的 `.sql` 函数文件重新刷入数据库。

### 3. API 交付（无后端的核心优势）
*   当 **开发 A** 编写完底层的物理表、业务函数后，他只需要在 `/db/src/api_v1/views/`（或 `rpc/`）下创建一个胶水视图/RPC 函数。
*   **PostgREST** 检测到 `api_v1` 的变化，会**在一微秒内自动将这个新视图或函数暴露为前端可以直接调用的 HTTP 接口（如 `POST /rpc/checkout`）** [1.1.9]。整个过程**不需要写任何一行 Go/Java 控制器代码，也不需要停机**。

### 总结

这种 **“Schema 隔离 + DDL 增量迁移 + PL/pgSQL 独立源码文件 + api 胶水暴露层”** 的物理组织架构，是在不引入复杂后端框架的前提下，将项目平滑扩展至支撑数百张表、多个独立子项目的唯一工程级解法。它让无后端系统既拥有极高的开发生产力，又具备清晰、规范的工业级工程脉络。
