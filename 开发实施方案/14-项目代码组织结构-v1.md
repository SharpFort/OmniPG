# 14-项目代码组织结构-v1

> **版本：** v1.2  
> **日期：** 2026-07-22  
> **项目：** OmniPG 零后端代码统一权限管理平台  
> **架构：** Pigsty + PostgreSQL + PostgREST + APISIX + Casdoor + Go Syncer

---

## 1. 项目整体架构组织哲学

这套项目结构的底层逻辑是：**"配置即代码，数据库即代码，一切皆可版本化"**。

1. **Pigsty 层（基础设施基座）：** 提供 PostgreSQL 集群、监控、高可用等基础能力
2. **APISIX 层（声明式配置）：** 存放网关路由和 Lua Serverless 脚本
3. **PostgREST 层（声明式配置）：** 存放 API 暴露策略与密钥映射
4. **Database 层（版本化迁移）：** 存放数据表、视图定义、PL/pgSQL 业务函数
5. **Syncer 层（Go 独立模块）：** 存放 Policy Syncer 的 Go 源码和单元测试
6. **Test 体系（双层测试）：** pgTAP 测试数据库逻辑 + Go test 测试 Syncer + E2E 全链路测试

## 2. 核心架构策略：表前缀模块隔离

在无后端（Database-Driven）架构中，我们**使用表前缀作为业务模块的逻辑边界**，参考 Odoo、Nexterp 等大型 ERP/CRM 软件的设计实践。

### 2.1 业务模块前缀划分

| 前缀 | 模块 | 说明 | 示例表 |
|:---|:---|:---|:---|
| `sys_` | 权限域 | RBAC 核心表、会话、黑名单 | `sys_user`, `sys_role`, `sys_api` |
| `sales_` | 销售域 | 订单、账单、客户 | `sales_orders`, `sales_order_items` |
| `inventory_` | 库存域 | 库存、仓储、物流 | `inventory_stock`, `inventory_warehouse` |
| `hr_` | 人力资源 | 员工、部门、考勤 | `hr_employees`, `hr_departments` |
| `finance_` | 财务域 | 账户、流水、发票 | `finance_accounts`, `finance_transactions` |

### 2.2 跨模块数据关联

使用表前缀后，跨模块关联直接使用表名，无需 `schema.table` 前缀：

```sql
-- 跨模块查询（直接表名关联）
SELECT o.id, o.amount, u.username
FROM sales_orders o
JOIN sys_user u ON o.user_id = u.id;

-- 跨模块外键（直接引用）
ALTER TABLE sales_orders 
ADD CONSTRAINT fk_user 
FOREIGN KEY (user_id) REFERENCES sys_user(id);
```

### 2.3 租户隔离方案

| 隔离级别 | 实现方式 | 适用场景 |
|:---|:---|:---|
| **行级隔离（推荐）** | `tenant_id` + RLS 策略 | 多租户共享数据库，性能高 |
| **数据库级隔离** | 每个租户独立数据库 | 金融/医疗等高隔离要求 |

```sql
-- 行级隔离示例
ALTER TABLE sales_orders ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON sales_orders
    USING (tenant_id = current_setting('app.current_tenant')::uuid);
```

> **决策说明：** 不采用 Schema 作为租户隔离，因为模块边界已用表前缀，再用 Schema 会导致命名混乱。RLS 行级隔离是 Supabase、Odoo 等主流方案，性能最高。

### 2.4 表前缀 vs Schema 隔离对比

| 维度 | 表前缀隔离 | Schema 隔离 |
|:---|:---|:---|
| **命名冲突** | 通过前缀规范避免 | ✅ 完全隔离 |
| **权限控制** | 表级 GRANT | Schema 级 GRANT |
| **跨域查询** | 直接表名 | 需要 `schema.table` |
| **迁移复杂度** | 较低 | 较高 |
| **PostgREST 暴露** | 直接暴露 | 需配置 `db-extra-search-path` |
| **参考案例** | Odoo, Nexterp, ERPNext | 部分 SaaS 平台 |
| **适合规模** | 中大型（10+ 模块） | 超大型（50+ 模块） |

## 3. 物理文件组织策略

### 3.1 DDL（建表/改表）文件组织

**规则：** "按业务 Feature/模块拆分，关联表放一起"

- 不要把所有表塞在一个 SQL 文件里
- 不要每张表拆一个文件（强关联的主子表应放一起）
- 订单表和订单明细表应放在同一个迁移文件中

### 3.2 PL/SQL（函数/触发器/视图）文件组织

**正确做法：** "Idempotent Source Code（幂等源码化管理）"

在 `/db/src/` 目录下，按照对象类型（函数/视图/触发器/权限）统一存放，**文件名前缀区分模块**。

### 3.3 DDL 与 src 的边界划分

| 类型 | 存放位置 | 管理方式 | 修改方式 |
|:---|:---|:---|:---|
| **物理表 DDL** | `db/migrations/` 时间戳文件 | Dbmate 增量迁移，只增不改 | 新建迁移文件 |
| **索引** | `db/migrations/` 时间戳文件 | Dbmate 增量迁移，只增不改 | 新建迁移文件 |
| **自定义类型** | `db/src/types/` | `CREATE TYPE`，幂等源码 | 直接修改文件 |
| **视图** | `db/src/views/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **函数** | `db/src/functions/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **触发器** | `db/src/triggers/` | `CREATE OR REPLACE`，幂等源码 | 直接修改文件 |
| **权限 GRANT** | `db/src/privileges/` | 独立 SQL 文件 | 直接修改文件 |
| **种子数据** | `db/migrations/` 时间戳文件 | Dbmate 增量迁移 | 新建迁移文件 |

> **决策说明：** `db/src/` 下不按模块分目录，而是统一按类型存放。文件名自带模块前缀（如 `sys_user_login_sso.sql`、`sales_create_order.sql`），这样目录结构扁平，浏览和修改更直观。

## 5. 核心模块与代码组织实战

### 5.1 统一工作流的 Makefile

```makefile
# Makefile
.PHONY: dev-env db-migrate db-rollback apply-src test-db test-syncer test-e2e test-all run-syncer

# 1. 启动本地开发环境
dev-env:
	docker compose up -d

# 2. 执行数据库向前迁移
db-migrate:
	dbmate up

# 3. 回滚最近一次数据库迁移
db-rollback:
	dbmate rollback --single

# 4. 应用 db/src 中的幂等源码（函数/视图/触发器）
apply-src:
	@for f in $(shell find db/src -name "*.sql" | sort); do \
		echo "Applying $$f..."; \
		docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
	done

# 5. 运行数据库内部单元测试 (pgTAP)
test-db:
	@echo "Running PostgreSQL PL/pgSQL unit tests via pgTAP..."
	docker compose exec app-postgres pg_prove -U app_owner -d app_db --ext .sql db/tests/

# 6. 运行 Go 同步器的单元测试
test-syncer:
	@echo "Running Policy Syncer unit tests..."
	cd syncer && go test -v ./...

# 7. 运行全链路端到端集成测试
test-e2e:
	@echo "Running end-to-end API integration tests..."
	bash ./scripts/e2e-test.sh

# 8. 一键执行全栈测试 (用于 CI/CD 流水线)
test-all: test-db test-syncer test-e2e

# 9. 启动 Policy Syncer 同步器
run-syncer:
	cd syncer && go run cmd/syncer/main.go
```

### 5.2 数据库迁移层组织

迁移文件一旦创建，**原名永不修改**。如需回滚，使用 `dbmate rollback --single` 或新增修正迁移文件。

> **决策说明：** `migrations/` 不按模块分目录，按时间戳统一存放。Dbmate/Flyway 等迁移工具默认不支持子目录，时间戳已能体现执行顺序和模块归属。

### 5.3 全链路集成测试（E2E）

E2E（End-to-End）测试是模拟真实用户场景的全链路测试，覆盖从网关到数据库的完整请求流程。

#### 示例：`scripts/e2e-test.sh`

```bash
#!/usr/bin/env bash
set -eo pipefail

GATEWAY_URL="http://localhost:9080"

echo "=================================================="
echo "E2E Test: Starting Full-Stack Permission Workflow"
echo "=================================================="

# 1. 模拟用户登录
echo "Step 1: Simulating user login (Alice)..."
LOGIN_RESP=$(curl -s -X POST "${GATEWAY_URL}/rpc/user_login_sso" \
  -H "Content-Type: application/json" \
  -d '{"p_username": "alice", "p_password": "secure_password"}')

ACCESS_TOKEN=$(echo "${LOGIN_RESP}" | grep -o '"access_token":"[^"]*' | grep -o '[^"]*$')
echo "SUCCESS: Logged in. Access Token retrieved."

# 2. 访问受限接口 (正常放行)
echo "Step 2: Accessing restricted API with token..."
API_RESP=$(curl -s -o /dev/null -w "%{http_code}" -X GET "${GATEWAY_URL}/api/v1/sys_user?limit=1" \
  -H "Authorization: Bearer ${ACCESS_TOKEN}")

if [ "${API_RESP}" -eq 200 ]; then
  echo "SUCCESS: API accessed successfully (Status 200)."
else
  echo "FAILED: API access denied (Status ${API_RESP})."
  exit 1
fi

# 3. 模拟管理员撤销角色
echo "Step 3: Admin revokes Alice's role..."
# 删除操作执行后，触发器在 0.01 秒内自动拉黑当前的 ACCESS_TOKEN

# 4. 再次访问接口 (应该被阻断)
echo "Step 4: Accessing restricted API again (expecting block)..."
API_RESP_AFTER=$(curl -s -o /dev/null -w "%{http_code}" -X GET "${GATEWAY_URL}/api/v1/sys_user?limit=1" \
  -H "Authorization: Bearer ${ACCESS_TOKEN}")

if [ "${API_RESP_AFTER}" -eq 401 ]; then
  echo "SUCCESS: Token was successfully blacklisted instantly (Status 401)!"
else
  echo "FAILED: Token block failed (Status ${API_RESP_AFTER})."
  exit 1
fi

echo "=================================================="
echo "E2E Test: All tests passed successfully!"
echo "=================================================="
```

## 6. 开发与运维闭环优点

1. **极度高聚：** 开发人员在修改权限系统时，只需关注 `db/migrations` 和 `db/src` 下的 SQL 文件，所有业务逻辑通过 Git 管理，改动一目了然。

2. **安全合规：** 测试分为 `db/tests`（pgTAP）和 `tests/e2e`（全链路），任何 PL/pgSQL 触发器写错导致的权限泄露都无法通过 CI 流水线。

3. **开发体验顺畅：** 新开发者只需拉取 Git 仓库，安装 Docker，运行 `make dev-env && make db-migrate && make test-all`，即可在本地构建完整环境。

4. **多环境管理：** 通过 `.env.development`、`.env.staging`、`.env.production` 管理不同环境配置。

5. **生产级部署：** `deployments/` 目录支持 Docker 和 K8s 部署，与本地开发环境解耦。

## 7. 多人协作与部署流

### 7.1 物理表变更（Schema 变更）

- 开发 A 执行 `dbmate new add_coupons_to_sales`，在生成的迁移文件中写入 DDL
- 运行 `dbmate up` 在本地生效
- 提交 PR，CI 自动运行测试
- 合并后 CD 自动部署

### 7.2 业务存储过程开发

- 开发 A 在 `/db/src/functions/` 下新建 `sales_apply_coupon.sql`
- 部署时运行 `make apply-src`，自动将整个 `/db/src/` 下的函数文件刷入数据库

### 7.3 API 交付

- 开发 A 在 `/db/src/views/` 下创建视图，在 `/db/src/functions/` 下创建 RPC 函数
- PostgREST 自动将新视图/函数暴露为 HTTP 接口
- **不需要写任何一行 Go/Java 控制器代码，也不需要停机**

## 8. 关键设计决策汇总

| 决策点 | 选择 | 原因 |
|:---|:---|:---|
| **模块隔离** | 表前缀 | Odoo/Nexterp 实践，简单直观，跨域查询方便 |
| **租户隔离** | RLS 行级 | 与表前缀兼容，性能最高，Supabase/Odoo 采用 |
| **src 目录结构** | 按类型统一存放 | 目录扁平，文件名前缀区分模块，浏览直观 |
| **migrations 结构** | 时间戳统一存放 | Dbmate 工具兼容，执行顺序清晰 |
| **单函数单文件** | 是 | 幂等部署粒度清晰，Git 冲突影响最小 |
| **api_v1 组织** | 视图/函数独立文件 | PostgREST 自动暴露，URL 即文件名 |
| **Pigsty 扩展** | 只记录版本 | 基础设施层由 Pigsty 管理，应用不重复管理 |
| **Monitoring** | 不纳入代码组织 | Pigsty 自带 Prometheus + Grafana |

## 9. Pigsty 基础设施层

Pigsty 作为整个项目的地基组件，提供以下基础能力：

| 组件 | 用途 | 说明 |
|:---|:---|:---|
| **PostgreSQL** | 数据库 | Pigsty 提供高可用 PG 集群（主从、自动故障转移） |
| **Patroni** | 高可用 | PG 集群自动选主和故障转移 |
| **PgBouncer** | 连接池 | 数据库连接池，提升并发性能 |
| **Redis** | 缓存/限流 | APISIX 限流、Session 存储 |
| **Prometheus** | 监控 | 采集 PG、APISIX、Syncer 指标 |
| **Grafana** | 可视化 | 监控大盘展示 |
| **Alertmanager** | 告警 | 异常告警通知 |

> **说明：** Pigsty 的安装和配置不属于本代码仓库管理范围。本仓库仅依赖 Pigsty 提供的 PostgreSQL、Redis 等基础设施服务。
