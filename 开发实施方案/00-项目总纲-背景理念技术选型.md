## 1. 项目背景

### 1.1 要解决的问题

传统企业级后台管理系统需要为**每种编程语言**（Go/Java/Python/Node.js）**重复编写**权限管理模块（用户、角色、菜单、API 鉴权、数据隔离等）。这导致：

- 多语言微服务架构下，权限逻辑分散、不一致
- 每个新项目都要重写 RBAC 的 CRUD 接口和鉴权中间件
- 权限变更需要重启/重新部署各服务

### 1.2 项目目标

构建一套**零后端代码（Database-Driven）的统一权限管理与 API 自动生成系统**，实现：

- 用数据库直接生成完整的 RESTful API（PostgREST）
- 用 Casbin 权限引擎在前端网关层统一拦截鉴权
- 传统 RBAC 管理界面自动映射到高性能的 Casbin 运行策略
- 所有业务逻辑（登录、Token 签发、多租户隔离、审计）收敛在 PostgreSQL 数据库内部

### 1.3 适用范围

- 企业内部管理后台（Admin UI）
- 多语言微服务架构的统一权限层
- SaaS 多租户系统的底座
- 需要快速搭建 API + 权限体系的新项目

## 2. 核心设计理念

### 2.1 业务模型与运行模型分离

| 层级 | 模型 | 存储 | 用途 |
|:---|:---|:---|:---|
| **业务模型** | 传统的 `User - Role - Menu - API` 关系表 | `sys_*` 物理表 | 供管理员理解和编辑（人类友好） |
| **运行模型** | Casbin 的扁平策略行 | `casbin_rule` 视图/物化视图 | 供网关极速决策（机器高效） |

数据库视图自动将业务模型"编译"为运行模型，消除了手动同步代码。

### 2.2 读写分离

```
【写路径】 Admin UI → PostgREST → PostgreSQL 传统 RBAC 表
              └→ 视图自动映射 ─→ casbin_rule
                    └→ pg_notify ─→ Policy Syncer ─→ etcd ─→ APISIX 刷新

【读路径】 Client → APISIX (Casbin 内存鉴权) → 通过 → PostgREST → PG (RLS 行级过滤)
                  → 拒绝 → 403
```

- **写路径：** PostgREST 只操作传统 RBAC 表，绝不直接暴露 casbin_rule
- **读路径：** APISIX 从 etcd/内存中加载 Casbin 策略，毫秒级决策，不访问数据库

### 2.3 Role-in-JWT 优化

用户登录时，PL/pgSQL 函数一次性查库获取该用户的所有角色，将角色数组 `["role_admin","role_editor"]` 直接写入 JWT 的 Payload。

- **效果：** APISIX 不再需要加载百万级 `g` 规则（用户↔角色映射），策略从 1050 万行降到 50 万行
- **内存：** APISIX 单 Worker 从 2.6 GB 降到 125 MB
- **匹配器：** `regexMatch(r.sub, "(^|,)" + p.sub + "($|,)")` 替代了 `g(r.sub, p.sub)`

### 2.4 深度防御（Defense in Depth）

```
请求 → APISIX Casbin（API 级：能否调这个接口？）
     → PostgreSQL RLS（数据级：能否看这行数据？）
```

两层独立防线，即使网关被绕过，数据库 RLS 依然生效。

### 2.5 数据库即代码（Database as Code）

所有 DDL、PL/pgSQL、视图、触发器代码全部纳入 Git + Dbmate migration 管理，支持版本追溯、回滚和 CI/CD。

### 5.2 核心数据流

```text
管理员分配角色: INSERT sys_user_role
  │
  ├──→ casbin_rule 视图自动反映（g 规则不再需要，由 JWT 承载）
  │
  ├──→ 触发器 blacklist_at_on_role_change()
  │     └→ INSERT sys_token_blacklist (用户旧 Token 立即失效)
  │
  └──→ 用户下次请求:
        ├→ PostgREST db-pre-request 检测到 jti 在黑名单
        ├→ 返回 401
        └→ 前端 Axios 拦截器静默刷新 Token
              └→ refresh_token_rtr() 生成含新角色的 JWT
```

## 6. 安全模型总览

### 6.1 多层防线

| 层级 | 组件 | 作用 |
|:---|:---|:---|
| 传输层 | APISIX SSL 终止 + mTLS | 防窃听/DNS 劫持 |
| 认证层 | JWT RS256 + httpOnly Cookie | 身份凭证 |
| API 鉴权层 | APISIX authz-casbin | 能调哪个接口 |
| 数据鉴权层 | PostgreSQL RLS | 能看哪行数据 |
| 审计层 | pgaudit → Vector → Loki | 不可篡改的审计日志 |

### 6.2 JWT 链路（v4 更新：接入 Casdoor）

```
┌─────────────────────────────────────────────────────────────────┐
│                    认证流程（v4 更新）                            │
│                                                                  │
│  用户 → APISIX (9080)                                            │
│         │                                                        │
│         ├─ 未登录 → authz-casdoor 插件 → 重定向到 Casdoor 登录页  │
│         │          (支持 OAuth/OIDC/SAML/LDAP/WebAuthn/MFA)      │
│         │                                    │                   │
│         │                          ┌─────────┴──────────┐       │
│         │                          │ GitHub / Google /   │       │
│         │                          │ WeChat / LDAP / ... │       │
│         │                          └─────────┬──────────┘       │
│         │                                    │ 认证成功           │
│         │                                    ▼                   │
│         │                              Casdoor 签发 JWT          │
│         │                              (含 roles 数组, RS256)    │
│         │                                    │                   │
│         └─ 已登录 ← JWT ←────────────────────┘                  │
│              │                                                   │
│              ├─ jwt-auth 插件 (RS256 验签，Casdoor JWKS 公钥)    │
│              ├─ authz-casbin 插件 (API 鉴权)                     │
│              └─ → PostgREST → PG (RLS)                          │
│                                                                  │
│  本地账号密码登录（回退方案）:                                     │
│  PostgREST /rpc/user_login_sso → PL/pgSQL 验证密码 →签发 JWT    │
│  (仅作为 Casdoor 的补充，用于无外部 IdP 的场景)                   │
└─────────────────────────────────────────────────────────────────┘

密钥管理:
  Casdoor 私钥 → Casdoor 内部管理（不暴露给 PG）
  Casdoor JWKS → APISIX jwt-auth 插件自动获取公钥（轮询 JWKS 端点）
  sys_secret 表 → 仅存储应用级密钥（API Key 等），不存储 JWT 私钥

Token 策略:
  Access Token (15分钟) → APISIX jwt-auth 公钥验签
  Refresh Token (7天)  → httpOnly Cookie → PostgREST refresh_token_rtr
```

> **变更说明：**
> -JWT 签发从 `plpython3u + PyJWT` 改为 **Casdoor 原生 RS256 签发**
> - Casdoor 自动管理密钥轮换，APISIX 通过 JWKS 端点自动获取最新公钥
> - `generate_rs256_jwt()` 函数不再需要，由 Casdoor 替代
> - `sys_secret` 表中不再存储 JWT 私钥，仅存储其他应用级密钥

### 6.3 RLS 行级安全

每张业务表绑定 `AS RESTRICTIVE` 策略，强制 `AND tenant_id = current_tenant_id()`，从物理层面杜绝跨租户数据泄露。

## 8. 开发环境与生产环境架构差异说明

> **本节为 v3 审查新增**，回答 00/01 审查中标记的架构鸿沟问题（A5、N-1、N-2）。

### 8.1 两套环境的定位

| 环境 | 目标 | PG 来源 | 部署方式 |
|:---|:---|:---|:---|
| **开发环境** | 验证业务逻辑（DB Schema、API、前端交互） | Pigsty PGSQL 模块（PostgreSQL 18 + pgBouncer） | Pigsty 统一管理 PGSQL/INFRA/REDIS/DOCKER 模块 + Docker Compose（仅无状态服务） |
| **生产环境** | 高可用、高性能、可运维 | Pigsty + PostgreSQL 18 + Patroni ×3 | Pigsty Ansible 部署 |

### 8.2 开发环境缺失的生产特性

以下特性仅在生产环境（Pigsty）中存在，开发环境（Docker Compose）**刻意省略**：

| 特性 | 开发环境 | 生产环境 | 影响 |
|:---|:---|:---|:---|
| Patroni HA | ❌ 单 PG 容器 | ✅ Patroni ×3 | 故障切换行为无法在开发环境验证 |
| etcd 集群 | ❌ 单节点 | ✅ 3 节点集群 | 集群容错行为差异 |
| Pgbouncer 连接池 | ❌ 无 | ✅ Pigsty 内置 | 连接池行为不可复现 |
| HAProxy + Keepalived | ❌ 无 | ✅ VIP 高可用 | 负载均衡逻辑未模拟 |
| Pigsty 监控栈 | ❌ 仅 Loki | ✅ Grafana + VictoriaMetrics + 40+ Dashboard | 监控指标不可用 |
| pgBackRest 备份 | ❌ 无 | ✅ Pigsty 内置 PITR | 备份恢复流程不可验证 |
| Vector 日志采集 | ❌ 无 | ✅ Pigsty 内置 | 审计日志链路不完整 |

### 8.3 兼容性保证策略

为保证开发环境的代码能在生产环境运行：

1. **DDL/函数/视图** — 使用标准 SQL，不依赖特定 PG 发行版
2. **扩展依赖** — 核心扩展（pg_net、pgcrypto、pgaudit、pg_cron）在两个环境中均可用（Pigsty 侧已验证，Docker 侧通过 `pgcharles/pgtap:18` + 额外安装）
3. **配置差异** — 使用环境变量/配置文件分离（如 `db-uri` 端口、`sslmode` 等）
4. **验收策略** — 开发环境验证功能逻辑，生产环境需**独立验收**部署行为

### 8.4 开发环境的扩展安装

开发环境通过 Pigsty PGSQL 模块部署 PostgreSQL 18，扩展在 `pigsty.yml` 中声明，安装时自动启用：

```yaml
pg_extensions:
  - pgcrypto
  - pgsodium
  - pg_net
  - pgaudit
  - pgtap
  - pg_graphql
  - pg_cron
```

> **⚠️ 关键原则：** Agent 执行 01-环境搭建 文档时，必须理解 Pigsty 统一管理 PGSQL/INFRA/REDIS/DOCKER 模块，开发环境不再需要手动管理独立容器。

### 8.5 Pigsty Docker 模块对多数组件的统一管理

> **v4 新增：** Pigsty v4.3.0 的 DOCKER 模块可以在 Pigsty 管理的节点上统一部署和监控 Docker 容器。

Pigsty v4.3.0 内置的 DOCKER 模块可以管理的组件：

| 服务 | Pigsty 统一管理前 | Pigsty 统一管理后 |
|:---|:---|:---|
| MinIO | Docker Compose 独立容器 | ✅ Pigsty 内置模板（含 CVE 修复分支） |
| Redis | Docker Compose 独立容器 | ✅ Pigsty 内置模板（含 Dashboard） |
| APISIX | Docker Compose | Pigsty Docker 模块（自定义容器） |
| PostgREST | Docker Compose | Pigsty Docker 模块 |
| Casdoor | — | 🆕 Pigsty Docker 模块部署 |
| Policy Syncer | Go binary | Pigsty Docker 模块 |
| Swagger UI | Docker Compose | Pigsty Docker 模块 |
| Grafana + VictoriaMetrics | 未声明 | ✅ Pigsty 内置（40+ Dashboard） |
| etcd | Docker Compose | ✅ Pigsty 内置（Patroni DCS） |
| Pgbouncer | 无 | ✅ Pigsty 内置（生产必需） |
| HAProxy + Keepalived | 无 | ✅ Pigsty 内置（VIP 高可用） |
| pgBackRest | 无 | ✅ Pigsty 内置（PITR 备份） |

> **收益：** 通过 Pigsty 统一管理，Docker Compose 从 8 个容器缩减为 3-4 个无状态容器。所有服务纳入 Pigsty 监控体系（Grafana + VictoriaMetrics + Vector）。

## 9. 运维与可观测性（v4 新增）

> **本节为 v4 审查新增**，补齐审查中标记的运维空白（H1、H2、H3）。

### 9.1 监控与告警

| 层级 | 工具 | 监控内容 | 来源 |
|:---|:---|:---|:---|
| 基础设施 | Grafana + VictoriaMetrics | 节点 CPU/内存/磁盘/网络 | Pigsty 内置（Node Exporter） |
| 数据库 | Grafana + VictoriaMetrics | PG 连接数/锁/复制延迟/WAL | Pigsty 内置（pg_exporter） |
| 连接池 | Grafana Dashboard | Pgbouncer 等待/活跃连接数 | Pigsty 内置 |
| 网关 | VictoriaMetrics | APISIX QPS/延迟/错误率（Prometheus Plugin） | APISIX 插件输出 |
| 同步器 | 自定义 metrics | 同步延迟/失败次数/队列深度 | Policy Syncer 暴露 |
| 容器 | cAdvisor + Grafana | CPU/内存/网络 per 容器 | Docker 监控 |
| 审计日志 | Vector → VictoriaLogs | SQL 审计日志（pgaudit） | Pigsty 内置 Vector |

### 9.2 备份与恢复

| 数据类型 | 备份方案 | RPO | 工具 |
|:---|:---|:---|:---|
| PostgreSQL | 每日全量 + WAL 归档 | 零丢失（同步复制） | pgBackRest (Pigsty 内置) |
| etcd | 每 6 小时快照 | ≤6h | etcdctl snapshot |
| APISIX 路由配置 | Git 版本化 | 即时 | Ansible + Git |
| MinIO 对象存储 | 异地复制 (bucket replication) | 异步 | MinIO 自带 |

### 9.3 健康检查

| 组件 | 端点 | 检查内容 |
|:---|:---|:---|
| APISIX | `/health` | 所有插件加载正常，etcd 连接 |
| PostgREST | `/` (200 = ready) | DB 连接 + JWT 配置 |
| Casdoor | `/api/health` | Go 应用内建健康检查 |
| Policy Syncer | `:8080/healthz` | PG LISTEN 状态 + 同步进度 |
| PostgreSQL | `pg_isready` | 接受连接 + 复制状态 |
| HAProxy | `/haproxy?stats` | 后端健康状态 |
| Pgbouncer | `SHOW POOLS` (Admin DB) | 连接池统计 |

### 9.4 灾难恢复优先级

```
RTO 目标: < 5 分钟（PG 自动切换），< 30 分钟（全系统恢复）
RPO 目标: 0（同步复制）或 ≤5 分钟（异步复制）

恢复优先级:
P0 → PostgreSQL（核心数据，pgBackRest PITR 恢复）
P1 → etcd（APISIX 路由配置）
P2 → APISIX + PostgREST（无状态，可快速重启）
P3 → Casdoor（认证服务）
P4 → Policy Syncer + MinIO
```

## 10. 数据模型补充说明（v4 新增）

> **本节为 v4 审查新增**，补齐审查中标记的数据模型细节空白（E1）。

### 10.1 软删除与唯一约束

所有业务表（`sys_user`、`sys_role`、`sys_api`、`sys_menu`）建议包含以下字段：

```sql
-- 通用字段模式
created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
deleted_at   TIMESTAMPTZ,        -- 软删除标记，NULL = 活跃
is_active    BOOLEAN NOT NULL DEFAULT TRUE;  -- 快速禁用/启用

-- 唯一约束（仅对活跃记录生效）
CREATE UNIQUE INDEX idx_user_username ON sys_user (username) WHERE deleted_at IS NULL;
CREATE UNIQUE INDEX idx_role_code ON sys_role (role_code) WHERE deleted_at IS NULL;
CREATE UNIQUE INDEX idx_api_path_method ON sys_api (path, method) WHERE deleted_at IS NULL;
```

> **注意：** `casbin_rule` 视图自动过滤 `deleted_at IS NULL` 的记录。删除操作必须用 UPDATE SET deleted_at = now() 而非物理 DELETE。

### 10.2 外键级联策略

| 父表 | 子表 | 删除策略 | 说明 |
|:---|:---|:---|:---|
| sys_user | sys_user_role | CASCADE | 用户删除自动移除角色关联 |
| sys_role | sys_user_role | CASCADE | 角色删除自动移除用户关联 |
| sys_role | sys_role_api | CASCADE | 角色删除自动移除权限关联 |
| sys_role | sys_role_menu | CASCADE | 角色删除自动移除菜单关联 |
| sys_api | sys_role_api | CASCADE | API 删除自动移除权限关联 |
| sys_menu | sys_role_menu | CASCADE | 菜单删除自动移除权限关联 |
| sys_department | sys_user | RESTRICT | 部门下有用户时禁止删除 |

### 10.3 审计历史表

除 pgaudit 的 SQL 审计日志外，建议增加业务级审计表：

```sql
CREATE TABLE sys_audit_log (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       INT NOT NULL,
    user_id         INT,
    action          TEXT NOT NULL,        -- INSERT/UPDATE/DELETE/LOGIN/LOGOUT
    entity_type     TEXT NOT NULL,        -- sys_user, sys_role, etc.
    entity_id       INT,
    old_values      JSONB,
    new_values      JSONB,
    ip_address      INET,
    user_agent      TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

> **说明：** pgaudit 只记录 SQL 语句，不记录业务语义（"谁把用户X从角色A移到角色B"）。`sys_audit_log` 通过触发器补充业务级变更历史。
