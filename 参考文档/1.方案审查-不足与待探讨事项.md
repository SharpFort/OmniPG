### 🔴 阻塞级（Blocking）：必须投产前解决

#### B1. APISIX 订阅 PostgreSQL NOTIFY 的具体实现方案 —— 最大盲区

**问题描述：**
方案中反复提到 `pg_notify('casbin_channel')` → APISIX 刷新缓存，但 **从未给出 APISIX 侧如何订阅 PostgreSQL LISTEN/NOTIFY 的具体实现方式**。

APISIX 是基于 OpenResty (Nginx + LuaJIT) 的，而 PostgreSQL 的 `LISTEN` 需要一个**长连接**。这在 Nginx 的事件驱动模型中是反模式的——worker 进程不应该持有阻塞的长连接。具体挑战：
- APISIX 的 Lua 层（`ngx.timer`、`resty.postgres` 等）如何实现 LISTEN？
- 是否需要额外部署一个 sidecar/sidekick 进程来桥接？
- 如果使用外部桥接服务，如何调用 APISIX Admin API 触发策略重载？
- 桥接服务本身的高可用如何保证？

**追问方向：**
> "APISIX 侧具体如何订阅 PostgreSQL 的 `pg_notify` 通知？是否有官方或社区验证过的方案？是否需要额外部署一个 sidecar 进程（如用 Go/Python 写一个 pg_notify → APISIX Admin API 的桥接器）？请给出可操作的架构图和关键代码。"

---

#### B2. APISIX 的 Lua-Casbin 如何连接 PostgreSQL 加载视图 —— 配置缺失

**问题描述：**
方案说 APISIX 通过 `luasql-adapter` 连接 PostgreSQL 读取 `casbin_rule` 视图，但：
- `luasql-postgres` 在 OpenResty 生态中并不主流，实际可用性和性能未验证
- APISIX 的 `authz-casbin` 插件示例中通常用的是**文件适配器**，数据库适配器配置文档极少
- 连接池管理策略未讨论（多少个 DB 连接？连接断开如何重连？）

**追问方向：**
> "APISIX 的 authz-casbin 插件如何配置 PostgreSQL 数据库适配器？请给出完整的插件配置 YAML 示例（包含数据库连接串、连接池大小、重试策略）。如果 luasql-postgres 不可行，是否有替代方案（如用 etcd 做中间缓存层）？"

---

#### B3. 安全威胁模型 —— 完全缺失

**问题描述：**
整个方案没有讨论任何安全威胁模型。一个权限管理系统如果不做威胁建模，等于没有安全保障。关键缺失：
- **JWT 密钥管理**：密钥如何安全存储？如何轮转（rotation）？
- **SQL 注入**：PostgREST 使用参数化查询是安全的，但自定义的 PL/pgSQL 函数中的动态 SQL（如 `EXECUTE`）是否安全？
- **CSRF 防护**：前端 Admin UI 直接调用 PostgREST API 时如何防御 CSRF？
- **CORS 配置**：跨域访问策略如何设计？
- **认证绕过**：如果有人伪造 JWT 或重放攻击，防御方案是什么？
- **PostgREST 的安全配置**：`PGRST_DB_ANON_ROLE`、`PGRST_JWT_SECRET` 等关键环境变量的安全存储方案
- **APISIX 与 PostgREST 之间的通信**：是否需要 mTLS？

**追问方向：**
> "请为这套架构设计完整的安全威胁模型，至少覆盖：JWT密钥生命周期管理（生成/存储/轮转）、TLS/mTLS部署方案、CSRF防护、CORS策略、SQL注入风险评估、认证绕过防护。特别是JWT密钥在多环境（dev/staging/prod）下的管理方案。"

---

#### B4. 高可用（HA）架构 —— 几乎空白

**问题描述：**
方案使用 Pigsty（主打高可用），但只提到了数据库层的 HA。整套链路中：
- **APISIX 的 HA**：需要 etcd 集群存储配置，如何部署？
- **PostgREST 的 HA**：多实例水平扩展时如何负载均衡？
- **数据库主备切换**：Pigsty 的 Patroni + etcd 切换时，APISIX 的 DB 连接如何自动切换？
- **pg_notify 在 HA 下的行为**：主库故障切换后，APISIX 的 LISTEN 连接会断开，重连到新主库的机制是什么？
- **casbin 策略缓存一致性**：多 APISIX 实例各自持有内存缓存，主备切换后如何确保所有实例的策略一致？

**追问方向：**
> "请给出这套架构的完整高可用部署拓扑图，覆盖 APISIX（含 etcd 集群）、PostgREST（多实例+负载均衡）、Pigsty PostgreSQL（主备+自动切换）三层的高可用方案。特别关注：pg_notify 在主备切换时的行为、多 APISIX 实例下 casbin 内存缓存的最终一致性保证。"

#### I7. 数据权限（行级权限）方案 —— 未讨论

**问题描述：**

- 部门经理只能看本部门数据

**追问方向：**

## 四、总结

这份方案在**架构理念层面**非常出色——"读写分离"、"视图替代触发器"、"网关层无感鉴权"的设计思想是先进且可落地的。但在**工程化落地层面**存在明显缺口，最核心的三个问题是：

1. **APISIX 与 PostgreSQL NOTIFY 的桥接机制**——方案中对此一笔带过，但这是整个"实时同步"链路的关键环节，目前业界没有标准方案
2. **安全体系设计**——作为权限管理系统，安全威胁模型和防护措施的缺失是致命缺陷
3. **高可用和容量规划**——方案目前停留在单机原型阶段，缺乏生产级部署的考量

建议在正式开发前，先围绕以上三个核心问题与技术专家深入讨论，补齐工程化方案的最后一公里。
