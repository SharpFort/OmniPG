## 🔴 P0 — 阻塞级（系统不可用或安全漏洞）

### P0-1. MD5 对账哈希永远不匹配 — 10分钟对账机制完全失效

**位置：** Go 第 296 行（DB 侧） vs 第 359 行（Go 侧）

**问题详情：**

DB 侧计算哈希（第 296 行）：
```sql
SELECT md5(COALESCE(string_agg(concat_ws(',', ptype, v0, v1, v2, v3, v4, v5), E'\n' ORDER BY ...), ''))
```
- `concat_ws(',', ...)` 使用 **逗号无空格** 连接各列
- `string_agg(..., E'\n' ORDER BY ...)` 使用 **换行符** 连接各行

Go 侧生成 CSV（第 359 行）：
```go
builder.WriteString(strings.Join(elements, ", "))  // 逗号+空格
builder.WriteString("\n")
```

**实际差异：**

| 维度 | DB 侧 | Go 侧 |
|:---|:---|:---|
| 列分隔符 | `,`（无空格） | `, `（逗号+空格） |
| NULL 处理 | `concat_ws` 自动跳过 NULL，不产生空字符串 | `cols[i].String` 对 NULL 输出空字符串 `""` |
| 行尾 | 无尾随换行（string_agg 行为） | 每行末尾 `\n`（包括最后一行） |

**后果：**
- 假设 DB 中有一条策略 `p,role_admin,/sys_user,GET`，DB 哈希基于 `"p,role_admin,/sys_user,GET"` 计算
- Go 侧生成 `"p, role_admin, /sys_user, GET"`（多了空格）
- MD5 完全不同 → 每 10 分钟触发一次不必要的全量同步
- 如果 APISIX 策略与 DB 实际一致，对账永远报告 "Hash mismatch"，日志被噪音淹没

**修复方案（推荐方案 A）：**

```go
// 修改 formatToCSV 函数，使用与 DB 完全一致的格式
func (s *Syncer) formatToCSV(rows []PolicyRow) string {
    var builder strings.Builder
    for _, r := range rows {
        // 使用逗号无空格分隔，与 concat_ws(',', ...) 一致
        parts := []string{r.Ptype}
        cols := []*sql.NullString{&r.V0, &r.V1, &r.V2, &r.V3, &r.V4, &r.V5}
        
        // 找到最后一个非 NULL 列（与 concat_ws 行为一致：跳过 NULL）
        lastValidIdx := -1
        for i := len(cols) - 1; i >= 0; i-- {
            if cols[i].Valid {
                lastValidIdx = i
                break
            }
        }
        
        for i := 0; i <= lastValidIdx; i++ {
            if cols[i].Valid {
                parts = append(parts, cols[i].String)
            }
            // NULL 值被跳过（与 concat_ws 行为一致）
        }
        
        builder.WriteString(strings.Join(parts, ","))  // 无空格！
        builder.WriteString("\n")
    }
    return builder.String()
}
```

**同时修改 DB 侧查询（确保 NULL 处理一致）：**
```sql
SELECT md5(COALESCE(string_agg(
    concat_ws(',', ptype, 
        COALESCE(v0, ''), COALESCE(v1, ''), COALESCE(v2, ''),
        COALESCE(v3, ''), COALESCE(v4, ''), COALESCE(v5, '')
    ), E'\n' ORDER BY ptype, v0, v1, v2, v3, v4, v5
), '')) FROM casbin_rule;
```

> ⚠️ **注意：** 02 文档中 `casbin_rule` 视图的 v3/v4/v5 列定义为 `NULL::varchar`，永远为 NULL。DB 侧 `concat_ws` 会跳过它们，Go 侧截断尾部 NULL 后也不包含它们。但 `v0`（role_code）和 `v1`（path）和 `v2`（method）在视图中永远非 NULL，所以实际差异主要是 **分隔符空格** 问题。

### P0-2. serverless-pre-function 不验证 JWT 签名即注入 Header — 严重安全漏洞

**位置：** 04 文档第 92-93 行（Lua 代码）

**问题详情：**

当前插件执行顺序：
1. `serverless-pre-function`（rewrite 阶段）→ Base64url 解码 JWT payload → 注入 `X-User-Role` 和 `X-User-Id`
2. `jwt-auth`（access 阶段）→ 验证 JWT 签名
3. `authz-casbin`（access 阶段）→ 使用 `X-User-Role` 做权限判定

**攻击向量：**

攻击者可以构造一个任意 payload 的 JWT：
```
eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJyb2xlcyI6WyJzdXBlcl9hZG1pbiJdLCJ1c2VyX2lkIjoiMSJ9.fake_signature
```

1. `serverless-pre-function` 在 rewrite 阶段解码 payload → 提取 `roles=["super_admin"]` → 注入 `X-User-Role: super_admin`
2. `jwt-auth` 在 access 阶段验证签名 → **失败** → 返回 401

**但是：** 如果 `jwt-auth` 验证失败，`authz-casbin` 不会执行（APISIX 插件链在 jwt-auth 处中断）。所以**当前配置下这个漏洞不会直接被利用**。

**残留风险：**
- 如果未来有人将 `jwt-auth` 改为可选（如对某些路径跳过验证），则漏洞立即可利用
- `X-User-Id` 被注入后可能被日志记录，造成日志污染
- 代码设计违反了"零信任"原则：先注入信任依据，后验证

**修复方案（推荐）：**

**方案 A（最佳）：** 完全移除 `serverless-pre-function` 的 JWT 解析，改用 APISIX 原生方式
```lua
-- 使用 APISIX 的 jwt-auth 插件自动将 claims 注入为变量
-- 在 authz-casbin 中使用 ctx.var.jwt_payload_roles 而非 X-User-Role
```

**方案 B（次选）：** 如果必须手动解析，至少添加签名预检查
```lua
-- 在 serverless-pre-function 中添加 HMAC 快速校验（仅检查格式，不替代 jwt-auth）
-- 但这是重复工作，不推荐
```

**方案 C（最小改动）：** 在文档中明确标注此设计决策，并添加注释说明依赖 jwt-auth 的验证
```lua
-- WARNING: This function does NOT verify JWT signature.
-- Signature verification is performed by jwt-auth plugin in the access phase.
-- If jwt-auth is bypassed or disabled, this header MUST NOT be trusted.
```

### P0-3. 缺少 Advisory Lock 选主 — 多实例部署会产生竞态条件

**位置：** Go 第 200-238 行（main 函数）+ 00 文档第 133 行（设计要求）

**问题详情：**

00 文档明确要求："多实例 + PostgreSQL Advisory Lock 选主"。当前代码完全没有实现选主逻辑。

**后果（如果部署 2+ 实例）：**
1. 两个实例同时 `LISTEN casbin_channel` → 每个 NOTIFY 事件被两个实例同时接收
2. 两个实例同时执行 `Sync()` → 同时 `PUT /apisix/admin/plugin_metadata/authz-casbin`
3. 后到达的响应可能覆盖先到达的响应（如果存在网络延迟差异）
4. 10 分钟对账时，两个实例同时计算哈希并可能同时触发全量同步

**修复方案：**

```go
func main() {
    // ... 现有初始化代码 ...
    
    // Advisory Lock 选主
    tx, err := db.Begin()
    if err != nil {
        log.Fatalf("Failed to begin transaction: %v", err)
    }
    
    var acquired bool
    err = tx.QueryRow("SELECT pg_try_advisory_lock(12345)").Scan(&acquired)
    if err != nil {
        log.Fatalf("Failed to acquire advisory lock: %v", err)
    }
    
    if !acquired {
        log.Println("Another instance is running as leader. This instance will stand by.")
        // 可选：定期重试获取锁
        go func() {
            for {
                time.Sleep(30 * time.Second)
                tx2, _ := db.Begin()
                var got bool
                tx2.QueryRow("SELECT pg_try_advisory_lock(12345)").Scan(&got)
                if got {
                    log.Println("Acquired leadership. Starting event loop.")
                    tx = tx2
                    goto START_LOOP
                }
                tx2.Rollback()
            }
        }()
        // 阻塞等待
        select {}
    }
    
START_LOOP:
    log.Println("Acquired advisory lock. Acting as leader.")
    // ... 继续现有逻辑 ...
    
    // 退出时释放锁（通过 tx.Rollback() 或 tx.Commit()）
    defer tx.Rollback()
}
```

## 🟡 P1 — 重要级（功能缺陷或安全隐患）

### P1-1. 缺少优雅关闭（Signal Handling）

**位置：** Go 第 237 行（`StartEventLoop` 永久阻塞）

**问题：**
- `main()` 调用 `syncer.StartEventLoop(ctx, listener.Notify)` 后永久阻塞
- 如果进程收到 SIGTERM/SIGINT，`defer db.Close()` 和 `defer listener.Close()` 不会执行（因为 main 永远不会返回）
- 进行中的 `Sync()` 可能被中断，导致 APISIX 策略处于半写入状态

**修复方案：**

```go
import "os"
import "os/signal"
import "syscall"

func main() {
    // ... 现有初始化代码 ...
    
    // 信号监听
    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)
    
    // 在 goroutine 中运行事件循环
    go syncer.StartEventLoop(ctx, listener.Notify)
    
    // 等待信号
    <-sigCh
    log.Println("Shutdown signal received. Cleaning up...")
    cancel()  // 触发 ctx.Done()
    time.Sleep(1 * time.Second)  // 等待进行中的同步完成
}
```

### P1-6. 插件执行顺序风险

**位置：** 04 文档第 90-99 行

**当前顺序：**
1. `serverless-pre-function`（rewrite）
2. `jwt-auth`（access）
3. `authz-casbin`（access）

**问题：**
- `jwt-auth` 和 `authz-casbin` 都在 `access` 阶段执行
- APISIX 插件执行顺序由插件优先级（priority）决定
- `jwt-auth` 默认优先级 = 2500，`authz-casbin` 默认优先级 = 2599
- 数字越小越先执行 → `jwt-auth` 先于 `authz-casbin` → **当前配置正确**

**但：** 如果未来有人调整插件优先级或添加新插件，可能破坏此顺序。

**修复方案：** 在文档中显式声明执行顺序依赖：
```yaml
# 插件执行顺序（由 APISIX 插件优先级控制）：
# 1. serverless-pre-function (rewrite, priority=10000+) → 注入 X-User-Role
# 2. jwt-auth (access, priority=2500) → 验证 JWT 签名
# 3. authz-casbin (access, priority=2599) → 权限判定
# 注意：jwt-auth 必须在 authz-casbin 之前执行！
```

### P3-3. 缺少请求重试机制

**位置：** Go 第 398 行（pushToApisix）

**问题：**
- HTTP PUT 请求失败时直接返回错误

```go
import "github.com/cenkalti/backoff/v4"

func (s *Syncer) pushToApisixWithRetry(policy string) error {
    operation := func() error {
        return s.pushToApisix(policy)
    }
    return backoff.Retry(operation, backoff.WithMaxRetries(backoff.NewExponentialBackOff(), 3))
}
```

### A. 修复后的 formatToCSV（P0-1）

func (s *Syncer) formatToCSV(rows []PolicyRow) string {
    var builder strings.Builder
    for _, r := range rows {
        parts := []string{r.Ptype}
        cols := []*sql.NullString{&r.V0, &r.V1, &r.V2, &r.V3, &r.V4, &r.V5}
        
        // 找到最后一个非 NULL 列
        lastValidIdx := -1
        for i := len(cols) - 1; i >= 0; i-- {
            if cols[i].Valid {
                lastValidIdx = i
                break
            }
        }
        
        // 只追加非 NULL 值（与 concat_ws 行为一致）
        for i := 0; i <= lastValidIdx; i++ {
            if cols[i].Valid {
                parts = append(parts, cols[i].String)
            }
        }
        
        builder.WriteString(strings.Join(parts, ","))  // 无空格，与 DB 一致
        builder.WriteString("\n")
    }
    return builder.String()
}
```

func (s *Syncer) Reconcile() error {
    var dbHash string
    // 使用 SHA256 替代 MD5
    query := `SELECT encode(digest(COALESCE(string_agg(
        concat_ws(',', ptype, v0, v1, v2, v3, v4, v5), E'\n' 
        ORDER BY ptype, v0, v1, v2, v3, v4, v5
    ), ''), 'sha256'), 'hex') FROM casbin_rule;`
    
    err := s.db.QueryRow(query).Scan(&dbHash)
    if err != nil {
        return fmt.Errorf("calculate DB policy hash failed: %w", err)
    }
    
    apisixPolicy, err := s.fetchActivePolicyFromApisix()
    if err != nil {
        log.Printf("Failed to fetch active policy from APISIX: %v", err)
        return s.Sync()
    }
    
    apisixHash := fmt.Sprintf("%x", sha256.Sum256([]byte(apisixPolicy)))
    log.Printf("Reconciliation Hash Check -> DB: %s | APISIX: %s", dbHash, apisixHash)
    
    if dbHash != apisixHash {
        log.Println("Hash mismatch detected! Triggering full synchronization...")
        return s.Sync()
    }
    
    log.Println("Hash matched. APISIX cache is consistent with Database.")
    return nil
}
```

func main() {
    log.Println("Initializing Policy Syncer...")
    
    db, err := sql.Open("postgres", PostgresDSN)
    if err != nil {
        log.Fatalf("Failed to open database: %v", err)
    }
    defer db.Close()
    
    // Advisory Lock 选主
    tx, err := db.Begin()
    if err != nil {
        log.Fatalf("Failed to begin transaction: %v", err)
    }
    defer tx.Rollback() // 退出时释放锁
    
    var acquired bool
    if err := tx.QueryRow("SELECT pg_try_advisory_lock(12345)").Scan(&acquired); err != nil {
        log.Fatalf("Failed to acquire advisory lock: %v", err)
    }
    if !acquired {
        log.Println("Another instance is running as leader. Exiting.")
        return
    }
    log.Println("Acquired advisory lock. Acting as leader.")
    
    syncer := &Syncer{
        db:     db,
        client: &http.Client{Timeout: 5 * time.Second},
    }
    
    // 初始全量同步
    log.Println("Performing initial reconciliation...")
    if err := syncer.Reconcile(); err != nil {
        log.Printf("Initial reconciliation failed: %v", err)
    }
    
    // 启动事件循环
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()
    
    // 信号监听
    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)
    
    go func() {
        <-sigCh
        log.Println("Shutdown signal received.")
        cancel()
    }()

}
