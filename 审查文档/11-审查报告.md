## 🟡 P1 重要级

### P1-1. `password_hash?` 字段命名误导 — 前端发送明文后用户无法登录

> **位置：** `src/types/system.ts` 第110行 + `src/api/system.ts` 第583-584行

```typescript
/** 用户 */
export interface SysUser {
  id: string
  //...
  password_hash?: string  // ← 仅创建/编辑时传递
}
```

```typescript
/** 创建用户 */
export function createUser(data: Partial<SysUser>): Promise<SysUser[]> {
  return request.post('/sys_user', data)
}
```

以及使用示例 第1148-1152行：
```javascript
await createUser({
  username: 'newuser',
  password_hash: 'hashed_password',  // ← 看起来像明文密码
  tenant_id: 'tenant_default'
})
```

**问题：**

07 Migration 中的 `user_login_sso()` 函数认证逻辑为：

```sql
IF v_user.password_hash != crypt(p_password, v_user.password_hash) THEN
    RAISE EXCEPTION 'Invalid Credentials' ...
END IF;
```

这意味着：
- `p_password` = 用户输入的**明文密码**
- `v_user.password_hash` = 数据库存储的 **bcrypt 哈希**
- `crypt()` = PG 内置 crypt() 函数，以 bcrypt hash 为 salt，验证明文密码

当前端通过 `POST /sys_user` 直接插入时，存入 `password_hash` 列的值取决于 `data.password_hash` 的内容：

| 客户端发送的值 | 登录验证结果 |
|:---|:---|
| 明文 `"plaintext123"` | `crypt("plaintext123", "plaintext123")` ≠ `"plaintext123"` → ❌ 永远失败 |
| bcrypt `"$2b$10$..."` | `crypt("plaintext", "$2b$10$...")` = `"$2b$10$..."` → ✅ 成功 |

**后果：** 开发者在示例引导下发送明文密码，用户创建后所有登录尝试全部失败。

**修复方案：**

**方案 A（推荐 — 安全）：** 重命名字段为 `password`，触发服务端哈希
```typescript
export interface SysUser {
  password?: string  // 明文密码
}
```

可在 PostgREST 前添加 `BEFORE INSERT` 触发器自动哈希：
```sql
CREATE OR REPLACE FUNCTION hash_user_password()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.password IS NOT NULL THEN
    NEW.password_hash = crypt(NEW.password, gen_salt('bf', 10));
    NEW.password = NULL;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**方案 B（保留当前设计但改进文档）：** 在类型注释中明确说明：
```typescript
/** bcrypt 哈希（客户端必须使用 bcrypt 库预先哈希，cost=10）*/
password_hash?: string
```
并提供客户端哈希工具函数。

### P1-2. 缺少 CSRF 防护 — httpOnly Cookie 模式下 CSRF 攻击可行

> **位置：** `src/api/request.ts` 第340行 + 全局

```typescript
withCredentials: true,  // 允许跨域携带 Cookie
```

**问题：**
前端使用 `withCredentials: true` 发送包含 RT 的 httpOnly Cookie。但没有 CSRF token 或 SameSite 严格限制。攻击者可以：

1. 构建虚假页面 `<img src="http://localhost:9080/api/v1/rpc/kick_user?...">` 
2. 利用浏览器自动携带 Cookie 的特性
3. 执行未授权操作（如修改用户角色、删除数据）

虽然 07 Migration 中 Refresh Token Cookie 设置了 `SameSite=Strict`，但这**只保护 RT 不被跨站携带**。对于通过 JWT Authorization Header 的 API 调用，`SameSite` 不适用（跨站请求仍发送 Authorization 头只是被 CORS 阻止）。

而如果 APISIX 的 CORS `allow_origins` 配置为 `*` 或包含恶意域名：

```
APISIX CORS: allow_origins: "*", allow_credentials: true
```

这在浏览器端违反了 CORS 规范（credential + wildcard 不被允许），但如果配置为具体域名匹配，CSRF 攻击就可能成功。

**修复方案：**
1. APISIX CORS 设置同源域名，不使用 `*` + `credentials: true`
2. 添加 CSRF token 到请求头：
   ```typescript
   // 在每个非 GET 请求中
   headers: { 'X-CSRF-Token': csrfToken }
   ```
3. 或使用 Custom Request Header 验证（如直接从 JWT 中提取 session ID）

### P1-3. Logout 不失效 Refresh Token Cookie — 会话劫持风险

> **位置：** `src/api/auth.ts` 第499-502行 + `src/store/user.ts` 第1043-1046行

```typescript
export function logoutApi(): Promise<void> {
  // 后端无专门的退出接口（RT 自然过期）
  // 如果需要立即失效，可以调用 kick_user
  return Promise.resolve()
}
```

```typescript
function logout() {
  resetAll()
  router.push('/login')
}
```

**问题：**
- 前端清除 sessionStorage 中的 Access Token
- 调用 `resetAll()` 清除所有前端状态
- 但 **httpOnly Cookie 中的 RT 无法被 JavaScript 删除**（浏览器只在收到 `Set-Cookie: refresh_token=; Expires=Thu, 01 Jan 1970 00:00:00 GMT` 时删除）
- 当前端"退出登录"后，攻击者如果获取到该用户的 RT Cookie，仍可刷新获取新 AT

**修复方案：**

在后端添加 logout 接口，标记 RT 为已使用：

```sql
-- 新增 logout 函数
CREATE OR REPLACE FUNCTION user_logout()
RETURNS void AS $$
BEGIN
    -- 将所有当前用户的活跃 RT 标记为已使用
    UPDATE sys_user_session 
    SET is_used = TRUE 
    WHERE user_id = current_setting('request.jwt.claims', true)::json->>'user_id'::uuid
      AND is_used = FALSE;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

前端调用：
```typescript
axios.post(`${BASE_URL}/rpc/user_logout`).finally(() => {
  resetAll()
  router.push('/login')
})
```

同时使用 `Set-Cookie: refresh_token=; Path=/; Expires=Thu, 01 Jan 1970` 强制浏览器删除 Cookie。

### P2-1. Vite 代理配置缺失 — 开发环境无法访问 APISIX

> **位置：** `src/api/request.ts` 第333行 `const BASE_URL = import.meta.env.VITE_APP_BASE_API || '/api/v1'`

**问题：**
- 前端设置 `baseURL: /api/v1`
- 这需要 Vite dev server 配置 proxy 将 `/api/v1/*` 转发到 `http://localhost:9080`
- 文中未提供 `vite.config.ts` 配置

**补充配置：**

```typescript
// vite.config.ts
server: {
  proxy: {
    '/api/v1': {
      target: 'http://localhost:9080',
      changeOrigin: true,
    },
  },
},
```

### P2-2. `VITE_JWT_EXPIRES_MINUTES` 环境变量未使用

> **位置：** `.env.development` 第1099行

```ini
VITE_JWT_EXPIRES_MINUTES=15
```

**问题：**
JWT 过期时间在前端由 `parseJwtPayload()` 从 JWT `exp` claim 读取，而非环境变量。此变量无用，会导致新手困惑。

**修复：** 从 `.env` 中移除，或在 JWT 过期预警逻辑中使用该值作为默认阈值。

### P2-3. 路由守卫中 `loadUserMenu()` 缺少错误处理 — 页面加载卡死

> **位置：** 第1173-1179行

```typescript
router.beforeEach(async (to, from, next) => {
  //...
  if (userStore.dynamicRoutes.length === 0) {
    await userStore.loadUserMenu()  // ← 如果 API 报错 → 永远 pending
    next({ ...to, replace: true })
  }
})
```

**问题：** 如果 `/rpc/get_user_menu` 返回 401 或其他错误，`await` 抛出异常被路由守卫捕获但不会调用 `next()`，导致页面"白屏"。

**修复：**
```typescript
try {
  await userStore.loadUserMenu()
  next({ ...to, replace: true })
} catch (e) {
  userStore.resetAll()
  next('/login')
}
```

### P2-5. Refreshing Token 并发竞态条件 — `refreshAttempts` 未重置

> **位置：** `src/api/request.ts` 第351行 + 第437行

```typescript
let refreshAttempts = 0
const MAX_REFRESH_ATTEMPTS = 3

// 在 try 块成功后
refreshAttempts = 0
```

**问题：** 当前逻辑正确（成功刷新后 reset），但如果用户在达到 `MAX_REFRESH_ATTEMPTS` 后被登出，然后再次登录成功，`refreshAttempts` 的引用在模块作用域。由于用户重新登录时已经 `setToken`，但没有重置 `refreshAttempts`，下次刷新超时 3 次后会立即跳转到登录页（还没到真正的 3 次失败限制）。

**修复：**
在 `setToken` 时也重置：
```typescript
function setToken(newToken: string) {
  token.value = newToken
  refreshAttempts = 0  // ← 新增
  // ...
}
```
