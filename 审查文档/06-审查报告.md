## 🔴 P0 阻塞级问题

### P0-1. CSV 尾部换行符导致 SHA256 对账 100% 失败

> **位置：** `formatToCSV()` 第467行 vs `Reconcile()` SQL 第317行

**问题：** Go 和 PostgreSQL 生成的策略文本相差一个尾部换行符，导致哈希**永远不匹配**。

**DB 侧**（`string_agg`）：
```sql
SELECT encode(digest(COALESCE(string_agg(
    concat_ws(',', ptype, v0, v1, v2, v3, v4, v5), E'\n'  -- 行间分隔
    ORDER BY ptype, v0, v1, v2, v3, v4, v5
), ''), 'sha256'), 'hex') FROM casbin_rule;
```
`string_agg` 在多行之间插入 `E'\n'`，但**最后一行后无换行符**。例如2行策略结果为 `"p,role_admin,/sys_user,GET\np,role_admin,/sys_role,GET"`。

**Go 侧**：
```go
for _, r := range rows {
    builder.WriteString(strings.Join(parts, ","))
    builder.WriteString("\n")  // ← 每行末尾都有 \n，包括最后一行
}
return builder.String()
```
结果末尾多一个 `\n`：`"p,role_admin,/sys_user,GET\np,role_admin,/sys_role,GET\n"`。

**后果：**
- APISIX plugin_metadata 存储的策略从有尾部换行符 → DB无尾部换行符 → 哈希必然不匹配 → 每10分钟触发一次不必要的全量同步
- 如果策略量大，全量同步会额外消耗APISIX etcd带宽

**修复方案（选择其一）：**

**方案A（推荐）：Go侧去掉最后一个换行符**
```go
func (s *Syncer) formatToCSV(rows []PolicyRow) string {
    var lines []string
    for _, r := range rows {
        parts := []string{r.Ptype}
        cols := []*sql.NullString{&r.V0, &r.V1, &r.V2, &r.V3, &r.V4, &r.V5}
        lastValidIdx := -1
        for i := len(cols) - 1; i >= 0; i-- {
            if cols[i].Valid {
                lastValidIdx = i
                break
            }
        }
        for i := 0; i <= lastValidIdx; i++ {
            if cols[i].Valid {
                parts = append(parts, cols[i].String)
            }
        }
        lines = append(lines, strings.Join(parts, ","))
    }
    return strings.Join(lines, "\n")  // 无尾部换行
}
```

**方案B：DB侧加尾部换行（不推荐，影响其他读取方）**
```sql
SELECT encode(digest(CONCAT(COALESCE(string_agg(
    concat_ws(',', ptype, v0, v1, v2, v3, v4, v5), E'\n'
    ORDER BY ptype, v0, v1, v2, v3, v4, v5
), ''), E'\n'), 'sha256'), 'hex') FROM casbin_rule;
```

- [P0-1] **必须修复。** 这是逻辑正确性问题，不修复则对账功能完全失效。

### P0-2. Sync 到一半崩溃导致 APISIX 与 DB 永久不一致

> **位置：** `Sync()` 第296-309行

**问题：** `pushToApisix` 成功后、`log.Printf("✅ 已同步 %d 条策略...` 之前如果进程被 kill -9 或 OOM，APISIX 中已写入新策略，但实际上 Sync 是原子的（单次 PUT），不存在"写了一半"。所以这个问题仅在以下场景发生：

1. `pushToApisix` 成功（APISIX已更新）
2. 进程崩溃，`Sync()` 返回未完成
3. 但 APISIX 中的策略已经是正确的

实际上这不是真正的问题——因为 `pushToApisix` 是一个完整的 PUT 操作，APISIX 要么完全更新要么不更新。

**降级为信息：** 如果 APISIX PUT 超时但请求实际已到达 etcd（网络分区场景），etcd 已写入但 Go 端超时返回错误。此时下次对账会检测到哈希不一致并触发全量同步，最终一致。可接受。

## 🟡 P1 重要级问题

### P1-1. 无 `/healthz` 端点 — 容器编排无法做存活/就绪探针

> **位置：** 文档§10 生产环境建议第853行建议"HTTP /healthz 返回 200"，但代码未实现

**问题：** Docker Compose/Kubernetes 需要健康检查来判断 Syncer 是否存活。没有 HTTP 端点就无法配置：
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
```

**影响：**
- Docker Compose 无法配置 `healthcheck`
- Kubernetes 无法配置 liveness/readiness probe
- 运维脚本 `syncer-status.ps1` 只能检查容器是否 running，无法确认进程是否正常处理事件

**修复方案：**
添加轻量 HTTP 服务器（单独的 goroutine）：
```go
http.HandleFunc("/healthz", func(w http.ResponseWriter, r *http.Request) {
    if err := s.db.Ping(); err != nil {
        w.WriteHeader(http.StatusServiceUnavailable)
        return
    }
    w.WriteHeader(http.StatusOK)
})
go http.ListenAndServe(":8080", nil)
```

- [P1-1] **生产环境必须修复。** 开发环境可跳过。

### P1-2. Advisory Lock 断连后失效 — 多实例可能并发写入 APISIX

> **位置：** 第169-184行 `tx` 保持打开状态

**问题：**
1. `tx` 对象连接断开（PG重启、网络抖动）→ Advisory Lock 自动释放
2. 但 `tx` 对象在代码中仍然"存活"，`tx.Rollback()` 可能返回错误
3. 如果此时另一个实例启动并成为 leader，原 leader 的 `pq.Listener` 自动重连后继续执行 Sync
4. **两个实例同时 PUT APISIX Admin API**

**根因：** Advisory Lock 绑定在 `tx` 所在的连接上，而 `pq.Listener` 使用独立连接。Listener 重连成功不代表 Advisory Lock 仍然持有。

**修复方案：**
在事件循环中定期验证锁是否仍被持有：
```go
// 每分钟验证 advisory lock 是否仍在手中
lockTicker := time.NewTicker(1 * time.Minute)
defer lockTicker.Stop()

case <-lockTicker.C:
    var stillLeader bool
    err := s.db.QueryRow("SELECT pg_try_advisory_lock($1)", AdvisoryLockKey).Scan(&stillLeader)
    if err != nil || !stillLeader {
        log.Println("⚠️  Advisory Lock 丢失，退出事件循环...")
        cancel()
    }
```

- [P1-2] **多实例部署时必须修复。** 当前文档说"保持单实例"，但§10提到"如需多活，可使用 etcd 选主"，如果用户阅读§10后尝试多实例部署就会踩坑。

### P1-3. 事件循环潜在竞态条件

> **位置：** `StartEventLoop()` 第251-288行

**问题：** `debounceTimer` 和 `debounceChan` 变量在 `case n := <-notifyChan` 中修改，在 `case <-debounceChan` 中使用。如果两个 case 同时就绪（Go select 随机选择一个），可能出现：

1. `notifyChan` 触发，重置 `debounceTimer`
2. `debounceChan` 触发（旧定时器）→ 执行 Sync，设置 `debounceChan = nil`
3. 新定时器的触发丢失（因为 debounceChan 被设为 nil）

实际上因为 select 只选一个执行，不会真正并发。但如果一个 debounce 周期内收到多次 notify：

```
T=0: notify → debounceTimer.Reset(1s)
T=0.5: notify → debounceTimer.Stop(), debounceTimer.Reset(1s)  
T=1.5: 触发 Sync
```

这个行为是正确的。但如果 notify 和 ticker 同时触发：
```
T=0: notify到达 + ticker到达
select 随机选：
  - 如果选了ticker → Reconcile() 执行
  - 同时 debounceTimer 仍在后台运行，1秒后也会触发 Sync()
```

**影响较小**（顶多多触发一次 Sync），但在高负载场景下会导致重复写入 APISIX。

**修复方案：**
在 `case <-ticker.C` 中暂停 debounce 定时器，或添加 `syncing` 标志防止并发 Sync：
```go
var syncing int32 // atomic flag

case <-debounceChan:
    if atomic.CompareAndSwapInt32(&syncing, 0, 1) {
        go func() {
            defer atomic.StoreInt32(&syncing, 0)
            s.Sync()
       }()
    }
```

- [P1-3] **建议修复。** 高负载下可能导致并发 Sync。

### P1-4. DSN 密码未 URL 编码 — 特殊字符导致连接失败

> **位置：** 第153行 `fmt.Sprintf("postgres://%s:***@%s:%s/%s?sslmode=%s", ...)`

**问题：** 如果 `DBPassword` 包含 `@`、`/`、`:`、`%`、`#`、`?` 等 URL 特殊字符，DSN 会被错误解析，导致连接失败。

**修复方案：**
```go
import "net/url"

password := url.QueryEscape(DBPassword)
dsn := fmt.Sprintf("postgres://%s:%s@%s:%s/%s?sslmode=%s",
    DBUser, password, DBHost, DBPort, DBName, SSLMode)
```

- [P1-4] **建议修复。** 开发环境密码简单可能没问题，但生产环境密码通常包含特殊字符。

### P1-5. Reconcile 错误降级为 Sync — 网络抖动触发全量同步

> **位置：** 第326-329行

**问题：**
```go
apisixPolicy, err := s.fetchActivePolicyFromApisix()
if err != nil {
    log.Printf("⚠️  无法从 APISIX 读取当前策略: %v", err)
    return s.Sync()  // ← 网络错误触发全量同步
}
```

如果 APISIX 短暂不可用（重启、网络分区），`fetchActivePolicyFromApisix` 返回错误，代码直接调用 `Sync()` 进行全量同步。如果策略量大（10万+行），这会造成：
- APISIX Admin API 短时间内收到大量 PUT 请求
- etcd 写入压力增大
- 对账逻辑从"轻量哈希比较"退化为"重量全量同步"

**修复方案：**
区分错误类型：
```go
apisixPolicy, err := s.fetchActivePolicyFromApisix()
if err != nil {
    if strings.Contains(err.Error(), "connection refused") || strings.Contains(err.Error(), "timeout") {
        log.Printf("⚠️  APISIX 暂时不可用，跳过本次对账: %v", err)
        return nil  // 跳过，下次对账再试
    }
    return fmt.Errorf("对账失败: %w", err)
}
```

- [P1-5] **建议修复。** APISIX 重启后恢复期间的全量同步是不必要的。
