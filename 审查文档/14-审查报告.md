## 🔴 与已确认架构的冲突（4项）

### N-1. `plpython3u` 仍在迁移文件中使用 — 已移除的技术栈

> **位置：** §2 目录结构第48行 + §3.2 多处

```
├── db/migrations/
│   ├── 20260703000000_security_extensions.sql# 安装 pgcrypto, pgtap, plpython3u  ← ⚠️
│   ├── 20260704000000_rs256_sign_func.sql   # PL/Python3u 的 RS256 JWT 签署函数  ← ⚠️
│   └── ...
```

**问题：**

07/09 文档（已审查确认）：
- **plpython3u** 在 Pigsty v4.3.0 中**不存在**
- JWT RS256 签发已委托 **Casdoor** 完成
- `sys_secret` 不再存储 RSA 私钥，改为存储 `casdoor_jwks_url`

但本文档仍建议：
1. 安装 `plpython3u` (`20260703000000_security_extensions.sql`)
2. 使用 PL/Python3u 编写 RS256 JWT 签名函数 (`20260704000000_rs256_sign_func.sql`)
3. 推荐 PL/Python 编写业务函数（如 `/db/src/sales/functions/calculate_tax.sql`）

**后果：**
- 开发者照此实施将无法获取/安装 plpython3u
- 即使强行安装，PyJWT 库也需要单独部署到 PG 的 Python 运行时
- 与 Casdoor JWT 签发方案冲突，导致 APISIX 和 PostgREST 使用不同密钥验签

**修复方案：**
移除所有 `plpython3u` 相关迁移和函数：
```
├── db/migrations/
│   ├── 20260703000000_security_extensions.sql  →  仅安装: pgtap, pg_net, pgaudit, pgsodium, pg_graphql
│   └── (删除 rs256_sign_func.sql — 此功能由 Casdoor 提供)
```

### N-3. `casbin_rule` 视图双源管理冲突

> **位置：** §2 的 `20260702000000_create_casbin_view.sql` vs §3 的 `db/src/auth/views/casbin_rule.sql`

**14 文档建议：**
- §2（基础结构）：视图定义在 `db/migrations/` 中作为 DDL 迁移
- §3（扩展结构）：视图定义在 `db/src/auth/views/` 中作为幂等源码

**07 文档实际：**
- 视图定义在 `20260707000003_create_casbin_view.sql`（DDL 迁移）

**问题：**

`casbin_rule` 视图被两个不同的机制管理：
1. **DDL 迁移**（时间戳文件，不可修改）
2. **Idempotent src**（可修改的独立文件）

**后果：**
- 如果两者定义不同（如 07 新增 `is_active` 过滤），同步更新难以保证
- CI/CD 部署顺序产生歧义：先执行 migrations 还是先 apply src？

**修复方案：**
明确 DDL 与 src 的边界：
- **DDL**：只包含 `CREATE TABLE`、`CREATE INDEX`、初始数据
- **Views/Functions/Triggers**：全部放在 `db/src/` 中用 `CREATE OR REPLACE` 管理
- Migrations 中的 View 定义保留为初始版本（首次部署用，后续由 src 覆盖）

### N-4. PostgREST 暴露 Schema 与实际表 Schema 不一致

> **位置：** §3 扩展结构 + §4.3 API 交付

**14 文档建议：**
- **`auth` schema**：存放 RBAC 物理表
- **`sales` schema**：存放订单物理表
- **`api_v1` schema**：PostgREST 唯一暴露视图和 RPC 函数

**07 文档实际：**
- 所有 `sys_*` 表创建在 **`public` schema**
- PostgREST 配置 `db-schemas = "api_v1"` + `db-extra-search-path = "public"`

**冲突分析：**

| 方案 | 表位置 | 暴露方式 | 兼容性 |
|:---|:---|:---|:---|
| 14 建议 | `auth.sys_user` | `api_v1.user_view` → JOIN `auth.sys_user` | ✅ 需要调整 search_path |
| 07 实际 | `public.sys_user` | `api_v1.user_view` → JOIN `public.sys_user` | ✅ 当前工作 |

**两种方案都能工作，但必须统一。**

**修复方案：**

方案 A（保持 public — 与 07 对齐）：
```
所有 sys_* 表 → public schema
PostgREST: db-schemas="api_v1" + db-extra-search-path="public"
api_v1 视图: SELECT * FROM public.sys_user
```

方案 B（隔离 — 与 14 对齐）：
```
所有 sys_* 表 → auth schema
PostgREST: db-schemas="api_v1" + db-extra-search-path="auth, public"
api_v1 视图: SELECT * FROM auth.sys_user
```

## 🟡 内部一致性问题（5项）

### N-5. Makefile `test-db` 命令错误 — 引用不存在的容器名

> **位置：** §3.1 Makefile 第100行

```makefile
test-db:
	@echo "Running PostgreSQL PL/pgSQL unit tests via pg_TAP..."
	docker-compose exec -T db pg_prove -U postgres -d app_db --ext .sql /db/tests/
```

**问题：**
- 08 文档定义的容器名是 `app-postgres`，**没有别名 `db`**
- pg_prove 需要 TTY 交互（输出颜色），`-T` 禁用伪 TTY 会导致输出格式混乱

**修复：**
```makefile
test-db:
	@echo "Running PostgreSQL PL/pgSQL unit tests via pgTAP..."
	docker exec -i app-postgres psql -U app_owner -d app_db -c "SELECT * FROM pg_prove('docker-entrypoint-initdb.d/tests/');"
```

或（Docker Compose v2）：
```makefile
test-db:
	docker compose exec app-postgres pg_prove -U app_owner -d app_db --ext .sql /db/tests/
```

## 🟢 文档完整性问题（2项）

### N-10. 缺少 `apply-src` Makefile 目标

> **位置：** §3 Makefile + §4.2 部署流

**14 文档建议：**
> "部署时：迁移脚本或 CI/CD 只需利用一个简单的 Shell 脚本，扫描 `/db/src/` 并将最新的函数代码一键'重载（Apply）'到数据库中"

但 14 文档的 `Makefile` 中没有定义对应目标。

**修复：**
添加：
```makefile
# 9. 应用 db/src 中的幂等源码（函数/视图/触发器）
apply-src:
	@echo "Applying idempotent SQL sources from db/src/..."
	docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 < <(find db/src -name "*.sql" -exec cat {} \;)
```

或使用更精致的工具：
```makefile
apply-src:
	@for f in $(shell find db/src -name "*.sql" | sort); do \
		echo "Applying $$f..."; \
		docker exec -i app-postgres psql -U app_owner -d app_db -v ON_ERROR_STOP=1 -f "$$f"; \
	done
```
